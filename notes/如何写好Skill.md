# 如何写好 Skill：核心要点总结

> 来源：腾讯技术工程《如何写好 Skill：一份终极实战经验手册》（2026年6月）
> 作者：jackjchou

---

## 一、Skill 本质

Skill = 结构化 Prompt Engineering。物理上是一个文件夹，包含：
- `SKILL.md`（必需）：YAML 头信息 + Markdown 指令
- `scripts/`（可选）：可执行脚本
- `references/`（可选）：参考文档
- `assets/`（可选）：模板等静态资源

**三层渐进式加载机制：**

| 层级 | 加载时机 | 内容 | Token 成本（参考） |
|------|---------|------|-------------------|
| Level 1 | 始终驻留 | name + description（≤100字） | 50-150 Token/个 |
| Level 2 | 匹配触发时加载 | SKILL.md 正文（建议≤500行） | 2,000-5,000 Token |
| Level 3 | 执行中按需读取 | 脚本、参考文档 | 按实际引用计算 |

## 二、Skill vs Rule

| 维度 | Rule | Skill |
|------|------|-------|
| 定位 | 全局约束，始终生效 | 按需触发的能力包 |
| 加载方式 | 每次对话都加载 | 匹配用户意图时才加载 |
| 典型内容 | 编码规范、安全红线 | 迁移流程、审查模板、项目初始化 |
| 长度 | 宜短 | 可长 |

> 判断：所有对话都该遵守的 → Rule；特定任务 → Skill。

## 三、写好 Skill 的 6 个关键技巧

### 1. Description 写准了，成功一半
- 用通用语言 + 技术关键词，避免内部黑话
- 说明：什么场景触发、不触发什么
- 验证法：造 20 个问题（一半该触发、一半不该），测命中率

### 2. 开头说清三件事：做什么、为什么、要不要做
- 目标明确：从哪迁到哪、输入什么输出什么
- 给前置检查条件，告诉 AI 什么时候**跳过**

### 3. 用祈使句下指令，解释"为什么"
- ❌ "你应该检查 Go 版本，然后选择合适的方案"
- ✅ "检查 Go 版本。根据版本号选择方案"
- 解释原因让 AI 理解意图，遇未覆盖场景也能合理判断

### 4. 给出 Before/After 对比 + Few-Shot 示例
- 每个关键操作至少 1 个 Before/After（Diff 格式最佳）
- 3-5 个 Few-Shot 示例（正常/边界/错误各一），AI 输出质量显著提升
- 第一个示例放最常见场景

### 5. 用表格和 ASCII 流程图替代长段文字
- 表格：场景分类、方案对比、配置字段
- 决策流程图：分支判断一目了然

### 6. 关键步骤之间加检查点（Checkpoint）
```
### Step 2: 替换 import 路径
...操作...
**检查点**：
grep -rn "old-http-client" . --include="*.go" | wc -l
# 预期输出：0
```
步骤间验证失败 → 停止后续，先修复当前。

## 四、模块化拆分

**拆分信号：**
- SKILL.md > 500 行
- 包含多个可独立工作流程
- 不同部分改动频率差异大

**拆分模式：**
```
主 Skill：流程总览与编排
├── SKILL.md          → 按顺序引用子步骤
├── steps/
│   ├── 00-env-setup.md
│   ├── 01-dependency-update.md
│   └── 02-api-migration.md
└── scripts/
```

**原则：**
- 一个子 Skill 只管一件事（单一职责）
- 依赖关系写清楚（前置/后续）
- 每个子 Skill 能独立使用（不依赖主流程）

## 五、MCP vs HTTP

| 场景 | 方案 |
|------|------|
| 已有现成 MCP Server | 优先 MCP |
| 多 Skill/多平台复用 | 封装 MCP |
| 需统一鉴权和审计 | 封装 MCP |
| 简单一次性调用 | HTTP 脚本 |
| 改造 MCP 成本太高 | 先用 HTTP 过渡 |

## 六、安全 4 条底线

1. **不硬编码密钥**：API Key/密码/Token 走环境变量
2. **危险操作加确认**：删除/覆盖/DDL 前先列出、等用户确认
3. **数据库操作先备份**：mysqldump → 执行变更 → 验证 → 清理凭据
4. **防 Prompt 注入**：外部数据标记为"数据"而非"指令"，文件名/路径做格式校验

## 七、验证与评估

### 验证清单（每个 Skill 必带）
- 功能验证：所有改动点覆盖
- 构建验证：编译/静态检查通过
- 运行验证：核心流程正常

### 评估指标

| 指标 | 达标线 | 优秀线 |
|------|--------|--------|
| 触发准确率（Precision） | ≥85% | ≥95% |
| 触发召回率（Recall） | ≥85% | ≥95% |
| 效果通过率 | ≥80% | ≥90% |
| 相对提升率（vs 无 Skill） | ≥30% | ≥50% |

## 八、常见反模式速查

| 反模式 | 症状 | 解法 |
|--------|------|------|
| 大杂烩 Skill | 一个 Skill 干三四件事 | 拆分（§四） |
| Description 黑话 | 全是内部术语 | 通用语言 + 技术关键词 |
| 没有示例 | 纯文字描述 | 加 Before/After + Few-Shot |
| 没有验证点 | 做完才检查 | 关键步骤加检查点 |
| 写死数值 | 硬编码配置 | 给判断规则和参考范围 |
| 当 Wiki 写 | 背景 300 行正文 50 行 | 背景放 references/ |

## 九、核心检查清单

**内容层面：**
- [ ] 目标明确（做什么、为什么、什么时候触发）
- [ ] Description 精准
- [ ] 用祈使句直接下指令，解释原因
- [ ] 3-5 个 Before/After 或 Few-Shot 示例
- [ ] 多场景覆盖（决策树/分支处理）
- [ ] 易错点标注
- [ ] 验证清单 + 可执行检查命令

**结构层面：**
- [ ] YAML 元数据完整
- [ ] 正文 ≤ 500 行
- [ ] 超 500 行已拆分
- [ ] 依赖明确（前置条件/相关 Skill）
- [ ] 善用表格和流程图

**安全层面：**
- [ ] 无硬编码密钥
- [ ] 危险操作有确认
- [ ] 输入有校验
- [ ] 网络请求用 HTTPS

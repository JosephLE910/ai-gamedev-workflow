## 第1章 网络游戏概述

### 1.1 多人游戏的简要历程

#### 1.1.1 本地多人游戏
最早的多人游戏是本地多玩家模式，两人共用一台计算机。代表作：《双人网球》（Tennis for Two, 1958）和《太空战争》（Spacewar, 1962）。本地多人游戏与单人游戏在编程上非常相似，区别仅在于对多视点和多输入设备的支持。

#### 1.1.2 早期网络多人游戏
最早的网络多人游戏运行在大型机系统组成的小网络中。代表系统：伊利诺伊大学开发的 PLATO（柏拉图）系统。早期代表作：《帝国》（Empire, 1973）和《迷宫战争》（Maze War）——哪款更早至今无定论。

20世纪70年代末，个人计算机兴起后，开发者利用串口实现两台计算机之间的通信（串口每次传输1比特，主要用于与打印机、调制解调器通信）。菊花链连接模式允许连接两台以上的计算机，是一种早期网络拓扑形式。

#### 1.1.3 多用户网络游戏（MUD）
**MUD（Multi-User Dungeon）**：多玩家同时接入一个虚拟世界的游戏形式，起源于艾塞克斯大学（Essex University）Rob Trushaw 于1978年创建的游戏 *MUD*。MUD 可视为角色扮演游戏《龙与地下城》的早期网络版本。

个人计算机普及后，MUD 开始运行在 BBS（电子公告栏系统）上，通过调制解调器连接，传输速率极低但得以广泛传播。

#### 1.1.4 局域网游戏
局域网（LAN）游戏以《毁灭战士》（Doom, 1993）为现代网络游戏的起源。id Software 开发的这款第一人称射击游戏支持一个游戏会话中4个玩家（合作或竞争）。局域网联机游戏聚会（LAN party）随之兴起：一群人聚集在同一地方，以局域网连接各自计算机进行多人游戏。近年来开发者有放弃局域网版本而追求在线版本的趋势。

#### 1.1.5 在线游戏
在线游戏通过大型网络将地理上有一定距离的计算机连接起来。随着20世纪90年代末互联网爆发，早期流行作品包括 id Software 的《雷神之锤》（Quake, 1996）和 Epic Game 的《虚幻》（Unreal, 1998）。

**关键挑战：延迟**。《雷神之锤》最初版本不支持互联网连接，直到补丁 QuakeWorld 才使其可在互联网上可靠运行。延迟补偿方法将在第7章和第8章详细介绍。

Xbox Live 和 PlayStation Network 等服务的创建，以及 GameSpy 和 DWANGO 等 PC 网络服务的完善，推动了主机在线游戏的兴起。高峰时段通常有数百万活跃用户。

#### 1.1.6 大规模多人在线游戏（MMO）
**MMO（Massively Multiplayer Online Game）**：成百上千玩家同时在同一游戏会话中。大多数 MMO 是角色扮演游戏（MMORPG），也存在第一人称射击类 MMO（MMOFPS）。

发展脉络：
- MMORPG 可视为 MUD 的图形化改进
- 早期运行在拨号网络中：Quantum Link（即后来的 AOL）、CompuServe
- 代表作：《栖息地》（Habitat, 1986）、《网络创世纪》（Ultima Online, 1997）
- EverQuest（1999）成功，《魔兽世界》（World of Warcraft, 2004）出现震惊世界，巅峰时全球超过1200万活跃用户

构建大规模多人在线游戏的技术挑战将在第9章介绍，但其大部分技术超出本书范围。

#### 1.1.7 移动网络游戏
移动平台上的多人游戏以**异步模型**为主（不要求实时传输数据的回合制游戏）——轮到玩家时收到通知，有充足时间采取行动。

代表作：《填字游戏》（Words with Friends, 2009）。随着 Wi-Fi 和移动网络发展，实时网络游戏也可在移动设备上运行，如《炉石传说：魔兽英雄传》（Hearthstone: Heroes of Warcraft, 2014）。

---

### 1.2 星际围攻：部落（Starsiege: Tribes, 1998）

一款基于科幻小说的第一人称射击游戏，1998年底发布。支持128个玩家在局域网和互联网上同时运行。开发时调制解调器速度普遍只有56.6kbit/s，游戏实际支持最低28.8kbit/s。

#### 核心设计思想

**4种数据类型**（按重要性分类）：
1. **非保障数据**：不是游戏所必须的数据，带宽有限时首先丢弃
2. **保障数据**：需要确保准确到达及到达顺序，如对游戏至关重要的事件（标识玩家发起攻击）
3. **最近的状态数据**：只有最新版本的数据才重要（如玩家生命值），旧数据失去意义
4. **最快保障数据**：最高优先级，在可靠传输的基础上保证快速到达（如玩家移动信息）

**网络拓扑**：采用**客户端-服务器模型（C/S）**，而非对等网络模型（P2P）。P2P 需要 O(n²) 的带宽（128玩家时带宽需求过大），C/S 模型服务器只需处理 O(n) 级别的带宽。

#### 网络模型分层（蛋糕模型）

```
┌──────────────────────────────────────┐
│              游戏模拟层               │
│  ghost管理器 │ 移动管理器 │ 事件管理器 │ 其他 │
├──────────────────────────────────────┤
│              流管理器                 │
├──────────────────────────────────────┤
│              连接管理器               │
├──────────────────────────────────────┤
│              平台数据包模块            │
└──────────────────────────────────────┘
```

**1.2.1 平台数据包模块**：最底层，是标准套接字 API 的封装，针对特定平台实现，用于构建和发送不同数据包格式。

**1.2.2 连接管理器**：将网络中两台计算机之间的连接抽象化。从上层流管理器接收数据，再将数据传输给底层平台数据包模块。提供**投递状态通知**（确认数据是否成功传输），通过滑动窗口实现。

**1.2.3 流管理器**：决定允许数据传输的最大速率（根据网络连接质量调整）。早期文献示例：28.8kbit/s 调制解调器用户每秒传输约2KB数据。负责按优先级排序请求，在带宽有限时，移动管理器、事件管理器和 ghost 管理器拥有最高优先级。数据包可能包含来自多个管理器的混合数据。

**1.2.4 事件管理器**：维护一个由游戏模拟层产生的事件队列。事件可视为**远程过程调用（RPC）**的简单形式。例如"玩家发起攻击"事件发送到服务器，服务器确认并执行。事件管理器追踪标记为可靠数据的传输记录；若未确认则重新传输。

**1.2.5 ghost 管理器**：最重要的系统，支持128玩家的关键。任务是复制被认为与指定客户端相关的动态对象状态，并发送给客户端。

- 游戏模拟层负责决定客户端必须知道什么（"必须知道"优先级最高，"最好知道"优先级较低）
- 每个相关对象被赋予一个 **ghost 记录**：包含唯一ID、状态掩码、优先级、状态变换（是否已进入/离开范围）
- ghost 管理器保证最近的数据总能成功传输到所有客户端（因为只有最新信息才有用）

**1.2.6 移动管理器**：最高优先级传输玩家的移动数据。移动信息延迟会导致其他玩家击中该玩家之前的位置。输入捕获速度为30FPS（每秒读取输入30次），最近的数据应尽快发送出去。

**1.2.7 其他系统**：数据块管理器处理本质上静态的游戏对象（如炮塔）的传输，与 ghost 系统所处理的动态对象不同。

### 1.3 帝国时代（Age of Empires, 1997）

同样面临拨号网络带宽和延迟的限制，但采用了截然不同的网络模型：**确定性锁步（Deterministic Lockstep）网络模型**。

**核心思想**：所有计算机相互连接（点对点拓扑），游戏中并不将每个游戏单元的信息发送到网络中，而是发送**命令**。命令在每个节点上被独立评估执行。

**与 Tribes 的对比**：
- Tribes 中，128玩家游戏在任何特定时刻与某一客户端相关的玩家只有一小部分，ghost 管理器每次发送约20-30个 ghost
- AoE 中，玩家控制大量游戏单元（最初每玩家50个，后增加到200个），8玩家战争中可能有400个相关单元需要同步
- AoE 的解决方案：同步**命令**而非**单元信息**。即使最专业的玩家每分钟发出的命令也不超过300条，传输命令的带宽需求远低于传输单元信息

#### 1.3.1 轮班计时器（Turn Timer）
解决不同玩家运行游戏采用不同帧率、网络连接质量不同的问题：

- 默认200毫秒为一个轮班
- 200毫秒内的所有命令存储在缓冲区中
- 200毫秒结束时，该玩家的所有命令通过网络传输给所有其他玩家
- **两轮延迟执行**：50轮发出的命令到52轮才被执行，给其他玩家接收和确认命令提供了充足时间
- 缺点：输入延迟约600毫秒
- 若某玩家经历滞后尖峰，游戏可能暂停模拟，等待是否可以解决问题，否则可能被强行退出

传输命令的优点：内存需求极小，可实现**可保存的比赛重播**（如 Age of Empires II 中的功能）。重播时引擎只需正常运行游戏并执行之前存储的命令。

#### 1.3.2 同步
仅使用轮班计时器不能保证节点间同步。关键问题：不同步错误非常微妙，会随时间累积（如随机地图创建时一只鹿的方位轻微偏离，几分钟后可能导致村民完全不同的行为）。

解决方案：使用**伪随机数生成器（PRNG）**。因为所有伪随机数生成器都使用某个特定的初始数值，同步方法是同步所有游戏实例的**种子值**。每个游戏实例不仅要使用相同的种子，调用 PRNG 的次数也要相同。

作弊检测：每个游戏实例都独立地模拟游戏中的每个单元，可通过作弊检测游戏状态的不同步（如玩家额外给自己500个资源）。然而，也存在"地图作弊"问题——通过作弊查看整个地图，在大多数即时战略游戏中仍是常见问题。

---

### 1.4 总结

- 网络多人游戏起源于大型机网络（Empire, 1973），后扩展到 MUD、BBS
- 20世纪90年代初，以 Doom (1993) 为首的局域网游戏在游戏产业中掀起一阵风潮
- 20世纪90年代末，互联网爆发，在线游戏流行；21世纪早期，在线游戏开始在主机上出现
- **Tribes 模型（C/S）**：平台数据包模块→连接管理器→流管理器→游戏模拟层（事件/ghost/移动管理器）
- **AoE 模型（锁步P2P）**：所有节点点对点连接，传输命令而非单元信息，轮班计时器保存命令缓冲并两轮后执行，确定性模拟要求同步 PRNG 种子

## 第2章 互联网

互联网基于**分组交换**（packet switching）：将数据拆分为数据包，每包独立路由，在同一链路上与其他数据包复用（vs. 电路交换——独占一条物理线路）。TCP/IP 协议族以**分层**方式组织，共5层，每层为上层提供服务。

---

### 2.1 物理层

传输原始比特信号的媒介（铜线、光纤、无线电波）。有时被视为链路层的一部分而非独立层。

### 2.2 链路层

提供同一网络内相邻主机间的通信。

#### 以太网帧格式

| 字段 | 长度 |
|------|------|
| 前导码（Preamble） | 7字节（用于同步）|
| 帧起始定界符（SFD） | 1字节 |
| 目的 MAC 地址 | 6字节 |
| 源 MAC 地址 | 6字节 |
| 长度/类型 | 2字节 |
| 数据 | 46–1500字节 |
| 帧校验序列（FCS/CRC32） | 4字节 |

**MAC 地址**：48位硬件地址，前24位为 OUI（组织唯一标识符），由 IEEE 分配给制造商。

**MTU**（最大传输单元）：以太网 MTU = **1500字节**，是数据字段的最大值。

**交换机**（switch）取代早期集线器（hub）：集线器向所有端口广播，交换机学习 MAC-端口映射后只向目标端口转发，减少碰撞。

### 2.3 网络层

在不同链路层网络的主机间提供逻辑地址系统和路由。

#### IPv4 数据包头部

| 字段 | 说明 |
|------|------|
| 版本（4位） | IP 版本号，IPv4 = 4 |
| 头部长度（4位） | 以32位字计，最小5（20字节） |
| 服务类型（ToS，8位） | QoS 标记 |
| 总长度（16位） | 整个数据包长度（字节） |
| 标识（16位） | 用于分片重组 |
| 标志+分片偏移（16位） | 分片控制 |
| TTL（8位） | 每经过一个路由器减1，归零则丢弃 |
| 协议（8位） | 上层协议（6=TCP, 17=UDP）|
| 头部校验和（16位） | 头部完整性校验 |
| 源 IP 地址（32位） | |
| 目的 IP 地址（32位） | |

#### ARP 协议

**ARP（地址解析协议）**：将 IP 地址解析为 MAC 地址。

流程：
1. 主机发送**广播** ARP 请求（目的 MAC = `FF:FF:FF:FF:FF:FF`）："谁有 IP 地址 X.X.X.X？"
2. 该 IP 的主机回复单播 ARP 应答，包含自身 MAC 地址
3. 发起方将映射存入本地 **ARP 表**缓存

#### 子网与 CIDR

**子网掩码**：按位与 IP 地址得到网络地址。例如：
- IP 18.19.100.42，掩码 255.255.255.0（/24）→ 网络 18.19.100.0

**CIDR 表示法**：18.19.100.0/24（斜线后数字 = 掩码中连续的1的位数）

#### 路由表

路由器维护路由表，条目格式：目标子网 / 下一跳网关 / 出口网卡。查询时取最长前缀匹配。

#### 分片

当数据包超过 MTU 时，路由器将其**分片**：
- 每片携带相同的**标识符**（16位）用于重组
- **分片偏移**标识本片在原数据中的位置（以8字节为单位）
- 接收方缓存所有分片后重组

**建议**：游戏数据包应 < 1300字节，避免分片（分片增加丢包概率，任一片丢失则整包需重传）。

#### IPv6

- 地址长度：**128位**（解决 IPv4 地址耗尽）
- 格式：网络前缀 + 接口 ID
- 用 **NDP**（邻居发现协议）替代 ARP
- 路由器不分片（由端点负责路径 MTU 发现）

### 2.4 传输层

提供进程到进程的端到端通信。

**端口号（16位）**：
- 系统端口 0–1023（需管理员权限）
- 用户端口 1024–49151
- 动态/私有端口 49152–65535

#### UDP

**UDP 头部（8字节）**：源端口 / 目的端口 / 长度 / 校验和

特点：**轻量、无连接、不可靠、无序**。适合实时游戏数据（位置同步）——丢包比延迟等待重传代价更小。

#### TCP

特点：**可靠、有序、面向连接**。

**关键状态变量**：

| 变量 | 缩写 | 含义 |
|------|------|------|
| Send Next | SND.NXT | 下一个发送报文段的序列号 |
| Send Unacknowledged | SND.UNA | 最早未确认报文段的序列号 |
| Send Window | SND.WND | 允许发送的最大未确认数据量 |
| Receive Next | RCV.NXT | 期望收到的下一个序列号 |
| Receive Window | RCV.WND | 当前主机能接收的数据量 |

**三次握手**（建立连接）：
1. A → B：SYN（序列号1000）
2. B → A：SYN-ACK（序列号3000，确认号1001）
3. A → B：ACK（确认号3001）

注意：SYN/FIN 报文段占用1个序列号（TCP 幻影字节）。

**数据传输**：
- 每报文段标记数据第一字节的序列号
- 接收方发 ACK（确认号 = 期望的下一序列号）
- 超时未收到 ACK 则重传
- 乱序到达：可丢弃等待重传，或缓存后确认前序报文时一并递交应用层

**流量控制（Flow Control）**：
- 接收方在 ACK 中告知接收窗口大小（RCV.WND）
- 发送方已发送但未确认的数据量不得超过 SND.WND
- 理论带宽上限 = 接收窗口 / 往返时间（RTT）
- **延迟确认**：收到报文段后等待500ms或下一个报文段再确认，合并 ACK，提高窗口利用率

**拥塞控制（Congestion Control）**：
- **AIMD**（加性增，乘性减）：无丢包时每RTT增加1个MSS，丢包时立即减半
- 防止快速发送方压垮路由器和慢速接收方
- **纳格算法（Nagle's algorithm）**：收集小数据直到凑满 MSS 或无未确认数据时才发送——**游戏中应禁用**，因为实时游戏需要频繁发送小包，纳格算法会增加明显延迟

**断开连接（四次挥手）**：
- 主动方发 FIN → 被动方发 ACK
- 被动方发 FIN → 主动方发 ACK
- 双方各自关闭发送方向

### 2.5 应用层

TCP/IP 最顶层，是多人游戏代码所在的层。

#### 2.5.1 DHCP

**动态主机配置协议**：主机接入网络时自动获取 IP 地址。

流程：
1. 客户端广播 DHCPDISCOVER（UDP，目的地 255.255.255.255:67）
2. DHCP 服务器广播 DHCPOFFER（含可用 IP + 客户端 MAC）
3. 客户端广播 DHCPREQUEST（请求该 IP）
4. 服务器广播 DHCPACK（确认分配，含子网掩码、网关、DNS 等）

---

#### 2.5.2 DNS

**域名系统**：将域名转换为 IP 地址。DNS 查询通过 UDP，端口53。

查询链：本地缓存 → 配置的 DNS 服务器 → 权威 DNS 服务器（递归查询）。结果缓存以减少查询次数。

---

### 2.6 NAT

**公开可路由地址**：互联网上任意路由器均可达的 IP。IPv4 仅有约42亿个，不够用。

**网络地址转换（NAT）**：使用私有 IP（不可公开路由）建立局域网，整个网络共享一个公开 IP。

**私有 IP 地址段**（IANA 保留）：
| 范围 | 子网 |
|------|------|
| 10.0.0.0–10.255.255.255 | 10.0.0.0/8 |
| 172.16.0.0–172.31.255.255 | 172.16.0.0/12 |
| 192.168.0.0–192.168.255.255 | 192.168.0.0/16 |

**NAT 工作原理**：路由器维护 **NAT 表**（源地址:源端口 → 外部端口 → 目标地址），发出包时重写源 IP+端口，收到响应时按表反查还原。

**NAT 穿越**（NAT Traversal）：P2P 游戏中双方都在 NAT 后，无法直接连接。

**STUN（UDP NAT 简单穿越）**：
1. 双方各向第三方服务器（Host N）发包，服务器记录各方的公开 IP:端口
2. 服务器告知双方彼此的公开地址
3. 双方互向对方发包，在各自 NAT 表中添加表项
4. 后续可直接通信（需绕过路由器安全检查）

**对称 NAT 问题**：每个新目标地址都分配不同外部端口，STUN 无法工作；可用端口分配预测或 TCP 打洞（TCP hole punching）。

**IGDP/UPnP**：允许局域网主机自动配置路由器端口映射（并非所有路由器支持）。

---

### 2.7 总结

- **分组交换**支持同一线路多路复用，催生了互联网
- **TCP/IP 5层模型**：物理层→链路层→网络层→传输层→应用层
- **链路层**（以太网）：MTU=1500字节，MAC地址，交换机
- **网络层**（IPv4/IPv6）：ARP解析 MAC，子网/CIDR，路由表，分片建议<1300字节
- **传输层**：UDP（轻量不可靠）vs TCP（可靠有序，流控+拥塞控制，纳格算法游戏中应禁）
- **应用层**：DHCP（动态IP），DNS（域名解析）
- **NAT**：私有网络共享公开IP；NAT穿越用 STUN 或端口映射（IGDP/UPnP）

## 第3章 伯克利套接字

本章介绍最常用的网络构建方法——伯克利套接字（Berkeley Socket）API，讲解创建、操作和销毁套接字的常用函数，讨论平台差异，并探索一种类型安全的 C++ 封装方法。

### 3.1 创建 Socket

Berkeley Sockets API 最初作为 BSD 4.2 的一部分发布，已被移植到几乎所有主流操作系统，是网络编程事实上的标准。

```cpp
SOCKET socket(int af, int type, int protocol);
```

**af（地址族）**：

| 宏 | 含义 |
|----|------|
| AF_INET | IPv4 |
| AF_INET6 | IPv6 |
| AF_IPX | Novell/MS-DOS 早期网络 |
| AF_APPLETALK | 苹果早期网络 |

**type**：

| 宏 | 含义 |
|----|------|
| SOCK_STREAM | 有序、可靠的数据流（TCP） |
| SOCK_DGRAM | 离散的不可靠报文（UDP） |
| SOCK_RAW | 应用层可自定义头部 |

**protocol**：`IPPROTO_UDP`、`IPPROTO_TCP`、`0`（按 type 取默认）

创建示例：
```cpp
SOCKET udpSock = socket(AF_INET, SOCK_DGRAM, 0);
SOCKET tcpSock = socket(AF_INET, SOCK_STREAM, 0);
```

关闭 socket 前先 `shutdown(sock, SD_BOTH)` 停止收发，再 `closesocket(sock)` 释放资源。

### 3.2 API 操作系统差异

- **SOCKET 类型**：Windows = `UINT_PTR`；POSIX（Linux/macOS/PS）= `int`（文件描述符）
- **头文件**：Windows = `WinSock2.h`（必须在 `Windows.h` **之前** include，或定义 `WIN32_LEAN_AND_MEAN`）；POSIX = `sys/socket.h`、`netinet/in.h`、`arpa/inet.h`、`netdb.h`
- **初始化**：Windows 需要 `WSAStartup(MAKEWORD(2,2), &data)` / `WSACleanup()`；POSIX 默认激活
- **错误码**：Windows = `WSAGetLastError()`；POSIX = `errno`（来自 `errno.h`）
- **命名约定**：小写函数（`socket`、`bind`）= 跨平台；大写 WSA 前缀（`WSAGetLastError`）= Windows 专用。开发时应将 Winsock2 专有调用与跨平台代码分离，方便移植。

### 3.3 socket 地址

**sockaddr** 通用结构：
```cpp
struct sockaddr {
    uint16_t sa_family;
    char     sa_data[14];
};
```

**sockaddr_in**（IPv4 专用）：
```cpp
struct sockaddr_in {
    short      sin_family;     // AF_INET
    uint16_t   sin_port;       // 网络字节序端口
    struct in_addr sin_addr;   // 4字节IPv4地址
    char       sin_zero[8];    // 填充，必须清零
};
```

**字节序转换**（主机字节序 ↔ 网络字节序，大端）：
```cpp
uint16_t htons(uint16_t hostshort);  // 端口：主机→网络
uint32_t htonl(uint32_t hostlong);   // 地址：主机→网络
uint16_t ntohs(uint16_t networkshort);
uint32_t ntohl(uint32_t networklong);
```

初始化示例（IP=65.254.248.180，端口=80）：
```cpp
sockaddr_in myAddr;
memset(myAddr.sin_zero, 0, sizeof(myAddr.sin_zero));
myAddr.sin_family = AF_INET;
myAddr.sin_port   = htons(80);
myAddr.sin_addr.S_un.S_un_b.s_b1 = 65;
// ...
```

#### 3.3.1 类型安全的 SocketAddress 封装

```cpp
class SocketAddress {
public:
    SocketAddress(uint32_t inAddress, uint16_t inPort) {
        GetAsSockAddrIn()->sin_family = AF_INET;
        GetAsSockAddrIn()->sin_addr.S_un.S_addr = htonl(inAddress);
        GetAsSockAddrIn()->sin_port = htons(inPort);
    }
    SocketAddress(const sockaddr& inSockAddr) {
        memcpy(&mSockAddr, &inSockAddr, sizeof(sockaddr));
    }
    size_t GetSize() const { return sizeof(sockaddr); }
private:
    sockaddr mSockAddr;
    sockaddr_in* GetAsSockAddrIn()
    { return reinterpret_cast<sockaddr_in*>(&mSockAddr); }
};
typedef shared_ptr<SocketAddress> SocketAddressPtr;
```

#### 3.3.2 用字符串初始化 sockaddr

IP 字符串 → 二进制：`inet_pton` (POSIX) / `InetPton` (Windows)（只能处理 IP 地址，不做 DNS 解析）

域名解析（DNS）：
```cpp
int getaddrinfo(const char* hostname, const char* servname,
                const addrinfo* hints, addrinfo** res);
void freeaddrinfo(addrinfo* ai);
```

`getaddrinfo` 会阻塞线程，应在游戏主线程之外（另开线程）调用。

封装为 `SocketAddressFactory::CreateIPv4FromString("host:port")` → `SocketAddressPtr`。

#### 3.3.3 绑定 socket

```cpp
int bind(SOCKET sock, const sockaddr* address, int address_len);
```

作用：① 告诉 OS 当目标地址匹配时该 socket 是接收者；② 设定发送数据包时的源地址和端口。

- `sin_addr = INADDR_ANY`：绑定本机所有网络接口
- `sin_port = 0`：由 OS 自动选择未用端口
- 客户端通常不需要手动 bind（首次 sendto 时自动绑定）；**服务器必须 bind 到公开端口**

### 3.4 UDP Socket

```cpp
int sendto(SOCKET sock, const char* buf, int len, int flags,
           const sockaddr* to, int tolen);
int recvfrom(SOCKET sock, char* buf, int len, int flags,
             sockaddr* from, int* fromlen);
```

- `len` 建议 ≤ 1300 字节（避免链路层分片，以太网 MTU=1500，减去多层头部）
- `flags` 通常为 0；`MSG_PEEK` 查看数据但不从队列删除
- `sendto` 非零返回值仅表示数据进入发送队列，不代表已送达
- `recvfrom` 的 `from` 参数由函数写入发送方地址，不需要预初始化

**类型安全的 UDPSocket 封装**：
```cpp
class UDPSocket {
public:
    int Bind(const SocketAddress& inToAddress);
    int SendTo(const void* inData, int inLen, const SocketAddress& inTo);
    int ReceiveFrom(void* inBuffer, int inLen, SocketAddress& outFrom);
    ~UDPSocket() { closesocket(mSocket); }
private:
    friend class SocketUtil;
    UDPSocket(SOCKET inSocket) : mSocket(inSocket) {}
    SOCKET mSocket;
};
typedef shared_ptr<UDPSocket> UDPSocketPtr;
```

创建通过 `SocketUtil::CreateUDPSocket(INET)` 工厂函数，私有构造函数防止创建含无效 mSocket 的实例。

### 3.5 TCP Socket

TCP 是有状态的、可靠的，每个 TCP 连接需要独立的 socket。

**服务器流程**：
```cpp
// 1. 创建并绑定监听 socket
SOCKET listenSocket = socket(AF_INET, SOCK_STREAM, 0);
bind(listenSocket, ...);
listen(listenSocket, SOMAXCONN);  // backlog=待接受连接队列最大值

// 2. 循环接受连接
SOCKET clientSocket = accept(listenSocket, &addr, &addrLen);
// accept 返回新 socket，专门与该客户端通信
```

**客户端流程**：
```cpp
connect(sock, addr, addrLen);  // 发送 SYN 启动三次握手
```

监听 socket 只做调度（accept 新连接），数据收发通过 accept 返回的新 socket 进行。

#### 3.5.1 通过连接的 socket 发送和接收

```cpp
int send(SOCKET sock, const char* buf, int len, int flags);
int recv(SOCKET sock, char* buf, int len, int flags);
```

- `send` 将数据放入输出缓冲区，纳格算法会积累到 MSS 大小再发；返回值=已入队字节数
- `recv` 返回 0 = 对端发了 FIN（承诺不再发送数据）；阻塞直到有数据
- 与 UDP 不同，`len` 不需要低于 MTU

#### 3.5.2 类型安全的 TCPSocket

```cpp
class TCPSocket {
public:
    int             Connect(const SocketAddress& inAddress);
    int             Bind(const SocketAddress& inToAddress);
    int             Listen(int inBackLog = 32);
    shared_ptr<TCPSocket> Accept(SocketAddress& inFromAddress);
    int             Send(const void* inData, int inLen);
    int             Receive(void* inBuffer, int inLen);
    ~TCPSocket() { closesocket(mSocket); }
};
typedef shared_ptr<TCPSocket> TCPSocketPtr;
```

`Accept` 返回 `TCPSocketPtr`（shared_ptr），确保不再引用时 socket 自动关闭。

### 3.6 阻塞和非阻塞 I/O

默认 socket 操作是**阻塞**的——没有数据时线程挂起。游戏不能让主线程阻塞。三种解决方案：

#### 3.6.1 多线程

每个可能阻塞的调用分配一个线程。5个客户端的服务器至少需要 7 个线程（监听线程+5×客户端线程+模拟线程）。

缺点：线程数随客户端增加无法扩展；多线程并发数据管理复杂；模拟线程和接收线程访问同一 socket 会互相阻塞。

#### 3.6.2 非阻塞 I/O

```cpp
// Windows
ioctlsocket(sock, FIONBIO, &arg);      // arg=1 开启，0 关闭
// POSIX
int flags = fcntl(sock, F_GETFL, 0);
fcntl(sock, F_SETFL, flags | O_NONBLOCK);
```

非阻塞时，阻塞操作立刻返回 -1，设置 `errno = EAGAIN` 或 `WSAEWOULDBLOCK`。

游戏循环模板：
```cpp
void DoGameLoop() {
    mySock->SetNonBlockingMode(true);
    while (gIsGameRunning) {
        int bytesReceived = mySock->ReceiveFrom(data, sizeof(data), socketAddress);
        if (bytesReceived > 0) ProcessReceivedData(data, bytesReceived, socketAddress);
        DoGameFrame();
    }
}
```

每帧限制读取的最大包数量，防止恶意客户端用大量小包打满服务器。

#### 3.6.3 Select

`select` 同时等待多个 socket，任意一个就绪就返回：

```cpp
int select(int nfds, fd_set* readfds, fd_set* writefds,
           fd_set* exceptfds, const timeval* timeout);
```

- `readfds`：检查可读的 socket 集合（有数据到达时返回）
- `writefds`：检查可写（输出缓冲区有空间时返回）
- `exceptfds`：检查错误
- `FD_ZERO`/`FD_SET`/`FD_ISSET` 宏操作 fd_set

封装为 `SocketUtil::Select(inReadSet, outReadSet, ...)` 实现 TCP 服务器循环：阻塞等待直到任一 socket 可读，如果是监听 socket 则 Accept 新连接，否则 Receive 数据。

### 3.7 其他 Socket 选项

`setsockopt(sock, level, optname, optval, optlen)` 配置 socket 行为：

| 选项 | 级别 | 说明 |
|------|------|------|
| SO_RCVBUF | SOL_SOCKET | 接收缓冲区大小（影响 TCP 接收窗口） |
| SO_SNDBUF | SOL_SOCKET | 发送缓冲区大小 |
| SO_REUSEADDR | SOL_SOCKET | 允许绑定已被占用的地址/端口（调试用） |
| SO_RCVTIMEO | SOL_SOCKET | 接收超时时间 |
| SO_KEEPALIVE | SOL_SOCKET | TCP 保活，定期发送保持连接的数据包，检测失连 |
| TCP_NODELAY | IPPROTO_TCP | 禁用纳格算法，减少延迟（可能引起拥塞，参见第2章） |

### 总结

伯克利套接字是最常用的网络数据传输架构，核心地址类型是 `sockaddr`。UDP socket 无状态无连接，用 `sendto`/`recvfrom` 收发；TCP socket 有状态，需先建立连接，用 `send`/`recv` 收发，服务器还需 `listen`/`accept`。阻塞操作可通过多线程、非阻塞 I/O 或 `select` 解决。封装 `SocketAddress`、`UDPSocket`、`TCPSocket` 等类型安全的 C++ 类，可隔离平台差异并提高代码健壮性。第4章开始讨论如何最好地使用这条通信路径。

## 第4章 对象序列化

为了实现多人游戏网络实体之间的对象传输，游戏必须给对象规定数据格式，才能通过传输层协议发送。本章讨论序列化系统的必要性和方法，探索引用数据、压缩、代码易维护性等问题的处理方法。

### 4.1 序列化的需求

**序列化**：将对象从内存中的随机访问格式转换为比特流格式，可存储到磁盘或通过网络传输，之后再恢复为原始格式。

直接用 `memcpy` 将对象复制为字节缓冲区（鲁棒的序列化）为什么不行？以下面更复杂的 RoboCat 为例：

```cpp
class RoboCat: public GameObject {
public:
    RoboCat(): mHealth(10), mMeowCount(3), mHomeBase(0) { mName[0] = '\0'; }
    virtual void Update();
    void Write(OutputMemoryStream& inStream) const;
    void Read(InputMemoryStream& inStream);
private:
    int32_t       mHealth;
    int32_t       mMeowCount;
    GameObject*   mHomeBase;
    char          mName[128];
    std::vector<int32_t> mMiceIndices;
};
```

直接 memcpy 发送有四个问题：

| 字段 | 问题 |
|------|------|
| **vTablePtr**（虚函数表指针）| 每个进程中虚函数表位置不同，覆盖目标进程的虚函数表指针会导致崩溃 |
| **mHomeBase**（指针）| 指针是特定进程的内存地址，跨进程无意义，必须序列化指针所指的数据或用网络ID替代 |
| **mName[128]**（固定大小数组）| 固定复制128字节，但名字可能只有几个字符，浪费带宽 |
| **mMiceIndices**（std::vector）| STL 内部结构有指针，直接复制二进制不安全 |

解决方案：引入**流（stream）**，逐字段手动序列化。

### 4.2 流

**流（stream）**：封装了一组有序数据元素的数据结构，允许顺序读写。

- **输出流**：数据输出槽，允许顺序插入，不能从流中读取数据
- **输入流**：数据来源，允许顺序提取，不能插入数据
- **文件输出流**：封装已打开的文件；**网络流**：封装 socket

#### 4.2.1 内存流

**输出内存流（OutputMemoryStream）**：封装堆上动态分配的缓冲区，有序写入，发送给另一系统。

```cpp
class OutputMemoryStream {
public:
    OutputMemoryStream(): mBuffer(nullptr), mHead(0), mCapacity(0)
        { ReallocBuffer(32); }
    ~OutputMemoryStream() { std::free(mBuffer); }
    const char* GetBufferPtr() const { return mBuffer; }
    uint32_t    GetLength()    const { return mHead; }
    void Write(const void* inData, size_t inByteCount);
    void Write(uint32_t inData) { Write(&inData, sizeof(inData)); }
    void Write(int32_t  inData) { Write(&inData, sizeof(inData)); }
    // 模板版本：防止非基本类型被序列化
    template<typename T> void Write(T inData) {
        static_assert(std::is_arithmetic<T>::value || std::is_enum<T>::value,
                      "Generic Write only supports primitive data types");
        Write(&inData, sizeof(inData));
    }
private:
    void ReallocBuffer(uint32_t inNewLength);
    char*    mBuffer;
    uint32_t mHead;
    uint32_t mCapacity;
};

void OutputMemoryStream::Write(const void* inData, size_t inByteCount) {
    uint32_t resultHead = mHead + static_cast<uint32_t>(inByteCount);
    if(resultHead > mCapacity)
        ReallocBuffer(std::max(mCapacity * 2, resultHead)); // 自动扩容，2倍策略
    std::memcpy(mBuffer + mHead, inData, inByteCount);
    mHead = resultHead;
}
```

有了输出内存流，RoboCat::Write 可以逐字段写入（暂不处理 mHomeBase 和 mMiceIndices）：

```cpp
void RoboCat::Write(OutputMemoryStream& inStream) const {
    inStream.Write(mHealth);
    inStream.Write(mMeowCount);
    inStream.Write(mName, 128); // 暂时写全部128字节
}
void SendRoboCat(int inSocket, const RoboCat* inRoboCat) {
    OutputMemoryStream stream;
    inRoboCat->Write(stream);
    send(inSocket, stream.GetBufferPtr(), stream.GetLength(), 0);
}
```

**输入内存流（InputMemoryStream）**：从缓冲区按顺序读取，对应 Read 方法：

```cpp
class InputMemoryStream {
public:
    InputMemoryStream(char* inBuffer, uint32_t inByteCount)
        : mCapacity(inByteCount), mHead(0) {}
    ~InputMemoryStream() { std::free(mBuffer); }
    uint32_t GetRemainingDataSize() const { return mCapacity - mHead; }
    void Read(void* outData, uint32_t inByteCount);
    void Read(uint32_t& outData) { Read(&outData, sizeof(outData)); }
    void Read(int32_t&  outData) { Read(&outData, sizeof(outData)); }
private:
    char*    mBuffer;
    uint32_t mHead;
    uint32_t mCapacity;
};

void ReceiveRoboCat(int inSocket, RoboCat* outRoboCat) {
    char* temporaryBuffer = static_cast<char*>(std::malloc(kMaxPacketSize));
    size_t receivedByteCount = recv(inSocket, temporaryBuffer, kMaxPacketSize, 0);
    if(receivedByteCount > 0) {
        InputMemoryStream stream(temporaryBuffer, static_cast<uint32_t>(receivedByteCount));
        outRoboCat->Read(stream);
    } else {
        std::free(temporaryBuffer);
    }
}
```

> **小窍门**：生产环境中不会每次数据包到达都为流分配内存，而是预分配最大尺寸的流，每次数据包到达直接读入，处理完后重置 mHead 为0。

#### 4.2.2 字节存储次序的兼容性

不同平台的字节序（endianness）不同：
- **小端字节序（little-endian）**：最低字节存储在起始地址（低位编址）——x86、x64、iOS
- **大端字节序（big-endian）**：最高字节存储在起始地址——Xbox 360、PS3、PowerPC

当流本身决定使用特定字节序时，写入/读取时若平台字节序不匹配，需要进行字节交换。

```cpp
// 字节交换函数
inline uint16_t ByteSwap2(uint16_t inData) { return (inData >> 8) | (inData << 8); }
inline uint32_t ByteSwap4(uint32_t inData) {
    return ((inData >> 24) & 0x000000ff) | ((inData >> 8) & 0x0000ff00)
         | ((inData << 8) & 0x00ff0000) | ((inData << 24) & 0xff000000);
}
// TypeAliaser：用union实现类型别名，解决float/double等非整数类型的字节交换
template <typename tFrom, typename tTo>
class TypeAliaser {
public:
    TypeAliaser(tFrom inFromValue): mAsFromType(inFromValue) {}
    tTo& Get() { return mAsToType; }
    union { tFrom mAsFromType; tTo mAsToType; };
};
// 模板化ByteSwapper（按sizeof特化）
template<typename T> T ByteSwap(T inData) {
    return ByteSwapper<T, sizeof(T)>().Swap(inData);
}
// 支持字节序的Write
template<typename T> void Write(T inData) {
    if(STREAM_ENDIANNESS == PLATFORM_ENDIANNESS)
        Write(&inData, sizeof(inData));
    else {
        T swappedData = ByteSwap(inData);
        Write(&swappedData, sizeof(swappedData));
    }
}
```

> 注意：单字节字符串不需要字节交换，只有多字节基本数据类型需要。

#### 4.2.3 比特流

内存流只能按字节读写，但网络传输希望用尽可能少的比特来表示数值。**输出内存比特流（OutputMemoryBitStream）**以比特精度读写。

```cpp
class OutputMemoryBitStream {
public:
    OutputMemoryBitStream() { ReallocBuffer(256); }
    ~OutputMemoryBitStream() { std::free(mBuffer); }
    void WriteBits(uint8_t inData, size_t inBitCount);
    void WriteBits(const void* inData, size_t inBitCount);
    const char* GetBufferPtr() const { return mBuffer; }
    uint32_t GetBitLength()  const { return mBitHead; }
    uint32_t GetByteLength() const { return (mBitHead + 7) >> 3; }
    void WriteBytes(const void* inData, size_t inByteCount)
        { WriteBits(inData, inByteCount << 3); }
    // 为各基本类型提供带默认比特数的重载
    void Write(bool inData) { WriteBits(&inData, 1); }  // bool只用1位！
    void Write(uint32_t inData, size_t inBitCount = sizeof(uint32_t) * 8)
        { WriteBits(&inData, inBitCount); }
    template<typename T> void Write(T inData, size_t inBitCount = sizeof(T) * 8) {
        static_assert(std::is_arithmetic<T>::value || std::is_enum<T>::value, "...");
        WriteBits(&inData, inBitCount);
    }
private:
    void ReallocBuffer(uint32_t inNewBitCapacity);
    char*    mBuffer;
    uint32_t mBitHead;
    uint32_t mBitCapacity;
};

void OutputMemoryBitStream::WriteBits(uint8_t inData, size_t inBitCount) {
    uint32_t nextBitHead = mBitHead + static_cast<uint32_t>(inBitCount);
    if(nextBitHead > mBitCapacity) ReallocBuffer(std::max(mBitCapacity * 2, nextBitHead));
    uint32_t byteOffset = mBitHead >> 3;    // 字节索引
    uint32_t bitOffset  = mBitHead & 0x7;   // 字节内偏移
    uint8_t currentMask = ~(0xff << bitOffset); // 保留当前字节已有的比特
    mBuffer[byteOffset] = (mBuffer[byteOffset] & currentMask) | (inData << bitOffset);
    uint32_t bitsFreeThisByte = 8 - bitOffset;
    if(bitsFreeThisByte < inBitCount)   // 当前字节放不下，溢出到下一字节
        mBuffer[byteOffset + 1] = inData >> bitsFreeThisByte;
    mBitHead = nextBitHead;
}
```

示例：`mbs.WriteBits(13, 5)` 写5位，`mbs.WriteBits(52, 6)` 写6位，两个数字紧密排列在一个字节中，无间隙。

输入内存比特流（InputMemoryBitStream）的实现类似，留作练习题。

### 4.3 引用数据

#### 4.3.1 内联或嵌入

当对象完全由序列化父对象拥有（不与其他对象共享）时，将其**内联（inlining）**嵌入到父对象的流中。

```cpp
// 序列化 std::vector<int32_t>
void Write(const std::vector<int32_t>& inIntVector) {
    size_t elementCount = inIntVector.size();
    Write(elementCount);
    Write(inIntVector.data(), elementCount * sizeof(int32_t));
}
// 模板化版本（支持更复杂类型）
template<typename T>
void Write(const std::vector<T>& inVector) {
    size_t elementCount = inVector.size();
    Write(elementCount);
    for(const T& element: inVector) Write(element);
}
template<typename T>
void Read(std::vector<T>& outVector) {
    size_t elementCount;
    Read(elementCount);
    outVector.resize(elementCount);
    for(const T& element: outVector) Read(element);
}
```

先写长度，接收端才知道应该读多少数据。

#### 4.3.2 链接

当数据被多个对象引用时（如 mHomeBase 被多个 RoboCat 共享），嵌入方法会在反序列化时创建多个副本，导致错误。解决方案：**链接（linking）**——给每个对象一个唯一网络ID，序列化时写ID，反序列化时通过 LinkingContext 查找对象。

```cpp
class LinkingContext {
public:
    uint32_t GetNetworkId(GameObject* inGameObject) {
        auto it = mGameObjectToNetworkIdMap.find(inGameObject);
        return (it != mGameObjectToNetworkIdMap.end()) ? it->second : 0;
    }
    GameObject* GetGameObject(uint32_t inNetworkId) {
        auto it = mNetworkIdToGameObjectMap.find(inNetworkId);
        return (it != mNetworkIdToGameObjectMap.end()) ? it->second : nullptr;
    }
private:
    std::unordered_map<uint32_t,    GameObject*> mNetworkIdToGameObjectMap;
    std::unordered_map<GameObject*, uint32_t>    mGameObjectToNetworkIdMap;
};

// 序列化 GameObject* 指针
void Write(const GameObject* inGameObject) {
    uint32_t networkId = mLinkingContext->GetNetworkId(inGameObject);
    Write(networkId);
}
void Read(GameObject*& outGameObject) {
    uint32_t networkId;
    Read(networkId);
    outGameObject = mLinkingContext->GetGameObject(networkId);
}
```

> 注意：链接系统需处理"收到网络ID但本地尚未创建对象"的情况——可以忽略整个对象，或序列化并链接到任何可用引用，将丢失的引用置空。

### 4.4 压缩

#### 4.4.1 稀疏数组压缩

不序列化整个固定大小数组，只序列化实际使用的数据。

```cpp
void RoboCat::Write(OutputMemoryStream& inStream) const {
    // ...其他字段...
    uint8_t nameLength = static_cast<uint8_t>(strlen(mName));
    inStream.Write(nameLength);
    inStream.Write(mName, nameLength); // 只写实际字符，不写全128字节
}
```

稀疏容器是压缩中最容易实现的目标。

#### 4.4.2 熵编码（Entropy Encoding）

利用数据的不确定性：常见值用更少比特表示，非期望数据用更多比特。

示例：mPosition 的 Y 分量（高度），假设 90% 的时间猫在地面（Y=0）：

```cpp
void OutputMemoryBitStream::WritePos(const Vector3& inVector) {
    Write(inVector.mX);
    Write(inVector.mZ);
    if(inVector.mY == 0) {
        Write(true);            // 1位：在地面
    } else {
        Write(false);
        Write(inVector.mY);     // 32位：实际高度
    }
}
// 平均比特数：P_OnGround × 1 + P_InAir × 33 = 0.9×1 + 0.1×33 = 4.2位
// 远小于直接序列化的32位
```

可进一步添加第二个常见值（天花板 Y=100），用2位标识三种情况（地面/天花板/其他）：

```
P_OnGround × 2 + P_InAir × 33 + P_OnCeiling × 2 = 0.9×2 + 0.07×33 + 0.03×2 = 2.93位
```

熵编码的形式：写死常见值查表（简单）→ 赫夫曼编码、算术编码、gamma编码、行程编码等（复杂）。

#### 4.4.3 定点（Fixed Point）

用已知范围和精度将浮点数映射为整数，减少序列化所需比特数。

游戏世界 4000×4000，精度 0.1：
- 可能值总数 = (4000 + 4000) / 0.1 + 1 = 40001
- 需要比特数 = log₂40001 ≈ 15.3 → **16位**（代替32位）

```cpp
inline uint32_t ConvertToFixed(float inNumber, float inMin, float inPrecision) {
    return static_cast<uint32_t>((inNumber - inMin) / inPrecision);
}
inline float ConvertFromFixed(uint32_t inNumber, float inMin, float inPrecision) {
    return static_cast<float>(inNumber) * inPrecision + inMin;
}

void OutputMemoryBitStream::WritePosFix(const Vector3& inVector) {
    Write(ConvertToFixed(inVector.mX, -2000.f, 0.1f), 16);
    Write(ConvertToFixed(inVector.mZ, -2000.f, 0.1f), 16);
    // ... Y分量处理 ...
}
// 平均从96位（3×32）降为约35位
```

#### 4.4.4 几何压缩

**归一化四元数**（表示旋转）：所有分量平方和为1，只需序列化3个分量（X,Y,Z），第4个（W）由 sqrt(1 - X²- Y²- Z²) 计算，再加1位记录W的符号：

```cpp
void OutputMemoryBitStream::Write(const Quaternion& inQuat) {
    float precision = (2.f / 65535.f);
    Write(ConvertToFixed(inQuat.mX, -1.f, precision), 16);
    Write(ConvertToFixed(inQuat.mY, -1.f, precision), 16);
    Write(ConvertToFixed(inQuat.mZ, -1.f, precision), 16);
    Write(inQuat.mW < 0);  // 1位符号
}
// 49位代替128位（4×32）
```

**变换矩阵**：分解为平移（3浮点）+旋转（四元数，49位）+缩放（3浮点）共10个浮点数，加上"是否均匀缩放"等标志位可进一步压缩。

### 4.5 可维护性

#### 4.5.1 抽象序列化方向

每对 Read/Write 方法必须保持同步，合并为单一 `Serialize` 方法：

```cpp
class MemoryStream {
    virtual void Serialize(void* ioData, uint32_t inByteCount) = 0;
    virtual bool IsInput() const = 0;
};
class InputMemoryStream:  public MemoryStream {
    virtual void Serialize(void* ioData, uint32_t inByteCount) { Read(ioData, inByteCount); }
    virtual bool IsInput() const { return true; }
};
class OutputMemoryStream: public MemoryStream {
    virtual void Serialize(void* ioData, uint32_t inByteCount) { Write(ioData, inByteCount); }
    virtual bool IsInput() const { return false; }
};
// 模板化 Serialize，统一处理读写
template<typename T> void Serialize(T& ioData) {
    static_assert(std::is_arithmetic<T>::value || std::is_enum<T>::value, "...");
    if(STREAM_ENDIANNESS == PLATFORM_ENDIANNESS)
        Serialize(&ioData, sizeof(ioData));
    else {
        if(IsInput()) { T data; Serialize(&data, sizeof(T)); ioData = ByteSwap(data); }
        else          { T swappedData = ByteSwap(ioData); Serialize(&swappedData, sizeof(T)); }
    }
}
```

代价：有虚函数调用开销（可用模板代替虚函数优化，留作练习）。

#### 4.5.2 数据驱动的序列化

用**反射系统**自动序列化，无需手写每个类的读写代码：

```cpp
enum EPrimitiveType { EPT_Int, EPT_String, EPT_Float };

class MemberVariable {
public:
    MemberVariable(const char* inName, EPrimitiveType inPrimitiveType, uint32_t inOffset)
        : mName(inName), mPrimitiveType(inPrimitiveType), mOffset(inOffset) {}
    EPrimitiveType GetPrimitiveType() const { return mPrimitiveType; }
    uint32_t       GetOffset()        const { return mOffset; }
private:
    std::string    mName;
    EPrimitiveType mPrimitiveType;
    uint32_t       mOffset; // 该成员变量在类中的内存偏移
};

class DataType {
public:
    DataType(std::initializer_list<const MemberVariable&> inMVs)
        : mMemberVariables(inMVs) {}
    const std::vector<MemberVariable>& GetMemberVariables() const { return mMemberVariables; }
private:
    std::vector<MemberVariable> mMemberVariables;
};

#define OffsetOf(c, mv) ((size_t) & (static_cast<c*>(nullptr)->mv))

// 为类注册反射数据
class MouseStatus {
public:
    std::string mName;
    int mLegCount, mHeadCount;
    float mHealth;
    static DataType* sDataType;
    static void InitDataType() {
        sDataType = new DataType({
            MemberVariable("mName",       EPT_String, OffsetOf(MouseStatus, mName)),
            MemberVariable("mLegCount",   EPT_Int,    OffsetOf(MouseStatus, mLegCount)),
            MemberVariable("mHeadCount",  EPT_Int,    OffsetOf(MouseStatus, mHeadCount)),
            MemberVariable("mHealth",     EPT_Float,  OffsetOf(MouseStatus, mHealth))
        });
    }
};

// 通用序列化函数：遍历成员变量，根据类型调用对应Serialize
void Serialize(MemoryStream* inMemoryStream, const DataType* inDataType, uint8_t* inData) {
    for(auto& mv: inDataType->GetMemberVariables()) {
        void* mvData = inData + mv.GetOffset();
        switch(mv.GetPrimitiveType()) {
            case EPT_Int:    inMemoryStream->Serialize(*(int*)mvData);         break;
            case EPT_String: inMemoryStream->Serialize(*(std::string*)mvData); break;
            case EPT_Float:  inMemoryStream->Serialize(*(float*)mvData);       break;
        }
    }
}
```

优点：
- 代码量减少，错误减少（只需保持反射数据正确）
- 可扩展：为每个成员变量存储压缩比特数、自定义取值范围
- 可复用于磁盘序列化、垃圾收集、GUI对象编辑器

代价：有更多分支，可能导致流水线清空；适合维护性要求高的场景。

### 4.6 总结

序列化是将复杂数据结构拆解成线性字节数组的过程，可以通过网络发送给另一台主机。

- **流**：提供了一种简单方法来创建缓冲区，使用对象中各字段的值填充缓冲区，顺序提取数据再插入到目标对象的合适字段
- **引用数据**：内联（完全由父对象拥有）或链接（共享引用，通过网络ID查找）
- **压缩技术**：稀疏数据结构 → 只序列化实际数据；熵编码 → 常见值用更少比特；定点数 → 利用已知范围精度转换整数；几何压缩 → 利用数学约束减少分量
- **可维护性**：抽象序列化方向（合并 Read/Write 为 Serialize）；数据驱动序列化（反射系统，自动遍历成员变量）

下一章讨论两个问题：如何封装数据给远程主机创建对应对象；当游戏仅需要序列化对象数据的子集时，如何高效处理部分序列化问题。

## 第5章 对象复制

对象序列化仅仅是主机之间状态传输的第一步。本章研究一个通用的复制框架，支持远程进程之间游戏世界和对象状态的同步。

### 5.1 世界状态

**世界状态（world state）**：世界中所有游戏对象状态的集合。传输世界状态的任务可分解为传输每个对象的状态。

常见方案：服务器给所有客户端发送世界状态，客户端收到后更新自己的世界状态，最终让所有玩家体验相同的世界。

### 5.2 复制对象

**复制（replication）**：从一台主机向另一台主机传输对象状态，比序列化更苛刻。成功复制一个对象，主机必须在序列化对象内部状态之前完成三步预处理：
1. 标记数据包为包含对象状态的数据包
2. 唯一标识复制对象
3. 指明被复制对象的类型

**步骤1**：用枚举类型 `PacketType` 标记每个数据包：

```cpp
enum PacketType {
    PT_Hello,
    PT_ReplicationData,
    PT_Disconnect,
    PT_MAX
};
```

每个数据包首先序列化相应的 PacketType，接收方可从每个到达的数据包中立即读取数据包类型来决定如何处理。

**步骤2**：依赖第4章介绍的 `LinkingContext`，扩展支持自动分配网络ID：

```cpp
class LinkingContext {
public:
    LinkingContext(): mNextNetworkId(1) {}
    uint32_t GetNetworkId(const GameObject* inGameObject, bool inShouldCreateIfNotFound) {
        auto it = mGameObjectToNetworkIdMap.find(inGameObject);
        if(it != mGameObjectToNetworkIdMap.end()) return it->second;
        else if(inShouldCreateIfNotFound) {
            uint32_t newNetworkId = mNextNetworkId++;
            AddGameObject(inGameObject, newNetworkId);
            return newNetworkId;
        }
        return 0;
    }
    void AddGameObject(GameObject* inGameObject, uint32_t inNetworkId);
    void RemoveGameObject(GameObject* inGameObject);
    GameObject* GetGameObject(uint32_t inNetworkId);
private:
    std::unordered_map<uint32_t,          GameObject*> mNetworkIdToGameObjectMap;
    std::unordered_map<const GameObject*, uint32_t>    mGameObjectToNetworkIdMap;
    uint32_t mNextNetworkId;
};
```

`mNextNetworkId` 每使用一次自增1，4字节无符号整数足以支持超过40亿个不同对象，通常安全。

**步骤3（硬编码方式，有缺陷）**：用 `dynamic_cast` 判断类型写标识符，接收端用 `switch` 创建对应类型：

```cpp
// 发送端：缺点——需要RTTI（游戏中通常禁用），游戏代码和网络代码高度耦合
void WriteClassType(OutputMemoryBitStream& inStream, const GameObject* inGameObject) {
    if(dynamic_cast<const RoboCat*>(inGameObject))
        inStream.Write(static_cast<uint32_t>('RBCT'));
    else if(dynamic_cast<const RoboMouse*>(inGameObject))
        inStream.Write(static_cast<uint32_t>('RBMS'));
    // ...
}
```

缺点：每添加一个类都要同时修改 WriteClassType 和 CreateGameObjectFromStream，容易遗漏，且使单元测试变复杂（网络模块无法脱离游戏类加载）。

#### 5.2.1 对象创建注册表

**对象创建注册表（object creation registry）**：将类标识符映射到创建函数，使网络代码完全独立于游戏代码。

每个可复制类需提供：
- 唯一的 `kClassId`（四字节字符文本，如 `'RBCT'`）
- 虚函数 `GetClassId()`
- 静态函数 `CreateInstance()`

```cpp
// 用宏避免重复代码
#define CLASS_IDENTIFICATION(inCode, inClass) \
    enum{kClassId = inCode}; \
    virtual uint32_t GetClassId() const {return kClassId;} \
    static GameObject* CreateInstance() {return new inClass();}

class GameObject   { public: CLASS_IDENTIFICATION('GOBJ', GameObject)  ... };
class RoboCat: public GameObject { public: CLASS_IDENTIFICATION('RBCT', RoboCat) ... };
class RoboMouse: public GameObject { public: CLASS_IDENTIFICATION('RBMS', RoboMouse) ... };
```

> 警告：四字符文本是依赖于实现的，C++ 标准中归类为依赖于实现，不保证所有编译器转换方式相同（GCC 和 Visual Studio 使用相同方式，跨编译器需先测试）。

**ObjectCreationRegistry 单例**：

```cpp
typedef GameObject* (*GameObjectCreationFunc)();

class ObjectCreationRegistry {
public:
    static ObjectCreationRegistry& Get() {
        static ObjectCreationRegistry sInstance;
        return sInstance;
    }
    template<class T>
    void RegisterCreationFunction() {
        assert(mNameToGameObjectCreationFunctionMap.find(T::kClassId)
               == mNameToGameObjectCreationFunctionMap.end());
        mNameToGameObjectCreationFunctionMap[T::kClassId] = T::CreateInstance;
    }
    GameObject* CreateGameObject(uint32_t inClassId) {
        GameObjectCreationFunc creationFunc =
            mNameToGameObjectCreationFunctionMap[inClassId];
        return creationFunc();
    }
private:
    ObjectCreationRegistry() {}
    std::unordered_map<uint32_t, GameObjectCreationFunc> mNameToGameObjectCreationFunctionMap;
};

// 游戏启动时调用一次
void RegisterObjectCreation() {
    ObjectCreationRegistry::Get().RegisterCreationFunction<GameObject>();
    ObjectCreationRegistry::Get().RegisterCreationFunction<RoboCat>();
    ObjectCreationRegistry::Get().RegisterCreationFunction<RoboMouse>();
}
```

> 小窍门：如果游戏已经有反射系统（第4章数据驱动序列化），可在 GameObject 中添加 `GetDataType()` 虚函数，把类标识符到 DataType 的映射和 DataType 到实例化函数的映射结合起来。

#### 5.2.2 一个数据包中的多个对象

为提高效率（数据包大小尽可能接近 MTU），一个数据包可包含多个对象。每个对象重复执行：
1. 写对象的网络标识符
2. 写对象的类标识符
3. 写对象的序列化数据

接收方循环处理，直到流中没有剩余未使用数据。

### 5.3 朴素的世界状态复制方法

将整个世界状态装入一个数据包，适合小型游戏（如最初的 Quake）：

```cpp
class ReplicationManager {
public:
    void ReplicateWorldState(OutputMemoryBitStream& inStream,
                             const vector<GameObject*>& inAllObjects);
private:
    void ReplicateIntoStream(OutputMemoryBitStream& inStream, GameObject* inGameObject);
    LinkingContext* mLinkingContext;
};

void ReplicationManager::ReplicateIntoStream(OutputMemoryBitStream& inStream,
                                              GameObject* inGameObject) {
    inStream.Write(mLinkingContext->GetNetworkId(inGameObject, true)); // 写网络ID
    inStream.Write(inGameObject->GetClassId());                         // 写类ID
    inGameObject->Write(inStream);                                       // 写序列化数据
}

void ReplicationManager::ReplicateWorldState(OutputMemoryBitStream& inStream,
                                              const vector<GameObject*>& inAllObjects) {
    inStream.WriteBits(PT_ReplicationData, GetRequiredBits<PT_MAX>::Value); // 标记包类型
    for(GameObject* go: inAllObjects)
        ReplicateIntoStream(inStream, go);
}
```

> 边栏——**序列化枚举值时使用必要的比特数**：`GetRequiredBits<PT_MAX>::Value` 用模板元编程在编译期计算枚举最大值所需的比特数（将 tValue 右移一位直到变0，计数移位次数）。习惯上枚举类型以 `_MAX` 结尾，添加或删除其他元素时 `_MAX` 值自动变化。

**接收端**：

```cpp
class ReplicationManager {
public:
    void ReceiveReplicatedObjects(InputMemoryBitStream& inStream);
private:
    GameObject* ReceiveReplicatedObject(InputMemoryBitStream& inStream);
    unordered_set<GameObject*> mObjectsReplicatedToMe;
};

void ReplicationManager::ReceiveReplicatedObjects(InputMemoryBitStream& inStream) {
    unordered_set<GameObject*> receivedObjects;
    while(inStream.GetRemainingBitCount() > 0) {
        GameObject* go = ReceiveReplicatedObject(inStream);
        receivedObjects.insert(go);
    }
    // 销毁上一帧存在但本帧未出现的对象（保持同步）
    for(GameObject* go: mObjectsReplicatedToMe) {
        if(receivedObjects.find(go) == receivedObjects.end()) {
            mLinkingContext->Remove(go);
            go->Destroy();
        }
    }
    mObjectsReplicatedToMe = receivedObjects;
}

GameObject* ReplicationManager::ReceiveReplicatedObject(InputMemoryBitStream& inStream) {
    uint32_t networkId, classId;
    inStream.Read(networkId);
    inStream.Read(classId);
    GameObject* go = mLinkingContext->GetGameObject(networkId);
    if(!go) {
        go = ObjectCreationRegistry::Get().CreateGameObject(classId);
        mLinkingContext->AddGameObject(go, networkId);
    }
    go->Read(inStream);
    return go;
}
```

**限制**：整个世界状态必须能装入一个数据包，大规模游戏需要增量复制。

### 5.4 世界状态中的变化

大规模游戏中，发送方发送**世界状态增量（world state delta）**，接收方在自己的副本中更新这些变化。

每个**对象状态增量（object state delta）**是三种复制行为之一：创建、更新、销毁。

**ReplicationHeader（复制头）**：

```cpp
enum ReplicationAction { RA_Create, RA_Update, RA_Destroy, RA_MAX };

class ReplicationHeader {
public:
    ReplicationAction mReplicationAction;
    uint32_t          mNetworkId;
    uint32_t          mClassId;
    void Write(OutputMemoryBitStream& inStream) {
        inStream.WriteBits(mReplicationAction, GetRequiredBits<RA_MAX>::Value);
        inStream.Write(mNetworkId);
        if(mReplicationAction != RA_Destroy)
            inStream.Write(mClassId); // 销毁时不需要类ID
    }
    void Read(InputMemoryBitStream& inStream) {
        inStream.Read(mReplicationAction, GetRequiredBits<RA_MAX>::Value);
        inStream.Read(mNetworkId);
        if(mReplicationAction != RA_Destroy)
            inStream.Read(mClassId);
    }
};
```

**发送端三个方法**：

```cpp
// 创建
void ReplicationManager::ReplicateCreate(OutputMemoryBitStream& inStream, GameObject* go) {
    ReplicationHeader rh(RA_Create, mLinkingContext->GetNetworkId(go, true), go->GetClassId());
    rh.Write(inStream);
    go->Write(inStream);
}
// 更新
void ReplicationManager::ReplicateUpdate(OutputMemoryBitStream& inStream, GameObject* go) {
    ReplicationHeader rh(RA_Update, mLinkingContext->GetNetworkId(go, false), go->GetClassId());
    rh.Write(inStream);
    go->Write(inStream);
}
// 销毁（只写头，无数据）
void ReplicationManager::ReplicateDestroy(OutputMemoryBitStream& inStream, GameObject* go) {
    ReplicationHeader rh(RA_Destroy, mLinkingContext->GetNetworkId(go, false));
    rh.Write(inStream);
}
```

**接收端处理复制动作**：

```cpp
void ReplicationManager::ProcessReplicationAction(InputMemoryBitStream& inStream) {
    ReplicationHeader rh;
    rh.Read(inStream);
    switch(rh.mReplicationAction) {
        case RA_Create: {
            GameObject* go = ObjectCreationRegistry::Get().CreateGameObject(rh.mClassId);
            mLinkingContext->AddGameObject(go, rh.mNetworkId);
            go->Read(inStream);
            break;
        }
        case RA_Update: {
            GameObject* go = mLinkingContext->GetGameObject(rh.mNetworkId);
            if(go) {
                go->Read(inStream);
            } else {
                // 可能还没收到Create包——创建临时虚拟对象消耗流中的数据再删除
                go = ObjectCreationRegistry::Get().CreateGameObject(rh.mClassId);
                go->Read(inStream);
                delete go;
            }
            break;
        }
        case RA_Destroy: {
            GameObject* go = mLinkingContext->GetGameObject(rh.mNetworkId);
            mLinkingContext->RemoveGameObject(go);
            go->Destroy();
            break;
        }
    }
}
```

> 警告：世界状态增量复制仅适用于发送方可以精确地表示接收方当前世界状态的情况。由于互联网本身不可靠，不能简单假设接收方的世界状态基于发送方上一次发送的数据包。发送方和接收方都需要通过 TCP 发送数据包，或在 UDP 的上层使用应用层协议来提供可靠性。第7章讨论这个话题。

**局部对象状态复制（部分属性更新）**：

用**位域（properties bitmask）**标记哪些属性需要序列化，只发送发生变化的属性：

```cpp
enum MouseStatusProperties {
    MSP_Name      = 1 << 0,
    MSP_LegCount  = 1 << 1,
    MSP_HeadCount = 1 << 2,
    MSP_Health    = 1 << 3,
    MSP_MAX
};

void MouseStatus::Write(OutputMemoryBitStream& inStream, uint32_t inProperties) {
    inStream.Write(inProperties, GetRequiredBits<MSP_MAX>::Value); // 先写位域
    if((inProperties & MSP_Name)      != 0) inStream.Write(mName);
    if((inProperties & MSP_LegCount)  != 0) inStream.Write(mLegCount);
    if((inProperties & MSP_HeadCount) != 0) inStream.Write(mHeadCount);
    if((inProperties & MSP_Health)    != 0) inStream.Write(mHealth);
}

void MouseStatus::Read(InputMemoryBitStream& inStream) {
    uint32_t writtenProperties;
    inStream.Read(writtenProperties, GetRequiredBits<MSP_MAX>::Value);
    if((writtenProperties & MSP_Name)      != 0) inStream.Read(mName);
    if((writtenProperties & MSP_LegCount)  != 0) inStream.Read(mLegCount);
    if((writtenProperties & MSP_HeadCount) != 0) inStream.Read(mHeadCount);
    if((writtenProperties & MSP_Health)    != 0) inStream.Read(mHealth);
}
```

此方法也可结合第4章的数据驱动序列化，扩展 Serialize 函数支持 inProperties 位域参数（清单5.16）：对每个成员变量检查 `(1 << mvIndex) & inProperties` 决定是否序列化。

> 如果有多于32个可能属性，使用 uint64_t；超过64个属性则考虑将多个属性合并成组或拆分类。

### 5.5 RPC 作为序列化对象

**远程过程调用（RPC，remote procedure call）**：一台主机在另一台或多台远程主机上执行程序的动作。适合传输一次性事件（爆炸音效、屏幕闪烁）。

实现方式：将每个RPC调用视为一个唯一对象，参数对应成员变量。例：

```cpp
void PlaySound(const string& inSoundName, const Vector3& inLocation, float inVolume);
// 对应参数对象：
struct PlaySoundRPCParams {
    string  mSoundName;
    Vector3 mLocation;
    float   mVolume;
};
```

在 ReplicationAction 中新增 `RA_RPC`（不需要网络ID），ReplicationManager检测到后传给 RPCManager 处理。

**RPCManager**：

```cpp
typedef void (*RPCUnwrapFunc)(InputMemoryBitStream&);

class RPCManager {
public:
    void RegisterUnwrapFunction(uint32_t inName, RPCUnwrapFunc inFunc) {
        assert(mNameToRPCTable.find(inName) == mNameToRPCTable.end());
        mNameToRPCTable[inName] = inFunc;
    }
    void ProcessRPC(InputMemoryBitStream& inStream) {
        uint32_t name;
        inStream.Read(name);
        mNameToRPCTable[name](inStream);
    }
    std::unordered_map<uint32_t, RPCUnwrapFunc> mNameToRPCTable;
};

// 游戏代码注册解封装函数
void UnwrapPlaySound(InputMemoryBitStream& inStream) {
    string soundName; Vector3 location; float volume;
    inStream.Read(soundName); inStream.Read(location); inStream.Read(volume);
    PlaySound(soundName, location, volume);
}
void RegisterRPCs(RPCManager* inRPCManager) {
    inRPCManager->RegisterUnwrapFunction('PSND', UnwrapPlaySound);
}
```

**发送端调用RPC**：

```cpp
void PlaySoundRPC(OutputMemoryBitStream& inStream,
                  const string& inSoundName, const Vector3& inLocation, float inVolume) {
    ReplicationHeader rh(RA_RPC); // 不需要网络ID
    rh.Write(inStream);
    inStream.Write(inSoundName);
    inStream.Write(inLocation);
    inStream.Write(inVolume);
}
```

> 注意：大部分支持RPC的引擎使用构建工具自动生成封装/解封装胶水函数并与RPC模块注册，避免手工维护。

**RPC vs RMI（远程方法调用）**：
- **RPC**：针对自由函数，不针对特定对象，ReplicationHeader 中网络ID为0
- **RMI（remote method invocation）**：针对特定游戏对象（需要网络ID），新增复制动作 `RA_RMI`；节省带宽，但以标识符长度为代价

### 5.6 自定义解决方案

当通用框架无法满足特殊需求时，可以：
- 扩展 `ReplicationAction` 枚举，并在 `ProcessReplicationAction` 的 switch 中添加 case
- 扩展 `PacketType` 枚举类型创建全新的数据包类型和管理器来处理
- 参考 ObjectCreationRegistry 和 RPCManager 中使用的注册表设计模式，插入高层代码处理自定义数据包，而不破坏底层网络系统

### 5.7 总结

对象复制不仅仅是发送序列化数据，还需要：
- **唯一对象标识符**（网络ID）：接收方将传入状态指向恰当对象
- **唯一类标识符**：接收方如不存在该对象，可基于正确类创建
- **网络代码与游戏代码解耦**：通过对象创建注册表间接映射，避免相互依赖

| 方案 | 适用场景 | 特点 |
|------|----------|------|
| 全量世界状态复制 | 小型游戏（如Quake） | 每帧发送所有对象，简单但包大 |
| 世界状态增量复制 | 大型游戏 | 只发送创建/更新/销毁事件，需可靠传输 |
| 局部属性更新 | 对象属性频繁部分变化 | 位域标记变化属性，节省带宽 |
| RPC | 一次性事件（音效、特效） | 封装为序列化对象，集成到复制框架 |

对象复制是多人游戏底层工具箱的关键部分，也是第6章讲解的高层网络拓扑提供实现支持的关键元素。

## 第6章 网络拓扑和游戏案例

本章介绍多台计算机通信时可使用的两种主要配置（客户端-服务器和对等网络），并将前几章所有内容组合在一起，创建两个游戏案例的最初版本。

### 6.1 网络拓扑

**网络拓扑（network topology）**：决定网络中计算机之间是如何连接的，即参与游戏的计算机如何组织在一起。目标：保证所有玩家都可以看到游戏状态的最新版本。

#### 6.1.1 客户端-服务器

一个游戏实例被指定为**服务器**，其他所有实例被指定为**客户端**。每个客户端只能和服务器通信，服务器负责与所有客户端通信。

| 指标 | 说明 |
|------|------|
| 连接数 | 总共 O(n²) 个连接，但不对称——服务器有 O(n) 个，每个客户端只有1个 |
| 带宽（服务器）| 每秒 c×n 字节输出（给n个客户端各发c字节） |
| 带宽（客户端）| 每秒 c 字节下载 + b 字节上传 |
| 带宽扩展性 | 客户端数量增加时，服务器带宽线性增加；客户端带宽理论不变（实际略有增加） |

**权威（authoritative）服务器**：服务器上的游戏模拟是正确的。客户端若发现自己的游戏状态与服务器不同，应以服务器为准更新。

**往返时间（Round Trip Time, RTT）**：数据包从发送端到总线再回到原始主机所经历的总共时间。理想情况下100毫秒或更少。在客户端A→服务器→客户端B的路径中，B经历的最坏延迟 = A的RTT/2 + 服务器处理时间 + B的RTT/2。

服务器的两种形式：
- **专用（dedicated）服务器**：只运行游戏状态，无外设，与客户端进程完全分开（如《战地》Battlefield），允许在高性能机器上运行多个实例
- **监听服务器（listen server）**：服务器也是游戏本身的积极参与者，降低部署成本（玩家自己的机器），但主机性能需足够强。有时被误称为"对等网络"，正确说法是**对等托管（peer hosted）**
- **主机迁移（host migration）**：监听服务器断开时，客户端中的一个被升为新服务器，需要客户端间有通信，是客户端-服务器和对等网络的混合模型

#### 6.1.2 对等网络

每个独立参与者都与其他所有参与者连接。n个对等体：O(n²)个连接，带宽需求**对称**（上传和下载带宽要求相同）。

**输入共享（input sharing）模型**：每个对等体共享所有动作，每个对等体独立模拟这些动作的执行。《帝国时代》的**确定性锁步**实现：命令放入队列，200毫秒后再发送给所有对等体执行，每轮有2轮延迟。

| 对比 | 客户端-服务器 | 对等网络 |
|------|---------------|----------|
| 延迟（最坏情况）| 两端RTT之和 + 服务器处理时间 | RTT的1/2（对等体直接通信） |
| 服务器断开 | 灾难性 | 去掉该对等体游戏继续 |
| 权威 | 单一权威服务器 | 概念模糊，需要强确定性保同步 |
| 新玩家加入 | 服务器知道所有客户端地址 | 需要**主对等体（master peer）**协调 |

对等网络最大挑战：**保证所有对等体保持彼此同步**——游戏实现必须十分确定（相同输入必须始终产生相同输出）。

### 6.2 客户端-服务器的实现：《机器猫行动》

《机器猫行动》（*Robo Cat Action*）：自上而下的游戏，猫竞相收集老鼠同时互相投掷线球。

#### 6.2.1 服务器和客户端的代码分离

三个独立的目标代码：
1. **RoboCat共享库**：服务器和客户端都用的共享代码（UDPSocket、OutputMemoryBitStream等）
2. **RoboCatServer**：服务器可执行文件
3. **RoboCatClient**：客户端可执行文件

类层次结构：
```
GameObject
├── RoboCat（共享库基类）
│   ├── RoboCatServer（服务器代码）
│   └── RoboCatClient（客户端代码）
├── Mouse → MouseServer / MouseClient
└── Yarn → YarnServer / YarnClient
```

服务器需要知道每只猫的 mHealth（权威版本）；客户端也需要本地存储 mHealth（UI显示）。服务器用虚函数和继承在必要时重写和实现新的函数。

#### 6.2.2 网络管理器和欢迎新客户端

**四步连接过程**：
1. 客户端向服务器发送 "hello" 数据包（`kHelloCC='HELO'` + 玩家名字），未收到响应前持续发送（有频率限制 `kTimeBetweenHellos`）
2. 服务器收到后分配玩家ID，创建 `ClientProxy`，发送 "welcome" 数据包（`kWelcomeCC='WLCM'` + 玩家ID）
3. 客户端收到welcome，保存玩家ID，转入 `NCS_Welcomed` 状态，开始发送复制信息并接收服务器复制信息
4. 服务器向新客户端和已有客户端发送新客户端产生的对象信息

```cpp
enum NetworkClientState { NCS_Uninitialized, NCS_SayingHello, NCS_Welcomed };

// 客户端发送和接收数据包
void NetworkManagerClient::SendOutgoingPackets() {
    switch(mState) {
        case NCS_SayingHello: UpdateSayingHello(); break;
        case NCS_Welcomed:    UpdateSendingInputPacket(); break;
    }
}
void NetworkManagerClient::ProcessPacket(InputMemoryBitStream& inInputStream, ...) {
    uint32_t packetType; inInputStream.Read(packetType);
    switch(packetType) {
        case kWelcomeCC: HandleWelcomePacket(inInputStream); break;
        case kStateCC:   HandleStatePacket(inInputStream);   break;
    }
}

// 服务器处理新客户端
void NetworkManagerServer::ProcessPacket(InputMemoryBitStream& inInputStream,
                                          const SocketAddress& inFromAddress) {
    auto it = mAddressToClientMap.find(inFromAddress);
    if(it == mAddressToClientMap.end())
        HandlePacketFromNewClient(inInputStream, inFromAddress); // 未知客户端
    else
        ProcessPacket((*it).second, inInputStream);              // 已知客户端
}
```

服务器维护 `mAddressToClientMap`（key: SocketAddress, value: ClientProxy*）跟踪所有已知客户端。

#### 6.2.3 输入共享和客户端代理

服务器是权威的，对象仅从服务器复制到客户端。客户端发送**输入数据包**（而非对象复制命令）。

```cpp
class InputState {      // 某帧客户端输入快照
    float mDesiredRightAmount, mDesiredLeftAmount;
    float mDesiredForwardAmount, mDesiredBackAmount;
    bool  mIsShooting;
};
class Move {            // InputState + 时间戳 + 增量时间
    InputState mInputState;
    float      mTimestamp;
    float      mDeltaTime;
};
```

客户端每 x 秒抓取一次当前 InputState 存为 Move，累积到 MoveList。`SendInputPacket` 发送最近3个动作（防丢失冗余），发送后清空 MoveList。

**客户端代理（ClientProxy）**：服务器为每个客户端创建，职责：
- 跟踪需要给该客户端发送的复制数据（拥有独立的 `ReplicationManagerServer`）
- 存储来自该客户端的未处理动作（`mUnprocessedMoveList`）

```cpp
class ClientProxy {
    ReplicationManagerServer mReplicationManagerServer;
    MoveList                 mUnprocessedMoveList;
    bool                     mIsLastMoveTimestampDirty;
    // ... socket地址、名字、玩家ID ...
};

// 服务器更新RoboCat（使用来自客户端的动作）
void RoboCatServer::Update() {
    RoboCat::Update();
    ClientProxyPtr client = NetworkManagerServer::sInstance->GetClientProxy(GetPlayerId());
    if(client) {
        MoveList& moveList = client->GetUnprocessedMoveList();
        for(const Move& move: moveList) {
            ProcessInput(move.GetDeltaTime(), move.GetInputState());
            SimulateMovement(move.GetDeltaTime()); // 增量时间=两次动作之间的时间，而非服务器帧时间
        }
        moveList.Clear();
    }
    HandleShooting();
}
```

> 注意：每次调用 SimulateMovement 时传入的是两次动作之间的增量时间，使服务器和客户端可以以不同帧率运行。

### 6.3 对等网络的实现：《机器猫RTS》

《机器猫RTS》（*Robo Cat RTS*）：即时战略游戏，最多4名玩家，每人3只猫，对等网络模型，所有对等体使用相同可执行文件。

使用**主对等体（master peer）**思想：提供已知对等体的IP地址，唯一能给新玩家分配玩家ID，按下回车键开始游戏。其他行为与普通对等体一致；如果主对等体断开，游戏仍然可以继续。

#### 6.3.1 欢迎新对等体和开始游戏

新对等体发送 hello 数据包，可能收到三种响应：

| 响应 | 含义 |
|------|------|
| `Welcome('WLCM')` | 主对等体接受，包含新玩家ID、主对等体ID、玩家数、所有对等体的名字和IP地址 |
| `Not joinable('NOJN')` | 游戏进行中或已满，客户端退出 |
| `Not master peer('NOMP')` | 非主对等体收到hello，回复主对等体地址让新玩家重新连接 |

收到welcome后，新对等体还需向其他所有对等体发送介绍数据包（"INTR"，包含新玩家ID和名字），保证每个对等体都知道这个新对等体。

**局域网地址问题**：对等体地址从数据包中采集，局域网内的私有IP对外部不可见，导致跨网段对等体无法互通。

解决方案：
- **集中服务器（rendezvous server）**：仅用于对等体之间的初始化通信，保证连接使用外部可访问的IP地址（图6.6b）
- **中央服务器路由**：所有对等体数据包都先经过中央服务器再路由，不暴露公网IP（更安全，可防DDoS，但需更强服务器）

主对等体按下回车键，发送开始数据包（"STRT"），所有对等体进入3秒倒计时正式开始比赛。同步延迟：每个对等体从计时器时间中减去到主对等体RTT的1/2。

#### 6.3.2 命令共享和锁步回合制

游戏以锁定30帧/秒运行，增量时间固定~33毫秒。

- **轮（turn）**：100毫秒（每秒10轮）
- **子轮（sub-turn）**：每轮3个子轮（每个~33毫秒）

不复制对象，只在网络传输**轮数据包**。命令（command）与输入（input）有清晰界限：选择一只猫不产生命令（不影响游戏状态），移动/攻击才产生命令。

**轮数据发送流程**：

```cpp
    mSubTurnNumber++;

        TurnData data(mPlayerId, RandGen::GetRandomUInt32(...), ComputeGlobalCRC(),

        data.Write(packet);
        for(auto& iter: mPlayerToSocketMap) SendPacket(packet, iter.second);
        // 保存本地轮数据并清空命令列表
        mTurnData[mTurnNumber + 2].emplace(mPlayerId, data);
        InputManager::ClearCommandList();
        if(mTurnNumber >= 0) TryAdvanceTurn();
        else { mTurnNumber++; mSubTurnNumber = 0; } // 游戏开始时轮序号-2，初始延迟
    }
}

void NetworkManager::TryAdvanceTurn() {
    // 只有收到了所有对等体下一轮数据才推进
    if(mTurnData[mTurnNumber + 1].size() == mPlayerCount) {
        if(mState == NMS_Delay) { InputManager::ClearCommandList(); mState = NMS_Playing; }
        mTurnNumber++; mSubTurnNumber = 0;
        if(CheckSync(mTurnData[mTurnNumber])) {
            // 执行所有玩家的命令
            for(auto& iter: mTurnData[mTurnNumber])
                iter.second.GetCommandList().ProcessCommands(iter.first);
        } else {
            Engine::SetShouldKeepRunning(false); // 不同步直接结束游戏
        }
    } else {
        mState = NMS_Delay; // 缺少某对等体数据，进入延迟状态
    }
}
```

游戏开始时轮序号为-2（前200毫秒无命令执行，是锁步回合制固有延迟），第-2轮发出的命令在第0轮执行。

#### 6.3.3 保持同步

**6.3.3.1 同步伪随机数生成器（PRNG）**

两个条件缺一不可：
1. 每个对等体随机数生成器设置**相同的种子**（主对等体在倒计时时生成种子，写入开始数据包）
2. 每轮在代码中**相同的位置**调用**相同次数**的PRNG

标准 `rand`/`srand` 不适合（C标准未规定算法，不同平台/编译器可能实现不同）。使用C++11的**梅森旋转算法（Mersenne Twister, MT19937）**：

```cpp
class RandGen {
public:
    static std::unique_ptr<RandGen> sInstance;
    void     Seed(uint32_t inSeed)     { mGenerator.seed(inSeed); }
    float    GetRandomFloat()          { return mFloatDistr(mGenerator); }
    uint32_t GetRandomUInt32(uint32_t inMin, uint32_t inMax) {
        return std::uniform_int_distribution<uint32_t>(inMin, inMax)(mGenerator);
    }
private:
    std::mt19937                          mGenerator;
    std::uniform_real_distribution<float> mFloatDistr;
};
```

此外，每轮结束时各对等体生成一个随机数包含在轮数据中，供其他对等体验证PRNG保持同步。

**6.3.3.2 检查游戏同步（CRC）**

使用**循环冗余检验（CRC）**在每轮结束时计算32位检验和，验证所有对等体游戏状态一致：

```cpp
uint32_t NetworkManager::ComputeGlobalCRC() {
    OutputMemoryBitStream crcStream;
    uint32_t crc = crc32(0, Z_NULL, 0);
    // 按网络ID顺序遍历所有游戏对象，写入关键状态数据（位置、生命值、状态等）
    for(auto& iter: mNetworkIdToGameObjectMap)
        iter.second->WriteForCRC(crcStream);
    crc = crc32(crc, reinterpret_cast<const Bytef*>(crcStream.GetBufferPtr()),
                crcStream.GetByteLength());
    return crc;
}
```

CRC和随机数放入轮数据包，`TryAdvanceTurn` 时 `CheckSync` 验证。不同步直接结束游戏（简单方案）；更鲁棒：投票机制（4人游戏中3票相同则去掉异常对等体）。

> 警告：开发对等网络游戏时，不同步是痛苦的根源——不同步的错误往往最难解决。务必实现日志系统，记录每个对等体执行命令的细节和每轮结束时每只猫的位置，用于定位哪个对等体过了哪个轮之后出现不同步。

### 6.4 总结

| 方面 | 客户端-服务器（《机器猫行动》） | 对等网络（《机器猫RTS》） |
|------|------|------|
| 代码结构 | 3个目标：共享库+服务器+客户端 | 1个目标：所有对等体代码相同 |
| 对象复制 | 服务器→客户端（3种行为：创建/更新/销毁） | 不复制对象，传输轮数据包 |
| 客户端输入 | 客户端发输入包，服务器模拟 | 每个对等体独立模拟所有命令 |
| 同步保证 | 服务器权威，客户端以服务器为准 | 确定性锁步+PRNG同步+CRC检验 |
| 新玩家 | 服务器分配ID，发welcome包 | 主对等体分配ID，新玩家介绍给所有人 |
| 延迟 | RTT/2（每方）+ 服务器处理 | RTT/2（对等体直接通信），有2轮固定延迟 |

## 第7章 延迟、抖动和可靠性

真实网络环境充斥着延迟、抖动和丢包。本章分析各类延迟来源，介绍 TCP 的游戏局限性，并构建一套基于 UDP 的自定义可靠投递通知系统，最后讲解如何可靠地复制对象状态。

### 7.1 非网络延迟

即使没有网络，视频游戏本身也存在多种延迟源：

| 延迟源 | 说明 | 典型值 |
|--------|------|--------|
| 输入采样延迟 | 游戏按固定帧率采样输入，玩家在帧中间的操作需等到下一帧采样 | 16.7ms（60Hz）|
| 渲染管线延迟 | CPU 提交绘制命令→GPU 渲染→写入帧缓冲区，全部在下一次显示前完成 | 22.2ms（45fps，单缓冲）|
| 多线程渲染管线 | 渲染线程比游戏线程落后1帧，导致额外延迟 | 最多2帧≈33ms（60fps）|
| 垂直同步（VSync）| 等待下一次垂直消隐期才将缓冲区内容显示到屏幕 | 0–16.7ms（60Hz）|
| 显示延迟 | LCD 显示器内部图像处理（缩放、颜色修正等） | 1–8帧 |
| 像素响应时间 | 像素从一种颜色变为另一种颜色的物理响应时间 | 约1ms（灰阶转换）|

多线程渲染管线中，渲染线程在游戏线程处理完一帧后才开始渲染，因此游戏世界状态落后渲染线程一帧，显示到屏幕上时可能落后两帧。

### 7.2 网络延迟

数据包在网络中传输经历四类延迟：

| 延迟类型 | 含义 | 说明 |
|----------|------|------|
| 处理延迟 | 路由器/交换机处理数据包头部所需时间 | 通常极短，可忽略 |
| 传输延迟 | 将数据包的所有比特推送到链路所需时间 | 包大小 / 链路带宽；1500字节在1MB/s以太网上约12.5ms |
| 排队延迟 | 数据包等待路由器转发队列的时间 | 高峰期可导致较大波动 |
| 传播延迟 | 信号在物理介质中传播的时间 | 约0.3ns/m；跨大西洋（约20000km）约66ms |

**总 RTT** = 往返两段传播延迟 + 两端各自的处理/传输/排队延迟之和。

减少延迟的对策：
- 传输延迟：压缩数据、发送更小的数据包
- 排队延迟：减少发包频率以降低队列压力
- 传播延迟：将服务器部署在靠近玩家的位置

### 7.3 抖动

**抖动（jitter）**：连续数据包之间的延迟差异，主要由排队延迟的不稳定性导致。抖动可导致数据包到达顺序错乱（序号为3的包先于序号为2的包到达）。

减少抖动：
- 减少发包数量（每个包更大，包频更低）
- 将服务器部署在靠近玩家的地理位置

### 7.4 数据包丢失

物理层、链路层、网络层均不可靠，无法保证数据包一定送达。路由器在拥塞时会丢弃低优先级数据包（QoS机制）。

**游戏面临的选择**：
- 忍受少量丢包（适合实时游戏，旧状态无需可靠传输）
- 自定义可靠传输层（适合必须保证送达的数据）

### 7.5 TCP与UDP

#### TCP 的三大游戏问题

1. **优先级干扰**：紧急数据（如玩家当前位置）排在过时数据后面，不得不等待
2. **独立流干扰**：若用多个 TCP 连接传输不同数据流，一条流的丢包会阻塞整个连接
3. **过时状态重传**：TCP 重传丢失的数据包时，内容可能已过时，既占带宽又增加延迟

**Nagle 算法问题**：TCP 默认开启 Nagle 算法，等待缓冲区积累足够数据再发送，造成额外延迟（尽管可用 `TCP_NODELAY` 禁用）。

**UDP**：空白石板（blank slate），完全由应用层控制，没有上述限制，是游戏实时网络的首选。

#### 7.5.1–7.5.5 投递通知管理器（DeliveryNotificationManager）

灵感来自 *Starsiege: Tribes*，提供对每个 UDP 数据包投递状态的通知。

**核心数据结构**：

```cpp
class DeliveryNotificationManager {
    PacketSequenceNumber mNextOutgoingSequenceNumber;   // 下一个发出的序列号
    PacketSequenceNumber mNextExpectedSequenceNumber;   // 下一个期待收到的序列号
    std::deque<InFlightPacket> mInFlightPackets;        // 已发送、等待确认的包
    std::list<AckRange>        mPendingAcks;            // 待发送的确认范围
};
```

**标记发出数据包（WriteSequenceNumber）**：

```cpp
InFlightPacket* DeliveryNotificationManager::WriteSequenceNumber(OutputMemoryBitStream& inPacket) {
    PacketSequenceNumber sequenceNumber = mNextOutgoingSequenceNumber++;
    inPacket.Write(sequenceNumber);
    ++mDispatchedPacketCount;
    mInFlightPackets.emplace_back(sequenceNumber);
    return &mInFlightPackets.back();
}
```

**处理收到数据包序列号（ProcessSequenceNumber）**，三种情况：

```cpp
bool DeliveryNotificationManager::ProcessSequenceNumber(InputMemoryBitStream& inPacket) {
    PacketSequenceNumber sequenceNumber;
    inPacket.Read(sequenceNumber);
    if(sequenceNumber == mNextExpectedSequenceNumber) {
        // 正常：处理并确认
        mNextExpectedSequenceNumber = sequenceNumber + 1;
        AddPendingAck(sequenceNumber);
        return true;
    } else if(sequenceNumber < mNextExpectedSequenceNumber) {
        // 旧包（重复或乱序），丢弃
        return false;
    } else {
        // 跳号：中间序号的包丢失，跳过并确认当前
        mNextExpectedSequenceNumber = sequenceNumber + 1;
        AddPendingAck(sequenceNumber);
        return true;
    }
}
```

**确认范围（AckRange）**：压缩连续序列号，节省带宽：

```cpp
class AckRange {
    PacketSequenceNumber mStart;
    uint32_t             mCount;
public:
    inline bool ExtendIfShould(PacketSequenceNumber inSequenceNumber) {
        if(inSequenceNumber == mStart + mCount) { ++mCount; return true; }
        return false;
    }
    void Write(OutputMemoryBitStream& inPacket) const; // 写入 mStart + (mCount>1 ? 1bit+mCount : 1bit)
    void Read(InputMemoryBitStream& inPacket);
};
```

**写入待确认列表（WritePendingAcks）**：每次发包时捎带一个 AckRange：

```cpp
void DeliveryNotificationManager::WritePendingAcks(OutputMemoryBitStream& inPacket) {
    bool hasAcks = (mPendingAcks.size() > 0);
    inPacket.Write(hasAcks);
    if(hasAcks) {
        mPendingAcks.front().Write(inPacket);
        mPendingAcks.pop_front();
    }
}
```

**处理收到的确认（ProcessAcks）**：遍历 `mInFlightPackets`，对落在 AckRange 内的调用 `HandleDeliverySuccess`，跳过的（已超出范围前面的）调用 `HandleDeliveryFailure`。

**超时包处理（ProcessTimedOutPackets）**：从 `mInFlightPackets` 队头开始，对超过 `kAckTimeout` 仍未确认的包调用 `HandleDeliveryFailure`。

#### 7.5.6 InFlightPacket 与 TransmissionData

```cpp
class InFlightPacket {
public:
    void SetTransmissionData(int inKey, TransmissionDataPtr inTransmissionData) {
        mTransmissionDataMap[inKey] = inTransmissionData;
    }
    const TransmissionDataPtr GetTransmissionData(int inKey) const {
        auto it = mTransmissionDataMap.find(inKey);
        return (it != mTransmissionDataMap.end()) ? it->second : nullptr;
    }
    void HandleDeliveryFailure(DeliveryNotificationManager* inManager) const {
        for(const auto& pair: mTransmissionDataMap)
            pair.second->HandleDeliveryFailure(inManager);
    }
    void HandleDeliverySuccess(DeliveryNotificationManager* inManager) const {
        for(const auto& pair: mTransmissionDataMap)
            pair.second->HandleDeliverySuccess(inManager);
    }
private:
    PacketSequenceNumber mSequenceNumber;
    float mTimeDispatched;
    std::unordered_map<int, TransmissionDataPtr> mTransmissionDataMap;
};
```

**TransmissionData**：抽象基类，每个依赖模块（如 ReplicationManager）继承它，在写数据包时创建子类实例并通过 `SetTransmissionData` 附加到 `InFlightPacket`，投递成功/失败时由回调通知模块处理。

> 注：每对通信主机需要各自独立的 DeliveryNotificationManager。服务器与10个客户端通信，需要10个 DeliveryNotificationManager，每个客户端一个。

### 7.6 对象复制可靠性

扩展第5章的 ReplicationManager，支持基于 DeliveryNotificationManager 的可靠状态重传。灵感同样来自 *Starsiege: Tribes* 的 ghost 管理器。

**核心思想**：不重传原始数据包内容，而是重传**最新版本**的状态。丢包后，重传的是目标对象在重传时刻的最新状态，而不是发包时刻的旧状态。

#### 批量复制命令接口

```cpp
// 游戏系统通知 ReplicationManager 哪些对象发生了什么变化
void ReplicationManager::BatchCreate(int inNetworkId, uint32_t inInitialDirtyState);
void ReplicationManager::BatchDestroy(int inNetworkId);
void ReplicationManager::BatchStateDirty(int inNetworkId, uint32_t inDirtyState);
```

**ReplicationCommand** 内部结构：

```cpp
ReplicationCommand::ReplicationCommand(uint32_t inInitialDirtyState)
    : mAction(RA_Create), mDirtyState(inInitialDirtyState) {}

void ReplicationCommand::AddDirtyState(uint32_t inState) { mDirtyState |= inState; }
void ReplicationCommand::SetDestroy()                    { mAction = RA_Destroy; }
void ReplicationCommand::ClearDirtyState(uint32_t inStateToClear) {
    mDirtyState &= ~inStateToClear;
    if(mAction == RA_Destroy) mAction = RA_Update;
}
bool ReplicationCommand::HasDirtyState() const {
    return (mAction == RA_Destroy) || (mDirtyState != 0);
}
```

`mNetworkReplicationCommand`：`map<networkId, ReplicationCommand>`，存储每个对象的最新复制命令。

#### 写批量命令到数据包（WriteBatchedCommands）

```cpp
void ReplicationManager::WriteBatchedCommands(
    OutputMemoryBitStream& inStream, InFlightPacket* inFlightPacket)
{
    ReplicationManagerTransmissionDataPtr repTransData = nullptr;
    for(auto& pair: mNetworkIdToReplicationCommand) {
        ReplicationCommand& replicationCommand = pair.second;
        if(replicationCommand.HasDirtyState()) {
            int networkId   = pair.first;
            GameObject* gameObj = mLinkingContext->GetGameObject(networkId);
            if(gameObj) {
                ReplicationAction action = replicationCommand.GetAction();
                ReplicationHeader rh(action, networkId, gameObj->GetClassId());
                rh.Write(inStream);
                uint32_t dirtyState = replicationCommand.GetDirtyState();
                if(action == RA_Create || action == RA_Update)
                    gameObj->Write(inStream, dirtyState);
                // 首次才创建 TransmissionData
                if(!repTransData) {
                    repTransData = std::make_shared<ReplicationManagerTransmissionData>(this);
                    inFlightPacket->SetTransmissionData('RPLM', repTransData);
                }
                // 记录本次写入的内容
                repTransData->AddReplication(networkId, action, dirtyState);
                replicationCommand.ClearDirtyState(dirtyState);
            }
        }
    }
}
```

#### 响应投递通知（HandleDeliveryFailure / HandleDeliverySuccess）

```cpp
// 投递失败：重新将丢失数据加入批处理
void ReplicationManagerTransmissionData::HandleDeliveryFailure(...) {
    for(const ReplicationTransmission& rt: mReplications) {
        int networkId = rt.GetNetworkId();
        switch(rt.GetAction()) {
            case RA_Create:
                if(mReplicationManager->GetLinkingContext()->GetGameObject(networkId))
                    mReplicationManager->BatchCreate(networkId, rt.GetState());
                break;
            case RA_Update:
                if(mReplicationManager->GetLinkingContext()->GetGameObject(networkId))
                    mReplicationManager->BatchStateDirty(networkId, rt.GetState());
                break;
            case RA_Destroy:
                mReplicationManager->BatchDestroy(networkId);
                break;
        }
    }
}

// 投递成功：更新状态（RA_Create 确认后可改用 Update 发送）
void ReplicationManagerTransmissionData::HandleDeliverySuccess(...) {
    for(const ReplicationTransmission& rt: mReplications) {
        switch(rt.GetAction()) {
            case RA_Create:
                mReplicationManager->HandleCreateAckd(rt.GetNetworkId());
                break;
            case RA_Destroy:
                mReplicationManager->RemoveFromReplication(rt.GetNetworkId());
                break;
        }
    }
}
```

#### 避免多余重传（优化）

当 RA_Update 丢失需重传时，检查所有在途数据包（in-flight packets），若已有更新版的该对象状态在途中，则从脏状态中减去在途的状态位，只重传真正缺失的部分：

```cpp
// HandleDeliveryFailure 中 RA_Update 的优化处理
uint32_t state = rt.GetState();
for(const auto& inFlightPacket: inDeliveryNotificationManager->GetInFlightPackets()) {
    auto rmtdp = std::static_pointer_cast<ReplicationManagerTransmissionData>(
        inFlightPacket.GetTransmissionData('RPLM'));
    if(rmtdp) {
        for(const ReplicationTransmission& otherRT: rmtdp->mReplications)
            if(otherRT.GetNetworkId() == networkId)
                state &= ~otherRT.GetState(); // 已在途的状态无需重传
    }
}
if(state) mReplicationManager->BatchStateDirty(networkId, state);
```

### 7.7 模拟真实世界的条件

在套接字和游戏之间插入 **RLSimulator**，模拟丢包、延迟和抖动：

```cpp
void RLSimulator::ReadIncomingPacketsIntoQueue() {
    // ... 接收数据包 ...
    // 随机丢包
    if(RoboMath::GetRandomFloat() >= mDropPacketChance) {
        // 计算模拟到达时间：当前时间 + 平均延迟 + 随机抖动
        float simulatedReceivedTime =
            Timing::sInstance.GetTimef()
            + mSimulatedLatency
            + (RoboMath::GetRandomFloat() - 0.5f) * mDoubleSimulatedMaxJitter;
        // 插入按模拟到达时间排序的列表
        mPacketList.emplace(it, simulatedReceivedTime, inputStream, fromAddress);
    }
}

void RLSimulator::ProcessQueuedPackets() {
    float currentTime = Timing::sInstance.GetTimef();
    while(!mPacketList.empty()) {
        ReceivedPacket& packet = mPacketList.front();
        if(currentTime > packet.GetReceivedTime()) {
            ProcessPacket(packet.GetInputStream(), packet.GetFromAddress());
            mPacketList.pop_front();
        } else break;
    }
}
```

> 小窍门：丢包和延迟在现实中往往连续发生。当随机决定一个包应该丢弃时，可再用另一个随机数决定它影响了后面多少个包，模拟突发丢包场景。

| 问题 | 解决方案 |
|------|----------|
| 非网络延迟（输入/渲染/显示） | 了解来源，针对性优化（VSync、多线程渲染等） |
| 传输延迟 | 压缩数据，减少发包大小 |
| 传播延迟 | 服务器靠近玩家部署 |
| 抖动 | 减少发包频率，靠近玩家部署 |
| 丢包 | 自定义可靠层（DeliveryNotificationManager） |
| TCP的游戏缺陷 | 改用 UDP，自主控制可靠性策略 |
| 对象状态可靠传输 | ReplicationManager + DeliveryNotificationManager，重传最新状态而非原始状态 |
| 本地测试网络条件 | RLSimulator 模拟丢包/延迟/抖动 |

## 第8章 改进的延迟处理

延迟是多人游戏开发者的天敌。本章探讨客户端插值、客户端预测、移动重放和服务器端回退等技术，将延迟对玩家体验的影响降到最低。

### 8.1 沉默的客户终端

Tim Sweeney 在 Unreal 网络系统中有一句名言："服务器就是那个人"——服务器是唯一拥有真实和正确游戏状态的主机。

在最朴素的客户端-服务器实现中，客户端只负责发送输入、接收状态、显示给用户，不运行任何模拟，被称为**沉默的客户终端（dumb terminal）**，这种方式称为**保守算法（conservative algorithm）**：绝对不会显示错误状态。

**延迟计算**：玩家按下按键 → 1/2 RTT 到达服务器 → 服务器模拟 → 1/2 RTT 状态返回客户端 = RTT 总延迟。

**问题**：即使服务器和客户端都以60帧运行，若服务器受带宽限制每4帧才发一次状态包（每秒15个包），Z轴位置必须渲染4帧才能等到下一次更新，玩家实际感受到的只有每秒15帧的游戏速度。

### 8.2 客户端插值

**客户端插值（client side interpolation）**：客户端收到对象新状态时，不立即跳到新位置，而是用**本地感知过滤器（local perception filter）**根据时间平滑地插值过渡。

关键参数：
- **IP（interpolation period）**：插值周期（毫秒），客户端从旧状态插值到新状态所需时间
- **PP（packet period）**：数据包周期（毫秒），服务器两次发包之间的间隔

要求：IP ≥ PP（否则插值在新包到达前就停止，玩家感觉卡顿）。最优值：**IP = PP**。

感知延迟 = 1/2 RTT + IP。客户端插值**仍是保守算法**：只在已知服务器状态之间插值，不会产生错误状态。

> 特例：相机操控不影响游戏模拟，客户端可本地立即处理，无需等待服务器响应。

### 8.3 客户端预测

客户端插值仍让玩家落后至少 1/2 RTT。**客户端预测（client side prediction）**通过在客户端运行与服务器相同的模拟代码，向前推测 1/2 RTT，使客户端状态更接近服务器当前真实状态。

#### 8.3.1 航位推测法

**航位推测法（dead reckoning）**：基于"实体继续做当前正在做的事情"这一假设进行预测——跑步的玩家继续以同方向奔跑，转弯的飞机继续转弯。

- 适用于**远程玩家**：客户端无法知道远程玩家的输入，只能假设行为不变
- 属于**乐观算法（optimistic algorithm）**：能猜对大部分情况，但有时完全错误，必须纠正
- 对比保守算法：保守算法绝不显示错误状态，乐观算法可能错但更流畅

**预测失败时的三种纠正方式**：

| 方法 | 描述 | 适用场景 |
|------|------|----------|
| 即时状态更新 | 立即跳到正确状态，可能出现视觉跳变 | 快节奏游戏的大错误 |
| 插值 | 在若干帧内平滑过渡到正确状态 | 快节奏游戏的小错误 |
| 二阶状态调整 | 调整速度/加速度参数，使模拟平滑收敛到正确状态 | 慢节奏游戏（飞机、大型机器人）|

#### 8.3.2 客户端移动预测和重放

航位推测无法为**本地玩家**隐藏延迟。更好的方案：将本地玩家输入直接用于客户端模拟，同时保留输入历史，收到服务器确认后**重放（replay）**未被确认的移动：

1. 玩家按键 → 客户端立即模拟
2. 输入发送到服务器（携带时间戳）
3. 服务器处理输入，将最新处理的移动时间戳附带在状态包里发回
4. 客户端收到状态包：计算 RTT，删除已确认移动，从服务器状态向前重放剩余未处理移动

**RTT计算**（图8.4）：客户端在数据包中写入本地时钟时间戳，服务器原样回传，客户端用当前时间减去收到的时间戳得到精确 RTT。

```cpp
// 服务器：将最新处理移动的时间戳写回客户端
void NetworkManagerServer::WriteLastMoveTimestampIfDirty(
    OutputMemoryBitStream& inOutputStream, ClientProxyPtr inClientProxy)
{
    bool isTimestampDirty = inClientProxy->IsLastMoveTimestampDirty();
    inOutputStream.Write(isTimestampDirty);
    if(isTimestampDirty) {
        inOutputStream.Write(
            inClientProxy->GetUnprocessedMoveList().GetLastMoveTimestamp());
        inClientProxy->SetIsLastMoveTimestampDirty(false);
    }
}

// 客户端：读取服务器回显时间戳，计算 RTT，删除已处理移动
void NetworkManagerClient::ReadLastMoveProcessedOnServerTimestamp(
    InputMemoryBitStream& inInputStream)
{
    bool isTimestampDirty;
    inInputStream.Read(isTimestampDirty);
    if(isTimestampDirty) {
        inPacketBuffer.Read(mLastMoveProcessedByServerTimestamp);
        mLastRoundTripTime = Timing::sInstance.GetFrameStartTime()
                             - mLastMoveProcessedByServerTimestamp;
        InputManager::sInstance->GetMoveList()
            .RemovedProcessedMoves(mLastMoveProcessedByServerTimestamp);
    }
}
```

**发送移动时携带冗余**：每个数据包附带最近3个移动，提高可靠性（即使一个包丢失，下个包仍会重传这些移动）。

**重放代码（RoboCat::Read）**：

```cpp
void RoboCat::Read(InputMemoryBitStream& inInputStream) {
    // 保存旧状态，读取新状态...
    bool isLocalPlayer = (GetPlayerId() == NetworkManagerClient::sInstance->GetPlayerId());
    if(isLocalPlayer)
        DoClientSidePredictionAfterReplicationForLocalCat(readState);   // 移动重放
    else

}

// 本地猫：重放剩余未处理的移动（推测额外 1/2 RTT）
void RoboCatClient::DoClientSidePredictionAfterReplicationForLocalCat(uint32_t inReadState) {
    if((inReadState & ECRS_Pose) != 0) {
        const MoveList& moveList = InputManager::sInstance->GetMoveList();
        for(const Move& move: moveList) {
            ProcessInput(move.GetDeltaTime(), move.GetInputState());
            SimulateMovement(move.GetDeltaTime());
        }
    }
}

// 远程猫：航位推测 1/2 RTT（以帧大小为步长，避免穿墙等物理异常）
void RoboCatClient::DoClientSidePredictionAfterReplicationForRemoteCat(uint32_t inReadState) {
    if((inReadState & ECRS_Pose) != 0) {
        float rtt = NetworkManagerClient::sInstance->GetRoundTripTime();
        float deltaTime = 1.f / 30.f;
        while(true) {
            if(rtt < deltaTime) { SimulateMovement(rtt); break; }
            else { SimulateMovement(deltaTime); rtt -= deltaTime; }
        }
    }
}
```

#### 8.3.3 通过技巧和优化隐藏延迟

几乎所有视频游戏动作都有**通知（tell）**：开枪前枪口闪光，施法前双手摆动。这些通知持续约一个往返时间。

利用方式：客户端收到玩家输入时，立即播放动画和声音；服务器收到输入包，复制给客户端，正赶上显示施法结果。若服务器拒绝（如玩家被沉默），动画开始但没有出现抛射物——这种情况常见，与该方法带来的好处相比可以容忍。

### 8.4 服务器端回退

**问题**：长距离即时射击（狙击步枪）。玩家在客户端精准瞄准，但由于航位推测的不精确性，在服务器端可能偏差太大——目标100毫秒之前的位置并不是当前位置。

**Valve Source Engine 的解决方案（Counter-Strike 等采用）**：当瞄准和开火时，让服务器状态**回退（rewind）**到玩家感受到的那个状态。

实现要求：

1. **远程玩家使用客户端插值，而不是航位推测**：服务器需精确知道客户端每时刻看到了什么
2. **本地玩家保留移动预测和重放**
3. **每个输入数据包记录客户端当前插值的两帧 ID 及插值进度百分比**
4. **服务器端存储每个相关对象最近几帧的位置**
5. **收到含射击的输入包时**：将相关对象回退到客户端扣扳机那一刻的插值位置，然后从客户端的位置用光线投射确定是否击中

**优势**：射击玩家获得准确的命中确认体验（所见即所中）。
**不足**：被击中的玩家可能已经移动到安全位置，仍被判定击中——"绕角被击中"现象。

### 8.5 总结

| 技术 | 算法类型 | 延迟感知 | 适用对象 |
|------|----------|----------|----------|
| 沉默客户终端 | 保守 | RTT | 所有对象 |
| 客户端插值 | 保守 | RTT + IP | 远程对象 |
| 航位推测 | 乐观 | ≈0（大部分情况）| 远程对象 |
| 移动预测 + 重放 | 乐观 | 0（本地感受）| 本地玩家 |
| 技巧/通知 | — | 隐藏 | 技能/动作 |
| 服务器端回退 | — | 0（射手视角）| 即时射击武器 |

## 第9章 可扩展性

扩大网络游戏规模会引入小规模游戏不存在的新挑战。本章介绍游戏规模增大时突然出现的问题及解决方案。

### 9.1 对象范围和相关性

**范围（scope）/ 相关性（relevancy）**：对于特定客户端，一个对象是否应该接收关于该对象的更新信息。

小规模游戏中，所有对象对所有客户端都是相关的。大规模游戏中，这种方法既不切实际（带宽），也不必要——64人游戏中，几公里外的玩家信息毫无意义。

> 注意：对象相关性在商业游戏中很少完全是二元的。不同对象根据其优先级可能有不同的更新频率；本节以二元形式讨论，实际应用更复杂。

#### 9.1.1 静态区域

**静态区域（static zone）**：将世界划分为固定区域，只有与玩家在同一区域内的对象才是相关的。主要用于共享世界的游戏（MMORPG），如小镇是一个区域，森林是另一个区域。

**优点**：简单清晰，有效减少相关对象数量。

**缺点与问题**：
- 区域边界过渡延迟（需要加载画面）
- 玩家分布不均匀（热门区域拥挤）：Asheron's Call 用传送机制应对；区域内玩家数量上限后分割为多个实例

**不适用于动作游戏**：快速穿越区域边界引起的延迟不可接受。

#### 9.1.2 使用视锥

**视锥（view frustum）**：3D游戏中投影到2D图像中被显示区域的梯形棱柱，由水平视野角度、宽高比、近平面和远平面距离表示。

视锥裁剪用于可见性裁剪，也可用于网络游戏对象相关性判断。

**问题**：快速旋转180°时，身后所有对象突然变为不相关，服务器需时间发送突然进入范围的对象更新，造成不可接受的延迟（尤其当背后对象恰好是敌人时）。

**解决方案**：视锥 + 更小半径组合——在较小半径之内**或**视锥范围内的对象都认为是相关的（图9.3）。

#### 9.1.3 其他可见性技术

**赛车游戏（道路可见距离）**：直道能见距离远，转弯能见距离短；服务器知道玩家汽车的位置，可以知道玩家看出去多远，该区域很可能比赛道与视锥的交叉区域小得多。

**潜在可见集（potentially visible set，PVS）**：
- 核心问题：从世界每个位置，哪些区域是潜在可见的？
- 静态区域可能是整个小镇，PVS 区域可能仅是一间房间
- 典型实现：世界划分为凸多边形集合；离线为每个凸多边形计算潜在可见的其他多边形集合；运行时服务器确定玩家在哪个凸多边形，查找对应 PVS 确定相关对象
- 适用于《毁灭战士》（Doom）、《雷神之锤》（Quake）类走廊型 FPS
- **传送门（portal）裁剪系统**：每间屋子是区域，每扇门或窗是传送门；可大大降低相关对象数量，但需要更多运行时处理
- **树状裁剪**（BSP/四叉树/八叉树）：更高处理成本，一般动作游戏 PVS 已足够

#### 9.1.4 不可见时的相关性

可见性并非始终是相关性的唯一判断标准。例如：玩家可以扔手榴弹的 FPS——手榴弹在附近房间爆炸时，必须将爆炸复制给附近客户端，即使手榴弹对它们不可见（客户端希望听到爆炸声）。

解决方案：
- 使用**半径**（而非可见性）判断手榴弹是否被复制
- 通过 **RPC** 将爆炸特效发送给没有看到手榴弹的客户端（减少带宽，不需要复制实际手榴弹对象）

商业游戏中，声音遮蔽信息通常在客户端计算，半径方法或 RPC 方法适用于大多数游戏。

### 9.2 服务器分区

**服务器分区（server partitioning）/ 分片（sharding）**：同时运行多个独立服务器进程。大多数动作游戏自然使用此方法——每个动作游戏每次游戏的活跃玩家数量有上限（通常 8-16 人）。

代表游戏：《使命召唤》、《英雄联盟》、《战地》。各服务器进程独立，玩家统计数据写入共享数据库。

**多进程管理**：
- 主进程（main process）：决定什么时候在哪台机器上创建服务器进程；游戏结束时，服务器进程写永久性数据后退出；新比赛开始时，主进程找最小负载机器并创建新进程
- 一台机器可同时运行多个服务器进程；服务器可使用云托管（第13章详述）

**MMO 扩展**：服务器分区也用于 MMO 静态区域的扩展——每个静态区域或集合可以作为独立服务器进程运行。《魔兽世界》每块大陆运行在独立服务器进程上，跨大陆时显示加载画面。

**拥挤问题的处理**：
- 设置服务器上限，玩家排队等待
- 《星际舰夜》（Eve Online）**时间膨胀（time dilation）**：服务器减慢游戏时间步长，允许服务器将所有玩家保持在连接状态

### 9.3 实例化

**实例化（instancing）**：一个共享的游戏同时支持多个不同的实例。玩家群体可以体验高度定制化内容，不受其他玩家影响。

- MMORPG 副本（dungeon）：为固定数量玩家设计的地域使用实例化；传送门或类似结构将玩家从共享区域转到实例
- 解决拥挤：《星球大战：旧共和国》在特定区域设置玩家数量上限，超限后分割出第二个实例；同一区域内两个朋友可能在不同实例，解决方案是允许玩家移动到组员的实例

**优缺点**：
- 优点：内容更像单机/小型多人游戏，可控性强；结合服务器分区可进一步降低服务器负载
- 缺点：降低了共享世界的感觉

### 9.4 优先级和频率

在对象复制层面减少带宽占用的两种技术：

**优先级（priority）**：给不同对象赋予不同复制优先级。高优先级对象先复制；低优先级对象只有在高优先级对象复制完之后才能复制。**动态优先级**：低优先级对象长时间未更新时，提升其优先级，防止永远得不到更新。

**频率（frequency）**：重要的对象每秒更新几次；不重要的对象可能几秒才更新一次。可与优先级结合——低优先级对象通过较低的复制频率实现。

优先级也适用于 RPC：如果特定 RPC 与游戏状态无关，带宽不足时可在传输中丢弃。

### 9.5 总结

| 技术 | 说明 | 适用场景 |
|------|------|----------|
| 距离检查 | 简单快速，单纯距离有局限性 | 所有游戏的基础筛选 |
| 静态区域 | 世界划分为固定区域 | 共享世界（MMORPG）|
| 视锥 + 小半径 | 视锥与近距离半径组合 | FPS、动作游戏 |
| PVS | 预计算潜在可见集 | 走廊型 FPS（Doom/Quake）|
| 传送门系统 | 门/窗作为传送门的可见性裁剪 | 走廊型游戏 |
| 服务器分区 | 多个独立服务器进程并行 | 动作游戏、MMO 大陆分区 |
| 实例化 | 同一内容多个独立实例 | MMORPG 副本、拥挤区域 |
| 优先级 + 频率 | 限制复制带宽，重要对象优先 | 所有在带宽压力下的游戏 |

## 第10章 安全性

自从第一个网络游戏出现以来，玩家就在设计获得不公平优势的方法。本章介绍常见的漏洞及预防措施。

### 10.1 数据包嗅探

数据包在网络中经过多台路由器转发，任何中间节点理论上都可以读取数据包内容。深度包检测（deep packet inspection）是合法的 QoS 机制；恶意的**数据包嗅探（packet sniffing）**则用于窃取登录信息或在游戏中作弊。

#### 10.1.1 中间人攻击

**中间人攻击（man-in-the-middle attack）**：源和目的之间的某台计算机读取数据，而源主机和目的主机都不知道。可发生于公共 WiFi、恶意软件或 ISP 监控等场景。

**对策：公钥加密**。推荐使用 RSA（Rivest、Shamir、Adleman，1977年设计）：
- 通信双方各生成一对私钥和公钥，私钥保密不共享
- Alice 用 Bob 的公钥加密消息，只有 Bob 的私钥能解密
- 基于**半素数（semiprime）**（两个大素数之积）的数学难题，目前无法在多项式时间内分解1024位以上的数

> 使用 **OpenSSL** 等成熟开源实现，不要自己实现 RSA 系统。

若游戏只传输复制数据（无金融敏感数据），加密优先级较低，但防主机数据包嗅探仍有意义。

#### 10.1.2 在主机上的数据包嗅探

每个平台的游戏都可以执行的文件都可以被破解，加密游戏数据不能防止有人学习如何解密数据。

**对策（降低难度，而非完全防止）**：
- 逆向编写解密代码并找到存储在客户端的私钥需要时间
- 定期更换密钥及其内存偏移
- 定期改变数据包的格式和顺序，迫使依赖特定格式的欺骗工具失效

**信息作弊（information cheat）**：玩家收集不应该知道的信息（如隐身角色的位置）。最好对策：服务器只发送与每个客户端相关的数据。

### 10.2 输入验证

**输入验证（input validation）**：确保用户执行的动作在当时是有效的，再执行该动作。适用于客户端-服务器和对等网络拓扑。

例：射击请求应验证：玩家有武器、武器有子弹、武器不在冷却中。同时验证该操作确实是由正确玩家的客户端发出的（第6章 RoboCat 中两个版本均已执行此验证）。

**假阳性问题**：无效的输入可能是偶然的（延迟或数据包丢失造成）。例如，玩家被"沉默"了，但沉默更新包还未到达时玩家发起施法——应将玩家踢出视为错误的处理方式。

在**权威服务器模型**中，客户端无法验证来自服务器的命令是否真实来自客户端B（服务器可能发来虚假数据）。最根本的解决方案是使用云托管服务器，不让玩家自己主持游戏。

**对等网络的验证**：为增加额外验证层，客户端B抛球时同时广播给所有其他客户端；客户端A收到服务器的抛球数据时，可以验证是否与客户端B的直接广播一致。但该方案不总有效（NAT/防火墙可能阻止直接通信，且服务器包可能先到达）。

### 10.3 软件作弊检测

**软件作弊检测（software cheat detection）**：作为游戏进程的一部分或进程外，主动监测游戏完整性。可检测：
- **地图黑客（map hacking）**：在锁步对等网络中，每个对等体内存中存储所有单元位置，作弊程序删除战争迷雾。软件作弊检测可检测到迷雾是否被删除
- **机器人外挂（bot）**：自动化 MMO 升级/刷钱，或 FPS 完美自动瞄准

#### 10.3.1 VAC（Valve Anti-Cheat）

- 为每个游戏维护一个被禁用户列表
- 扫描游戏进程内存，检测已知作弊程序
- **禁止波（ban wave）**：不立即禁止，而是延迟到某一时刻批量禁止——让作弊者误以为作弊程序可用，一次性禁止更多人
- **纯服务器（pure server）**：服务器预知客户端上存在的所有文件的检验和，玩家加入时验证文件一致性

#### 10.3.2 典狱长反作弊系统（Warden，Blizzard）

- 暴雪为旗下所有游戏使用，扫描计算机内存检测已知作弊程序
- **战术优势**：在游戏运行时可以更新 Warden——作弊用户通常不知道 Warden 已经更新，补丁发布后不会立即使用作弊程序，甚至使用了也几乎不会被发现

### 10.4 保护服务器

#### 10.4.1 DDoS（分布式拒绝服务攻击）

**DDoS（distributed denial-of-service attack）**：目的是让服务器无法完成请求，最终使服务器不可达或对合法用户不可用——输入太多请求让服务器网络接口饱和，或耗尽处理能力。几乎每一个主流网络游戏都经历过 DDoS。

对策：与 ISP 密切合作、硬件升级、流量分配；云托管提供一定 DDoS 防护，但仍需规划测试。

#### 10.4.2 坏数据

恶意用户可能发送格式不正确的数据包，试图让服务器崩溃或触发缓冲区溢出执行恶意代码。

对策：**模糊测试（fuzz testing）**——向服务器发送大量非结构化随机数据，看是否导致崩溃，并修复发现的错误。推荐同时使用充分随机化的数据和更有结构化的数据（如包含预期签名的数据包，载荷随机）。

#### 10.4.3 时序攻击

**时序攻击（timing attack）**：通过分析处理特定哈希/签名所花费的时间，推断其实现细节。

例：简单比对函数在第一个不匹配的索引就提前返回 `false`——攻击者可通过测量每个 `b[0]` 值导致 Compare 函数返回时间长短，确定正确值，逐索引推导完整证书。

**修复**：恒定时间比较——对数组每个元素执行异或，将结果 OR 累积，最后判断是否为零：

```cpp
bool Compare(int a[8], int b[8]) {
    int retVal = 0;
    for (int i = 0; i < 8; ++i)
        retVal |= a[i] ^ b[i];
    return retVal == 0;
}
```

无论哪个元素不匹配，函数执行时间相同，攻击者无法通过时序推断数据。

#### 10.4.4 入侵

服务器安全最大的噩梦是被恶意用户闯入——目的可能是窃取用户数据、信用卡号和密码，或毁掉整个游戏数据库。

**防御措施**：
- 保持服务器所有软件最新（OS、数据库、自动化工具等）
- 限制开发人员机器访问关键服务器（防止鱼叉式钓鱼攻击 spear phishing attack）
- 假设服务器会被攻破：以适当的密码哈希算法（如**bcrypt**）对密码进行哈希，不要使用 SHA-256/MD5/DES（易被现代机器攻破）；以密码安全标准存储信用卡等账户信息（PCI DSS）
- 所有重要数据定期备份到线下物理设备

### 10.5 总结

| 威胁 | 对策 |
|------|------|
| 中间人攻击（网络传输） | 公钥加密（RSA + OpenSSL）加密敏感数据 |
| 主机数据包嗅探 | 定期更换密钥和数据包格式；只发相关数据 |
| 无效/恶意输入 | 输入验证（服务器端验证所有命令）|
| 信息作弊（地图黑客）| 服务器只发送相关数据；软件作弊检测 |
| 机器人外挂 | 软件作弊检测（VAC/Warden 模式）|
| DDoS | ISP 合作、云托管、流量分配 |
| 坏数据/崩溃攻击 | 模糊测试（fuzz testing）|
| 时序攻击 | 恒定时间比较（XOR 累积法）|
| 服务器入侵 | 软件更新、访问限制、数据加密、离线备份 |

## 第11章 真实世界的引擎

本章介绍两款主流商业引擎的多人网络实现：Unreal Engine 4 和 Unity UNET。

---

### 11.1 Unreal Engine 4

#### 网络架构

- 底层套接字：`ISSocketSubsystem`（跨平台抽象）→ `UNetDriver`（传输层驱动）
- 网络模型：**只支持客户端-服务器**（专用服务器 + 监听服务器），无内置对等体支持

#### Actor 层次

```
AActor
 └─ APawn（可被 Controller 控制的实体）
     └─ AController
         ├─ APlayerController（人类玩家）
         └─ AAIController（AI）
```

#### Actor 复制

- `AActor::bReplicates = true` 启用复制
- `bAlwaysRelevant = true` 对所有客户端始终可见
- 相关性（relevancy）决定哪些 Actor 被复制给哪些客户端

#### 角色（Role）系统

| 角色 | 含义 |
|------|------|
| `ROLE_Authority` | 权威端（服务器持有），对该 Actor 有完整控制权 |
| `ROLE_SimulatedProxy` | 模拟代理（其他客户端），接收服务器镜像状态 |
| `ROLE_AutonomousProxy` | 自主代理（当前游戏实例直接控制的 Actor），可自行发送输入 |

- `bReplicateMovement`：启用后自动同步位置/旋转/速度

#### Blueprint 系统

- 可视化脚本系统，与 C++ 共存
- 网络逻辑（RPC、变量复制）在 Blueprint 和 C++ 中均可使用

#### RPC（远程过程调用）

| 类型 | 方向 | 用途 |
|------|------|------|
| Server 函数 | 客户端 → 服务器 | 客户端发送输入/请求 |
| Client 函数 | 服务器 → 拥有该 Actor 的客户端 | 只通知特定客户端 |
| Multicast 函数 | 服务器 → 所有客户端 | 广播事件（如爆炸特效）|

- 每个 RPC 都有 `Reliable`（可靠）/ `Unreliable`（不可靠）标志
- `Reliable` RPC 保证到达，适合重要事件；`Unreliable` 适合频繁更新

---

### 11.2 Unity UNET

#### API 层次

- **高层 API（HLAPI）**：NetworkManager、NetworkBehaviour、[SyncVar]、[Command]、[ClientRpc]
- **低层传输 API（LLAPI）**：直接操作传输通道，更灵活

#### QosType（服务质量类型）

| 枚举值 | 含义 |
|--------|------|
| `Unreliable` | 不可靠，无序 |
| `UnreliableSequenced` | 不可靠，有序（丢弃旧包）|
| `Reliable` | 可靠，有序 |
| `ReliableFragmented` | 可靠，支持大包分片 |

#### NetworkManager

- 三种模式：**主机（Host）**、**客户端（Client）**、**服务器（Server）**
- 管理连接生命周期、场景同步、玩家预制体生成

#### NetworkBehaviour

- 所有需要网络同步的组件必须继承 `NetworkBehaviour`（而非 `MonoBehaviour`）
- `NetworkServer.Spawn(go)` 在服务器上生成对象并同步到所有客户端
- 使用的预制体必须在 NetworkManager 中注册

#### [SyncVar]

```csharp
[SyncVar]
public int health;
```

- 服务器修改后自动同步到所有客户端
- 支持 `hook` 回调：`[SyncVar(hook = "OnHealthChanged")]`

#### OnSerialize / OnDeserialize

- 自定义序列化：覆盖 `OnSerialize(NetworkWriter, bool)` 和 `OnDeserialize(NetworkReader, bool)`
- `bool initialState`：首次同步时为 true

#### RPC 命令

| 属性 | 方向 | 命名约定 | 用途 |
|------|------|----------|------|
| `[Command]` | 客户端 → 服务器 | 方法名前缀 `Cmd` | 客户端请求服务器执行 |
| `[ClientRpc]` | 服务器 → 所有客户端 | 方法名前缀 `Rpc` | 服务器广播调用 |

```csharp
[Command]
void CmdFire() { /* 在服务器执行 */ }

[ClientRpc]
void RpcExplode() { /* 在所有客户端执行 */ }
```

#### 匹配系统

- `NetworkMatch` 类：提供创建/搜索/加入匹配的 API
- 抽象了后端匹配服务，支持 Unity 云匹配服务

---

### 小结

| 引擎 | 网络模型 | 角色系统 | RPC | 状态同步 |
|------|----------|----------|-----|----------|
| Unreal 4 | C/S（专用+监听）| Authority/SimProxy/AutonomousProxy | Server/Client/Multicast + Reliable标志 | Actor bReplicates + 属性复制 |
| Unity UNET | C/S（Host/Client/Server）| isServer/isClient/isLocalPlayer | [Command]/[ClientRpc] | [SyncVar] + OnSerialize |

## 第12章 玩家服务

玩家服务（GamerServices）为游戏提供比赛安排、网络传输、统计、成就和排行榜等功能，以 Steamworks SDK 为主要示例。

---

### 12.1 GamerServices 架构

- **Pimpl 模式**（pointer to implementation）：`GamerServices` 持有 `unique_ptr<Impl> mImpl`，将平台相关代码隔离在 `Impl` 类中
- 好处：更换玩家服务平台时只需替换 `Impl`，上层调用代码不变
- 缺点：多一层间接访问，可读性略降

---

### 12.2 Steam 初始化

```cpp
SteamAPI_Init();          // 初始化 Steam 客户端
SteamAPI_RunCallbacks();  // 每帧调用，处理回调
SteamAPI_Shutdown();      // 退出时清理

// 服务器端
SteamGameServer_Init(...);
SteamGameServer_Shutdown();
```

- `CSteamID`：封装 `uint64`，唯一标识每个 Steam 用户
- `GetLocalPlayerId()` / `GetLocalPlayerName()` / `GetRemotePlayerName(CSteamID)`

---

### 12.3 大厅与比赛安排

#### 搜索大厅

```cpp
// CCallResult 用于一次性异步回调
CCallResult<Impl, LobbyMatchList_t> mLobbyMatchListResult;

void LobbySearchAsync() {
    SteamAPICall_t call = SteamMatchmaking()->RequestLobbyList();
    mLobbyMatchListResult.Set(call, mImpl.get(), &Impl::OnLobbyMatchListCallback);
}
```

- `STEAM_CALLBACK` 宏：注册持续性回调（如玩家加入/离开事件）
- `UpdateLobbyPlayers()`：刷新大厅玩家列表
- `SetLobbyReady()` / `LeaveLobby()`：控制大厅状态

#### 流程

1. 客户端通过 `LobbySearchAsync` 搜索大厅
2. 找到大厅后加入，等待 `mLobbyReady` 为 true
3. 离开大厅时连接到专用服务器（C/S 模式）或与其他玩家直连（P2P 模式）

---

### 12.4 网络传输

- 使用 Steam ID 路由，无需知道对方 IP 地址
- Steam 处理 NAT 穿越（relay 或直连）

```cpp
SendP2PReliable(steamID, data, size);         // 可靠发送（最大 1MB）
IsP2PPacketAvailable(&msgSize);               // 检查是否有包
ReadP2PPacket(buf, bufSize, &msgSize, &from); // 读取包
```

- `k_EP2PSendReliable`：可靠传输类型
- 首次通信触发 `P2PSessionRequest_t` 回调，需调用 `AcceptP2PSessionWithUser`
- 连接失败触发 `P2PSessionFail_t` 回调

### 12.5 玩家统计

#### X 宏模式

使用 `Stats.def` 集中定义所有统计项，通过 X 宏在多处保持同步：

```cpp
// Stats.def
STAT(NumGames, INT)
STAT(NumWins, INT)
STAT(KillRatio, AVGRATE)

// 枚举定义
enum Stat {
    #define STAT(a,b) a,
    #include "Stats.def"
    #undef STAT
    MAX_STAT
};
```

#### StatData 数据结构

```cpp
struct StatData {
    const char* Name;
    GamerServices::StatType Type;  // INT / FLOAT / AVGRATE
    int IntStat = 0;
    float FloatStat = 0.0f;
    struct { float SessionValue = 0.0f; float SessionLength = 0.0f; } AvgRateStat;
};

// 初始化数组
std::array<StatData, MAX_STAT> mStatArray;
mStatArray({
    #define STAT(a,b) StatData(#a, StatType::##b),
    #include "Stats.def"
    #undef STAT
});
```

#### 获取/更新统计

```cpp
// 从 Steam 服务器加载（初始化时调用）
void GamerServices::RetrieveStatsAsync() {
    SteamUserStats()->RequestCurrentStats();
}

// 回调：OnStatsReceived — 遍历 mStatArray 用 GetStat 填充本地缓存

// 读取
int GamerServices::GetStatInt(Stat inStat) {
    return mImpl->mStatArray[inStat].IntStat;
}

// 更新（本地+服务器）
void GamerServices::AddToStat(Stat inStat, int inInc) {
    stat.IntStat += inInc;
    SteamUserStats()->SetStat(stat.Name, stat.IntStat);
}
```

### 12.6 玩家成就

- 成就定义在 `Achieve.def`，通过 X 宏构建 `Achievement` 枚举和 `mAchieveArray`
- `AchieveData` 结构体含 `Name` 和 `Unlocked` 布尔值
- `RequestCurrentStats` 同时获取统计和成就信息

```cpp
bool GamerServices::IsAchievementUnlocked(Achievement inAch) {
    return mImpl->mAchieveArray[inAch].Unlocked;
}

void GamerServices::UnlockAchievement(Achievement inAch) {
    AchieveData& ach = mImpl->mAchieveArray[inAch];
    if(ach.Unlocked) return;
    SteamUserStats()->SetAchievement(ach.Name);
    ach.Unlocked = true;
}
```

- 最佳实践：多人游戏中将成就放入队列，**比赛结束时**统一解锁，避免玩家在 UI 提示时感到困惑

### 12.7 排行榜

- 排行榜定义在 `Leaderboards.def`，同样用 X 宏；每项含名称、排序方式、显示方式
- **一次只能查找一个排行榜**，查找完成后回调中继续查找下一个

```cpp
void GamerServices::FindLeaderboardAsync(Leaderboard inLead) {
    SteamAPICall_t call = SteamUserStats()->FindOrCreateLeaderboard(
        lead.Name, lead.SortMethod, lead.DisplayType);
    mImpl->mLeaderFindResult.Set(call, mImpl.get(), &Impl::OnLeaderFindCallback);
}

void Impl::OnLeaderFindCallback(...) {
    mLeaderArray[mCurrentLeaderFind].Handle = inCallback->m_hSteamLeaderboard;
    mCurrentLeaderFind++;
    if(mCurrentLeaderFind != MAX_LEADERBOARD)
        GamerServices::sInstance->FindLeaderboardAsync(...); // 继续找下一个
    else
        mAreLeadersReady = true;
}
```

- 查找后获得排行榜**句柄**（Handle），下载条目需调用 `DownloadLeaderboardEntries(handle, 范围类型, 起, 止)`
- 上传分数：`UploadLeaderboardScore(handle, 覆盖策略, score, details, detailCount)`
- Steam 排行榜支持关联用户生成内容（截图/视频），使排行榜更具交互性

## 第13章 云托管专用服务器

云托管让小工作室也能负担得起专用服务器，不再需要预购硬件。

---

### 13.1 托管或不托管

#### 托管的缺点

- **复杂度**：需要编写自定义进程/虚拟机管理代码，还需适配云服务商 API
- **成本**：不免费，需要靠玩家增长来覆盖
- **依赖第三方**：整个游戏命运掌握在云服务商手中
- **意外硬件改动**：供应商可能无通知更换硬件配置
- **丧失玩家所有权**：玩家无法自建服务器，失去社区感

#### 托管的优点（往往大于缺点）

- **可扩展、高带宽**：按需开启/关闭，无硬件限制
- **防作弊**：所有服务器由开发商运行，不可被玩家修改
- **合理的版权保护**：游戏后台只在开发商服务器运行，天然 DRM

---

### 13.2 行业工具

#### REST

- **表述性状态转移**（Representational State Transfer）
- 所有请求自包含，不依赖历史状态
- 基于 HTTP：GET（获取）/ POST（创建）/ PUT（存储到指定位置）/ DELETE（删除）/ PATCH（直接编辑）
- 优点：纯文本、人可读、可调试；HTTP 基于 TCP，天然可靠

#### JSON

- **JavaScript 对象符号**（JavaScript Object Notation）
- XML 的现代替代，语法简洁（无标签）
- 支持 bool / string / number / array / object
- 通过 HTTP 头 `Content-Type: application/json` 使用
- 可直接粘贴到 JavaScript 程序调试

#### Node.JS

- 基于 Google V8 JavaScript 引擎，用于构建后端服务
- **单线程、事件驱动**：主线程事件循环，异步 I/O 不阻塞
- 内置简单 HTTP 服务器；常配合 **Express JS** 框架路由请求
- `npm`（Node 包管理器）提供大量开源 REST API 封装

---

### 13.3 概述与术语

| 概念 | 定义 |
|------|------|
| 服务器游戏实例（server game instance）| 模拟游戏世界并向客户端复制信息的实体 |
| 游戏服务器进程（game server process）| 更新游戏实例、管理客户端、与 OS 交互的进程 |
| 游戏服务器（game server machine）| 运行一个或多个进程的物理/虚拟机 |

- 比赛安排服务端与专用服务器**分离部署**更好：便于嵌入 Steam/Xbox Live 等第三方比赛安排
- 部署完成后，游戏服务器进程向比赛安排系统注册自身，云部署系统专注启动/关闭实例

---

### 13.4 本地服务器进程管理器（LSPM）

LSPM 是运行在每台虚拟机上的 Node.js/Express 进程，负责：
- 响应 REST 请求，按需生成游戏服务器进程
- 监控进程心跳，自动终止无响应进程

#### API 端点

| `/api/processes/:uuid/kill` | POST | 终止进程 |
| `/api/processes/:uuid/heartbeat` | POST | 进程注册心跳 |

#### 进程监控（心跳）

### 13.5 虚拟机管理器（VMM）

VMM 是整个云部署系统的入口，负责：
- 跟踪所有 LSPM（虚拟机）的状态
- 按需向云供应商申请新虚拟机（VMI 镜像）
- 监控 LSPM 心跳，关闭空闲/异常虚拟机

#### 虚拟机镜像（VMI）

- VMI = VM 引导磁盘内容（OS + 可执行文件 + 启动脚本）
- 不同云供应商有不同 VMI 格式；未来趋势是标准化 Docker 容器格式

#### VMM 数据结构

```javascript
var gVMs = {};           // 所有活跃虚拟机
var gAvailableVMs = {};  // 可接受新进程的虚拟机（partial 状态）

// 每个 VM 对象字段：
// machineState, uuid, url, lastHeartbeat, lastSequenceIndex, cloudProviderId
```

#### VM 状态机（eMachineState 超集）

| 状态 | 含义 |
|------|------|
| empty | 无进程运行，可以关闭 |
| partial | 有进程但未满，可继续生成 |
| full | 已满 |
| pending | 正在启动中（暂不可用）|
| shuttingDown | 正在关闭 |
| recentLaunchUnknown | 启动结果不确定 |

#### 生成进程流程（`/api/processes/` 端点）

```
1. 查找 gAvailableVMs 中第一个可用 VM
2. 如无可用 VM → 向云供应商申请新 VM（askCloudProviderForVM）
3. 将 VM 标为 pending，防止并发重复启动
4. 向 VM 的 LSPM 发送 POST 请求，启动游戏进程
5. 根据 LSPM 响应更新 VM 状态（partial/full）
6. 失败时状态设为 recentLaunchUnknown（由心跳监控恢复）
```

- 使用 `sequenceIndex` 忽略乱序到达的旧响应

#### VMM 心跳监控

```javascript

router.post('/vms/:vmUUID/heartbeat', function(req, res) { ... });

function checkHeartbeats() {
    // 超时的非 pending VM → shutdownVM()
    // shutdownVM 向 LSPM 发 /api/shutdown，确认后向云供应商撤销 VM
}
```

- **关闭握手**：收到 empty 状态心跳 → VMM 向 LSPM 发关闭请求 → LSPM 确认 → VMM 通知云供应商撤销虚拟机
- LSPM 状态改变时立即发送额外心跳（无需等待心跳间隔），快速通知 VMM
- VMM `checkHeartbeats` 忽略 pending 状态的 VM 超时

#### 扩展性注意

- 如果 VMM 每秒处理大量请求，需要前置**动态负载均衡器**
- 此时 `gVMs` 状态应存储在**共享数据存储（如 Redis）**中，而非单进程本地内存

### 小结

| 组件 | 职责 |
|------|------|
| LSPM | 单台 VM 上生成/监控游戏服务器进程，向 VMM 报告状态 |
| VMM | 全局入口，管理所有 LSPM/VM，按需向云供应商申请/撤销 VM |
| REST + JSON + Node.js | 后端通信标准技术栈 |
| VMI（虚拟机镜像）| 包含 OS + 游戏可执行文件 + LSPM 的 VM 启动镜像 |

整个系统协同工作：比赛安排服务 → VMM POST /processes/ → VMM 找/建 VM → VMM 请求 LSPM 启动游戏进程 → 游戏进程运行，向 LSPM 发心跳 → LSPM 向 VMM 发心跳。

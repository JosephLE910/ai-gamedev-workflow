## 第1篇 概述

## 第1章 架构、性能和游戏

### 1.1 软件架构的本质

**好的架构**：当你做出改动时，整个程序好像在"期待"它——调用少量函数就能完美解决问题，没有副作用。

架构的核心价值在于**应对变化**。衡量架构好坏的标准就是它应对变化的灵活性。

编程流程：**理解代码 → 写代码 → 整理代码**（循环）。其中"理解代码"往往是最耗时的步骤（需要将代码加载进大脑）。

**解耦**的两层含义：
1. 若两块代码解耦，理解其一时不需要理解另一块
2. 更改一块代码时，不必更改另一块

架构的关键目标：**最小化前进前大脑中需要装载的知识量**。

### 1.2 架构的代价

过度架构的代价：
- 添加抽象和扩展点需要时间开发、调试和维护
- 猜测未来需求往往是错的，导致代码库充斥无用的抽象
- 接口、虚方法、插件系统泛滥——找不到有实际功能的代码
- 专注"引擎"而忘了做游戏（YAGNI：You aren't gonna need it）

### 1.3 性能与灵活性的权衡

- 灵活性依赖虚函数派发、接口、指针、消息等机制，有运行时开销
- 性能优化依赖具体假设（如"最多256个敌人"），假设越多，代码越僵化
- **实践结论**：保持代码灵活性直到设计稳定，然后针对性能瓶颈去除抽象
- 将一款有趣的游戏做得高效，比将一款高性能游戏做得有趣更简单

### 1.4 原型代码的取舍

- 原型（prototype）是完全合理的实践——快速验证想法
- 原型代码**必须能被丢弃**，否则会成为技术债务
- 防止原型变成正式代码的技巧：用不同语言编写原型，迫使自己重写

### 1.5 三角权衡

| 目标 | 描述 | 与其他冲突 |
|------|------|----------|
| 良好架构 | 长期可维护性 | 每次改动需更多努力保持整洁 |
| 运行时性能 | 快速执行 | 优化使代码僵化、难以修改 |
| 快速完成功能 | 短期开发速度 | 滥用"快速"会积累混乱和技术债 |

三者互相冲突，没有简单答案，只有权衡。

### 1.6 简单性

作者推崇**简单性**作为缓解三角矛盾的方法：
- 编写最干净、最直接的函数——读完就知道它在做什么
- 优先保证数据结构和算法的正确性，然后继续前进
- 好的解决方案：一小块逻辑覆盖一大片用例（而非为每个用例写条件分支）

### 小结

| 原则 | 内容 |
|------|------|
| 架构目标 | 减少做出改变时需要理解的代码量 |
| 解耦 | 改变 A 不需要改变 B；理解 A 不需要理解 B |
| 性能 vs 灵活性 | 先灵活，设计稳定后优化 |
| 简单性 | 一小块逻辑处理大片用例；少代码，少认知负担 |
| 实践建议 | 不要过早抽象；原型可以烂但必须能丢；乐在其中 |

## 第2篇 再探设计模式

## 第2章 命令模式

> 命令就是一个对象化（实例化）的方法调用。  
> 命令就是面向对象化的回调（Commands are an object-oriented replacement for callbacks）。

### 核心思想

将方法调用封装成对象（"reify"/"具象化"）。等价概念：回调、头等函数、函数指针、闭包——命令模式是没有闭包支持时的面向对象替代方案。

### 2.1 配置输入

**问题**：按钮与游戏行为硬编码，无法让玩家自定义按键。

**解决**：引入 `Command` 基类，每个游戏动作是一个子类；`InputHandler` 存储按钮→命令对象的映射，运行时可热替换。

```cpp
class Command {
public:
    virtual ~Command() {}
    virtual void execute() = 0;
};

class JumpCommand : public Command {
public:
    virtual void execute() { jump(); }
};

// InputHandler 存储 Command* buttonX_, buttonY_, ...
void InputHandler::handleInput() {
    if (isPressed(BUTTON_X)) buttonX_->execute();
    // ...
}
```

**为什么说改键"方便"**：`handleInput` 永远不用动，"改键"只是在别处换一个指针：

```cpp
buttonX_ = new JumpCommand();  // 默认：X 键跳
buttonX_ = new FireCommand();  // 玩家改键后：X 键开火
// handleInput() 完全不知道、也不需要知道
```

硬绑定若要支持改键，必须在 `handleInput` 里为每个按键写 switch，动作越多越膨胀（8 个键 × 20 种动作 = 160 个 case）；Command 方式下改键 = 换指针，`handleInput` 代码量永远不变。

> 提示：接口中只有一个返回 void 的方法时，命令模式很可能适用。

### 2.2 角色解耦

**改进**：命令不直接操作全局玩家，而是接受 `GameActor&` 参数，使同一命令可作用于任意角色。

```cpp
class Command {
public:
    virtual void execute(GameActor& actor) = 0;
};

// handleInput() 返回 Command*，调用方决定传入哪个 actor
Command* command = inputHandler.handleInput();
if (command) command->execute(actor);
```

- 同一套命令系统可用于玩家输入和 AI 驱动，只需替换"命令生产者"
- 命令队列/流：生产者（输入/AI）和消费者（角色）解耦
- 网络应用：将命令序列化后通过网络传输 → 多人游戏的基础

**UE 中的体现：Controller/Pawn 分离**

`APawn` = `GameActor`（只管能做什么），`APlayerController` / `AAIController` = 命令生产者（决定什么时候做）。Pawn 不知道自己被谁控制：

```cpp
// Pawn 上的函数——不关心驱动者是谁
void AMyCharacter::MoveForward(float Value) {
    AddMovementInput(GetActorForwardVector(), Value);
}

// 玩家控制器：绑定输入
PlayerInputComponent->BindAxis("MoveForward", this, &AMyCharacter::MoveForward);

// AI 控制器：直接调同一个函数
MyPawn->MoveForward(1.0f);
```

本项目同理：`KartAIController` 和玩家输入最终都写入同一套 `ArcadeKart` 物理状态，`ArcadeKart` 本身不区分来源。

### 2.3 撤销与重做

**关键**：在命令中保存执行前的状态，添加 `undo()` 方法。

```cpp
class Command {
public:
    virtual void execute() = 0;
    virtual void undo() = 0;
};

class MoveUnitCommand : public Command {
    Unit* unit_; int x_, y_;
    int xBefore_, yBefore_;  // 执行前保存
public:
    void execute() {
        xBefore_ = unit_->x(); yBefore_ = unit_->y();
        unit_->moveTo(x_, y_);
    }
    void undo() { unit_->moveTo(xBefore_, yBefore_); }
};
```

**多级撤销**：维护命令列表 + `current` 指针：
- 执行命令：追加列表，current 前移
- 撤销：调用 `current->undo()`，current 后移
- 重做：current 前移，重新 `execute()`
- 新命令后：舍弃 current 之后的所有命令

> 游戏回放：记录每帧每个实体执行的命令序列，回放时重新执行这些命令（比记录每帧状态省内存）。

### 2.4 函数式风格

支持闭包的语言（如 JavaScript）可直接用函数实现：

```javascript
function makeMoveUnitCommand(unit, x, y) {
    var xBefore, yBefore;
    return {
        execute: function() { xBefore = unit.x(); yBefore = unit.y(); unit.moveTo(x, y); },
        undo:    function() { unit.moveTo(xBefore, yBefore); }
    };
}
```

### 小结

| 用途 | 实现方式 |
|------|----------|
| 可配置输入 | 按钮 → Command* 映射，运行时替换 |
| 解耦控制者与角色 | execute(GameActor&)，延迟绑定 |
| 撤销/重做 | 保存执行前状态 + undo()；命令列表 + current 指针 |
| AI/玩家统一接口 | 命令队列生产者可以是输入也可以是 AI |
| 游戏回放 | 序列化命令流而非状态快照 |

### 在 UE 中的表现（命令 = 对象化的方法调用）

> "对象化(reify)"一次调用 = 把 **接收者+方法+参数** 变成对象 → 能**存/排队/延迟/转交/撤销/序列化重放**。Command 的全部威力都来自这几点。UE 里几乎处处是它：

| reify 解锁的能力 | UE 里的表现 |
|---|---|
| **最字面：方法即对象** | `UFUNCTION`→`UFunction`；调用走 `UObject::ProcessEvent(UFunction*, void* Params)`——方法和参数都成了数据。蓝图调用、RPC、复制底层都走它 |
| **绑定调用即对象** | 委托 `DECLARE_DELEGATE`/动态多播/`TFunction`/`FTimerDelegate`：存(对象+函数)、以后再 `Execute`。**我们工程**：悬挂力 `FCalculateCustomPhysics` 委托，交物理子步每步重放 |
| **配置输入**(书 2.1) | **Enhanced Input**：`UInputAction`(意图对象)+`UInputMappingContext`(键→action)，彻底解耦设备与动作。**我们车现在是 `IsInputKeyDown` 轮询=没 reify**，所以换不了键/接不了手柄→下一步该上 Enhanced Input |
| **延迟/排队** | `FTimerManager::SetTimer(委托)`、`AsyncTask`、Latent action、`UGameplayTask` |
| **网络/重放**(书：命令流过网络) | **RPC** `UFUNCTION(Server/Client/NetMulticast)`：把(函数+参数)打包成网络数据发出、对端 `ProcessEvent` 重放=纯正 Command-over-network；Replay/Demo 录制 |
| **动作即对象** | GAS `UGameplayAbility`(可激活/取消/联网的技能对象)、行为树 `UBTTaskNode`、控制台 `FAutoConsoleCommand` |
| **撤销/重做**(书 2.3) | 编辑器 Transaction：`FScopedTransaction`+`UObject::Modify()`+`GEditor` Undo/Redo（注：UE 是**状态快照式**撤销，不是纯反向 Command 对象，但落在同一需求上）|

**一句话**：UE 里凡是能"存下来/绑定/排队/发过网络/撤销"的调用，核心都是 **UFunction(方法对象化)+ 委托(绑定调用对象化)**，往上长出 Enhanced Input、Timer、RPC、GAS、行为树、Undo。

## 第3章 享元模式

> "使用共享以高效地支持大量的细粒度对象。"

### 核心思想

将对象数据分成两部分：
- **内部状态（intrinsic state）**：与上下文无关，可被所有实例共享（如树的网格/纹理）
- **外部状态（extrinsic state）**：每个实例独有（如树的位置、颜色、大小）

只存储一份内部状态，每个实例持有指向共享对象的指针。

### 3.1 树的例子

```cpp
// 共享数据（内部状态）
class TreeModel {
    Mesh mesh_; Texture bark_; Texture leaves_;
};

// 每棵树实例（外部状态 + 指向共享数据的指针）
class Tree {
    TreeModel* model_;      // 共享
    Vector position_;       // 独有
    double height_, thickness_;
    Color barkTint_, leafTint_;
};
```

GPU 渲染对应：实例绘制（Instancing）——只上传一次共享网格/纹理，再上传每个实例的差异参数，一次 DrawCall 渲染整片森林。

### 3.2 地形瓦片的例子

**问题**：用枚举 + switch 表示地形属性，数据散落在各方法中，难以维护。

**解决**：将地形数据封装为对象，多个瓦片共享同一地形实例。

```cpp
class Terrain {
    int moveCost_; bool isWater_; Texture texture_;
public:
    int getMoveCost() const { return moveCost_; }
    bool isWater() const { return isWater_; }
    const Texture& getTexture() const { return texture_; }
    // 所有方法都是 const！共享对象必须不可变
};

class World {
    Terrain* tiles_[WIDTH][HEIGHT];  // 指针网格
    // 只存三个实例
    Terrain grassTerrain_, hillTerrain_, riverTerrain_;
};
```

- 享元对象一般是**不可变的**（const）——被多处共享，修改会影响所有使用者

### 3.3 性能说明

- 指针间接访问会有缓存未命中风险，但实测中享元往往不比枚举慢
- 黄金法则：**先分析，再优化**，不要因担心性能而放弃享元带来的面向对象优点
- 如果你在写大量 switch/枚举，考虑用享元替代

### 相关模式

| 模式 | 关系 |
|------|------|
| 类型对象（第13章）| 相似——都共享数据，但意图不同：类型对象减少类定义数量，享元注重效率 |
| 对象池（第19章）| 存储/查找享元实例的实用容器 |
| 状态模式（第7章）| 无字段的状态对象可作为享元在多个状态机间共享 |
| 工厂方法 | 用于按需创建/查找享元实例（避免重复创建）|

## 第4章 观察者模式

> "在对象间定义一种一对多的依赖关系，以便当某对象的状态改变时，与它存在依赖关系的所有对象都能收到通知并自动进行更新。"

### 核心思想

被观察者（Subject）维护一个观察者（Observer）列表，状态改变时通知所有观察者。**解耦通知方和响应方**——物理引擎不需要知道成就系统的存在。

### 4.1 经典实现

```cpp
// 观察者接口
class Observer {
public:
    virtual void onNotify(const Entity& entity, Event event) = 0;
};

// 被观察者基类
class Subject {
public:
    void addObserver(Observer* observer);
    void removeObserver(Observer* observer);
protected:
    void notify(const Entity& entity, Event event) {
        for (int i = 0; i < numObservers_; i++)
            observers_[i]->onNotify(entity, event);
    }
private:
    Observer* observers_[MAX_OBSERVERS];
    int numObservers_;
};

// 物理引擎继承被观察者
class Physics : public Subject {
public:
    void updateEntity(Entity& entity) {
        // ...
        if (wasOnSurface && !entity.isOnSurface())
            notify(entity, EVENT_START_FALL);
    }
};
```

- Subject 持有观察者**列表**（而非单一观察者）——多个系统（成就/声音）可独立注册，互不干扰

### 4.2 性能说明

- 发送通知 = 遍历列表 + 调用虚函数，开销极小（虚函数开销几乎可忽略）
- 观察者模式是**同步**的——notify() 返回前所有观察者都执行完毕
  - 若观察者有耗时操作，应移至工作线程/队列（避免阻塞被观察者）
  - 多线程场景：考虑用**事件队列（第15章）**代替

### 4.3 避免动态内存分配：链式观察者

**方法一：侵入式链表**——Observer 本身作为链表节点（含 `next_` 指针）：

```cpp
class Observer {
    friend class Subject;
    Observer* next_ = nullptr;  // 侵入式
};
class Subject {
    Observer* head_ = nullptr;  // 链表头
};
```

优点：零动态分配，增删只是指针操作  
缺点：一个 Observer 同一时间只能观察一个 Subject

**方法二：链表节点池**——节点独立于 Observer，可指向同一 Observer，支持一对多观察，用对象池（第19章）预分配节点。

### 4.4 关键注意事项

| 问题 | 解决方案 |
|------|----------|
| 删除 Observer 时被观察者仍持有悬空指针 | Observer 析构时主动调用 `removeObserver()` |
| 删除 Subject 时 Observer 不知晓 | Subject 析构前发送"死亡通知" |
| GC 语言中失效观察者（lapsed listener）| 不再需要时及时取消注册 |
| 通知顺序不可预测导致耦合 | 观察者间不应依赖通知顺序 |

### 4.5 适用场景

- **适合**：不相关模块间的解耦通信（如物理 ↔ 成就）
- **不适合**：单个紧凑模块内部的通信（调试困难，因果链难以追踪）

### 4.6 现代演进

- 现代倾向：注册函数/闭包而非实现接口（C# event、JS addEventListener）
- 数据绑定（data binding）：自动化 "状态变 → UI 更新" 的样板代码
- 函数响应式编程（FRP）：更激进的数据流编程范式

## 第5章 原型模式

> "用原型实例指定创建对象的种类，并且通过拷贝这些原型创建新的对象。"

### 核心思想

对象通过**克隆自身**来创建新对象，而非通过类定义。

### 5.1 GoF 原型模式

**问题**：每种怪物需要单独的 Spawner 类（GhostSpawner、DemonSpawner…），类爆炸。

**解决**：所有怪物实现 `clone()` 接口，Spawner 持有一个"原型"实例并克隆它。

```cpp
class Monster {
public:
    virtual ~Monster() {}
    virtual Monster* clone() = 0;
};

class Ghost : public Monster {
    int health_; int speed_;
public:
    Ghost(int health, int speed) : health_(health), speed_(speed) {}
    virtual Monster* clone() { return new Ghost(health_, speed_); }
};

class Spawner {
    Monster* prototype_;
public:
    Spawner(Monster* prototype) : prototype_(prototype) {}
    Monster* spawnMonster() { return prototype_->clone(); }
};

// 使用：原型携带参数
Monster* ghostPrototype = new Ghost(15, 3);
Spawner* ghostSpawner = new Spawner(ghostPrototype);
```

- 原型携带初始状态（如血量、速度），克隆出来的实例自带这些参数
- 无需为"有参数的怪物变种"再定义子类

### 5.2 替代方案

**函数指针**：
```cpp
typedef Monster* (*SpawnCallback)();
// 注册函数指针映射，按需调用
```

**模板（C++）**：
```cpp
template <class T>
class SpawnerFor {
public:
    Monster* spawnMonster() { return new T(); }
};
// SpawnerFor<Ghost> ghostSpawner;
```

**头等类型（动态语言）**：直接存类引用并调用，无需模板或指针。

### 5.3 原型语言范式

**Self 语言（1986）**：
- 无类（classless）的面向对象，对象直接包含字段和方法
- 继承 = 委托（delegation）：对象持有 `parent` 指针，访问不到的字段/方法沿委托链查找
- 每个对象本身就是"原型"——克隆现有对象来创建新对象
- 影响深远：JIT 编译、垃圾回收等技术正是在 Self 研究中诞生

**JavaScript**：
- 技术上是基于原型的（prototype chain + `__proto__`）
- 实际上开发者通常用构造函数 + `new` 模拟类，并不直接使用原型语义
- ECMAScript 6 引入 `class` 语法糖进一步掩盖原型本质

### 5.4 数据建模中的原型

**场景**：JSON 配置文件中的游戏实体，需要表达"精灵弓箭手是精灵的变种"。

```json
{
  "name": "goblin",
  "health": 25,
  "attack": 14
}
{
  "name": "goblin wizard",
  "prototype": "goblin",
  "spells": ["fireball", "freeze"]
}
{
  "name": "goblin archer",
  "prototype": "goblin",
  "attack": 12,
  "ranged": true
}
```

- `"prototype"` 字段指向父实体名称
- 查找属性时先看自身，找不到就沿原型链向上委托
- 实现：哈希表 + 原型指针，属性查找递归

### 小结

| 方案 | 适用场景 |
|------|----------|
| GoF 原型（clone()）| 需要带参数克隆，且语言无头等类型支持 |
| 函数指针/模板 | 无需携带状态，仅需"生产 T 类型实例" |
| 头等类型（动态语言）| 直接存储并调用类型，最简洁 |
| 数据原型（JSON）| 数据驱动的实体继承，避免重复配置 |

> 作者观点：GoF 原型模式在现代语言中用得较少，组件模式和类型对象模式往往是更好的替代选择。

## 第6章 单例模式

> "保证一个类只有一个实例，并提供一个访问它的全局访问点。"

### 6.1 经典实现

```cpp
class FileSystem {
public:
    static FileSystem& instance() {
        static FileSystem* instance_ = new FileSystem();
        return *instance_;
    }
private:
    FileSystem() {}
};
```

**C++11 线程安全版本**（局部静态变量，编译器保证只初始化一次）：
```cpp
class FileSystem {
public:
    static FileSystem& instance() {
        static FileSystem instance;  // 局部静态，C++11 线程安全
        return instance;
    }
};
```

**平台多态**：继承单例 + 编译期 `#if` 选择子类：
```cpp
FileSystem& FileSystem::instance() {
#if PLATFORM == PLAYSTATION3
    static PS3FileSystem instance;
#elif PLATFORM == WII
    static WiiFileSystem instance;
#endif
    return instance;
}
```

### 6.2 单例模式的问题

**问题一：它是个全局变量**
- 全局变量破坏了函数的可理解性（副作用不明显）
- 促进了不必要的耦合——代码中任何地方都可以访问，依赖关系变得隐式
- 不适合并发：多线程访问全局状态需要加锁，单例天然是争用点

**问题二：它解决了两个问题**（YAGNI）
- 问题 A：确保只有一个实例
- 问题 B：提供全局访问点
- 通常只需要其中一个，单例强行捆绑两者

**问题三：懒初始化剥夺了控制权**
- 游戏需要精确控制内存布局和初始化时机（堆碎片、加载时间）
- 第一次访问时才初始化 = 不可预测的时机，难以调试

### 6.3 反模式：Manager 类

```cpp
// 错误倾向：
class BulletManager {
public:
    Bullet* createBullet() { ... }
    void updateBullets() { ... }
    void renderBullets() { ... }
};
// 更好的方式：把行为放在 Bullet 上，用容器管理生命周期
```

Manager/System 类往往是单例的温床，却破坏了对象封装。

### 6.4 替代方案

| 替代方案 | 做法 |
|----------|------|
| 依赖注入 | 构造时传入，而非全局获取 |
| 从基类获取 | `GameObject` 持有 `Log* log_` protected 方法，子类直接调用 |
| 聚合在顶层对象 | `Game` 对象同时持有 Log/FileSystem/Audio，全局只暴露 `Game` |
| 服务定位器（第16章）| 显式注册服务，统一查找点，但仍是全局 |

**只需确保单例、不需要全局访问**：在构造函数中断言：
```cpp
class FileSystem {
    static bool instantiated_;
public:
    FileSystem() {
        assert(!instantiated_);
        instantiated_ = true;
    }
    ~FileSystem() { instantiated_ = false; }
};
```

### 小结

| 原则 | 内容 |
|------|------|
| 经典单例 | 懒初始化 + 私有构造 + static instance() |
| C++11 线程安全 | 局部静态变量，无需手动加锁 |
| 主要问题 | 全局状态 + 隐式耦合 + 懒初始化失控 |
| 替代思路 | 依赖注入 > 基类存取 > 顶层聚合 > 服务定位器 |
| 避免 Manager | 行为属于对象本身，不应集中在管理类中 |

## 第7章 状态模式

> "允许一个对象在其内部状态改变时改变自身的行为。对象看起来好像是在修改自身类。"

### 核心问题

多个布尔标志（`isJumping_`, `isDucking_`）组合产生非法状态，每次加功能都引入新 bug：

```cpp
// 错误做法：布尔标志爆炸
void Heroine::handleInput(Input input) {
    if (input == PRESS_B)
        if (!isJumping_ && !isDucking_) { ... }
    else if (input == PRESS_DOWN)
        if (!isJumping_) { isDucking_ = true; ... }
        else { isJumping_ = false; ... }  // 俯冲攻击
}
```

**根本原因**：多个布尔的组合创造了隐式的"非法状态"（如同时跳跃+躲避）。

### 7.1 最简实现：枚举 + switch

```cpp
enum State { STATE_STANDING, STATE_JUMPING, STATE_DUCKING, STATE_DIVING };

void Heroine::handleInput(Input input) {
    switch (state_) {
        case STATE_STANDING:
            if (input == PRESS_B) { state_ = STATE_JUMPING; ... }
            else if (input == PRESS_DOWN) { state_ = STATE_DUCKING; chargeTime_ = 0; ... }
            break;
        case STATE_JUMPING:
            if (input == PRESS_DOWN) { state_ = STATE_DIVING; ... }
            break;
        // ...
    }
}
```

优点：每个状态代码集中；非法状态消失（枚举每个值都有效）。  
缺点：状态特有数据（`chargeTime_`）仍散落在 Heroine 类中。

### 7.2 GoF 状态模式

**三步走**：

**①定义状态接口**：
```cpp
class HeroineState {
public:
    virtual ~HeroineState() {}
    virtual void handleInput(Heroine& heroine, Input input) {}
    virtual void update(Heroine& heroine) {}
    virtual void enter(Heroine& heroine) {}  // 进入状态时调用
    virtual void exit(Heroine& heroine) {}   // 退出状态时调用
};
```

**②每种状态一个类**（状态特有数据封装进去）：
```cpp
class DuckingState : public HeroineState {
    int chargeTime_;  // 只在此状态有意义，放在这里
public:
    DuckingState() : chargeTime_(0) {}
    virtual void update(Heroine& heroine) {
        chargeTime_++;
        if (chargeTime_ > MAX_CHARGE) heroine.superBomb();
    }
    virtual HeroineState* handleInput(Heroine& heroine, Input input) {
        if (input == RELEASE_DOWN) return new StandingState();
        return nullptr;
    }
};
```

**③主类委托给状态**：
```cpp
class Heroine {
    HeroineState* state_;
public:
    void handleInput(Input input) {
        HeroineState* next = state_->handleInput(*this, input);
        if (next) {
            delete state_;
            state_ = next;
            state_->enter(*this);  // 进入动作
        }
    }
    void update() { state_->update(*this); }
};
```

### 7.3 状态对象的存储

| 方式 | 适用场景 | 特点 |
|------|----------|------|
| **静态实例**（`static StandingState standing`）| 状态无成员变量 | 零分配，多实体共享同一实例 |
| **动态分配**（`return new DuckingState()`）| 状态有实例数据（如 chargeTime_）| 切换时 `delete` 旧状态，`new` 新状态 |

### 7.4 并发状态机

两个独立维度（如"动作状态" vs "装备状态"）用两个独立状态机：
- n 种动作 × m 种装备 → 单机需 n×m 状态，双机只需 n+m
- 输入广播给两个状态机，各自独立处理

```cpp
class Heroine {
    HeroineState* state_;      // 动作状态机
    HeroineState* equipment_;  // 装备状态机
};
void Heroine::handleInput(Input input) {
    state_->handleInput(*this, input);
    equipment_->handleInput(*this, input);
}
```

### 7.5 层次状态机（Hierarchical State Machine）

**问题**：多个状态共享相同行为（如站立/走路/跑步都能跳跃）。

**解决**：父状态处理共有行为，子状态只处理差异；事件从子状态向上冒泡至父状态。

```cpp
class OnGroundState : public HeroineState {
    virtual void handleInput(Heroine& h, Input input) {
        if (input == PRESS_B) { /* Jump - shared */ }
        else if (input == PRESS_DOWN) { /* Duck - shared */ }
    }
};
class DuckingState : public OnGroundState {
    virtual void handleInput(Heroine& h, Input input) {
        if (input == RELEASE_DOWN) { /* Stand up */ }
        else OnGroundState::handleInput(h, input);  // 向上委托
    }
};
```

### 7.6 下推自动机（Pushdown Automata）

**问题**：FSM 无历史记录——开枪后不知道要回哪个状态。

**解决**：用**栈**存状态，而非单指针：
- **Push**：压入新状态（旧状态保留栈中）
- **Pop**：弹出当前状态，自动恢复上一状态

```
[STANDING] → 按开火 → push [FIRING] 
[FIRING]   → 动画结束 → pop → [STANDING]（自动恢复）
```

### 适用场景

FSM 适合：实体行为依赖内部状态、状态数量有限、响应输入和事件。  
游戏 AI 中更复杂的需求倾向行为树（Behavior Tree）和规划系统（Planning）。

### 小结

| 方案 | 特点 |
|------|------|
| 枚举 + switch | 简单，适合少量状态，数据仍在主类 |
| GoF 状态模式 | 状态数据和行为封装进状态类，有 enter/exit 动作 |
| 并发状态机 | 两个独立维度各用一个 FSM，状态数从乘法变加法 |
| 层次状态机 | 继承复用共有行为，事件向上冒泡 |
| 下推自动机 | 状态栈保存历史，支持"完成后返回上一状态" |

## 第8章 双缓冲

### 核心问题

渲染时显卡正在读取的帧缓冲（framebuffer）与游戏正在写入的是同一块内存，导致"撕裂"（tearing）。

### 模式定义

维护两个缓冲区：
- **当前缓冲区（current）**：外部只读，显卡从这里读取像素
- **后台缓冲区（next/back）**：游戏渲染写入这里
- **交换（swap）**：渲染完成后原子地交换两个缓冲区

```cpp
class Scene {
    Framebuffer buffers_[2];
    Framebuffer* current_;
    Framebuffer* next_;
public:
    Scene() : current_(&buffers_[0]), next_(&buffers_[1]) {}
    void draw() {
        next_->clear();
        next_->draw(1,1); /* ... */
        swap();
    }
    Framebuffer& getBuffer() { return *current_; }
private:
    void swap() {
        Framebuffer* temp = current_;
        current_ = next_;
        next_ = temp;
    }
};
```

### 非图形应用：角色状态同步

**问题**：多个角色顺序更新，A 在本帧扇了 B，B 在同帧感知到被扇——顺序影响结果。

**解决**：每个角色持有 `currentSlapped_` 和 `nextSlapped_` 双缓冲。写入 next，读取 current；所有角色更新完后再统一 swap。

```cpp
class Actor {
    bool currentSlapped_, nextSlapped_;
public:
    void slap()        { nextSlapped_ = true; }
    bool wasSlapped()  { return currentSlapped_; }
    void swap() {
        currentSlapped_ = nextSlapped_;
        nextSlapped_ = false;
    }
};
// Stage::update(): 先 update 所有角色，再 swap 所有角色
```

**优化**：用静态 `current_` 索引（0/1）代替每个对象各自的两个字段——`swap()` 只需改一次静态变量，所有实例立即生效。

### 两种交换策略

| 策略 | 速度 | 缺点 |
|------|------|------|
| 交换指针/引用 | 极快（两次指针赋值）| 外部不能持有持久指针；当前缓冲数据来自两帧前 |
| 拷贝数据 | 慢（O(n) 内存拷贝）| 当前缓冲数据只落后一帧，更"新鲜" |

### 适用场景

- 状态被逐步修改、同时可能被读取
- 希望外部观察到"原子"完整状态，看不到中间过程
- 典型：图形渲染、多实体同帧状态同步

## 第9章 游戏循环

> "实现用户输入和处理器速度在游戏行进时间上的解耦。"

### 核心结构

```cpp
while (true) {
    processInput();   // 非阻塞地处理输入
    update();         // 推进游戏状态一步
    render();         // 渲染当前帧
}
```

与 UI 事件循环的关键区别：游戏循环**不等待输入**，即使玩家无操作也持续运转。

### 四种时间步长方案

**①无限制运行**
```cpp
while (true) { processInput(); update(); render(); }
```
最简单，但游戏速度随硬件变化——快机器上跑得飞快，慢机器上卡顿。

**②固定时间步长 + sleep**
```cpp
double start = getCurrentTime();
// ... processInput/update/render ...
sleep(start + MS_PER_FRAME - getCurrentTime());
```
帧处理快时等待；帧处理慢时无效（sleep 时间为负）。

**③变时步长（浮动时间步长）**
```cpp
double elapsed = current - lastTime;
update(elapsed);  // 用实际流逝时间驱动更新
```
问题：浮点误差累积 + 物理引擎不稳定（阻尼系数依赖固定步长），**不推荐用于网络游戏或物理模拟**。

**④固定更新 + 变时渲染（推荐）**
```cpp
double lag = 0.0;
while (true) {
    lag += elapsed;
    processInput();
    while (lag >= MS_PER_UPDATE) {
        update();
        lag -= MS_PER_UPDATE;
    }
    render(lag / MS_PER_UPDATE);  // 传入 0~1 的插值参数
}
```
- 物理/AI 以固定步长更新 → 稳定、可重现
- 渲染以实际速率执行，传入 `lag/MS_PER_UPDATE` 做位置插值 → 流畅视觉
- `MS_PER_UPDATE` 须小于最慢机器上每次 `update()` 的耗时，否则永远追不上

### 设计决策

| 维度 | 选项 | 场景 |
|------|------|------|
| 谁控制循环 | 自己写 / 引擎提供 / 平台事件循环 | 使用引擎时无需自己写 |
| 省电 | 帧率上限 + sleep | 移动设备必须考虑 |
| 时间步长 | 固定更新 + 变时渲染 | 推荐方案，稳定且流畅 |

### Unity 中的映射
Unity 的 `FixedUpdate` 对应固定更新步长，`Update` 对应每帧渲染前调用——正是第④方案的具体体现。

## 第10章 更新方法

> "通过对所有对象实例同时进行帧更新来模拟一系列相互独立的游戏对象。"

### 核心思想

将每个游戏实体的行为封装进 `update()` 虚方法，游戏循环逐帧调用所有实体的 `update()`。

```cpp
class Entity {
public:
    virtual void update() = 0;
    double x_, y_;
};

class World {
    Entity* entities_[MAX_ENTITIES];
    void gameLoop() {
        while (true) {
            for (int i = 0; i < numEntities_; i++)
                entities_[i]->update();
            // render...
        }
    }
};
```

每个实体只需关心自己的行为逻辑，全局游戏循环保持整洁。

### 关键注意事项

**① 状态必须持久化到字段**  
`patrollingLeft` 从局部变量变成类成员，因为每次 `update()` 调用后控制权返回游戏循环。

**② 更新顺序影响结果**（非真正同步）  
先更新的对象 A 看到的是 B 上一帧的状态；后更新的 B 却看到 A 已更新的状态。若需真正同步，使用双缓冲（第8章）。

**③ 迭代中添加/删除对象须谨慎**
- **添加**：缓存循环开始前的 `numObjects`，新对象追加到末尾但本帧不处理
- **删除**：直接移除可能跳过后续对象；推荐方案：标记为"死亡"，本帧结束后批量清除

### update() 方法放在哪里

| 位置 | 适用场景 |
|------|----------|
| 实体类（Entity::update）| 简单游戏，实体种类少 |
| 组件类（Component::update）| 推荐：组件模式（第14章），避免类继承爆炸 |
| 代理类（State/Type::update）| 状态模式/类型对象模式，主类转发调用 |

### 变时步长

```cpp
void Skeleton::update(double elapsed) {
    x += (patrollingLeft_ ? -1 : 1) * elapsed;
    // ... boundary check
}
```

传入 `elapsed`（实际流逝时间）让对象以现实时间驱动移动，但边界判断逻辑更复杂。

### 非活跃对象优化

维护"活跃对象集合"（单独列表），避免每帧遍历全部对象——代价是两个集合需同步维护。

### 与其他模式的关系
- 游戏循环（第9章）：update 方法在游戏循环中被调用
- 组件模式（第14章）：update 方法最好放在组件上
- 双缓冲（第8章）：解决更新顺序影响结果的问题
- Unity：`MonoBehaviour.Update()` 即此模式的直接体现

## 第4篇 行为模式

## 第11章 字节码

> "通过将行为编码成虚拟机指令，而使其具备数据的灵活性。"

### 动机

**问题**：用 C++ 硬编码游戏行为（如魔法法术）存在三个痛点：
1. 修改法术需要程序员介入并重新编译
2. 无法安全地支持 Mod（用户自定义内容）
3. C++ 代码难以沙箱化——Mod 的 Bug 会崩溃游戏

**解决思路**：将行为移到数据文件中，引擎加载并执行。

### 解释器模式 vs 字节码

**GoF 解释器模式**：每个语法规则是一个对象（AST节点），通过虚函数递归求值。

缺点：大量小对象 → 内存碎片 + 缓存未命中 + 虚函数开销 = 很慢。

**字节码**：仿机器码风格——紧凑的字节数组 + 循环 + switch 执行。兼顾效率（线性内存访问）和安全（沙箱化）。

### 栈机实现

```cpp
enum Instruction {
    INST_LITERAL      = 0x00,  // 将下一字节压栈
    INST_GET_HEALTH   = 0x01,  // pop(wizard) → push(health)
    INST_SET_HEALTH   = 0x02,  // pop(amount), pop(wizard) → setHealth
    INST_ADD          = 0x03,  // pop b, pop a → push(a+b)
    INST_DIVIDE       = 0x04,
    INST_PLAY_SOUND   = 0x05,
    INST_SPAWN_PARTICLES = 0x06
};

class VM {
    int stack_[MAX_STACK];
    int stackSize_ = 0;
    void push(int v) { stack_[stackSize_++] = v; }
    int  pop()       { return stack_[--stackSize_]; }
public:
    void interpret(char bytecode[], int size) {
        for (int i = 0; i < size; i++) {
            switch (bytecode[i]) {
            case INST_LITERAL: push(bytecode[++i]); break;
            case INST_SET_HEALTH: { int amt = pop(); int wiz = pop(); setHealth(wiz, amt); break; }
            case INST_ADD: { int b = pop(); int a = pop(); push(a+b); break; }
            // ...
            }
        }
    }
};
```

**栈机执行示例**（将巫师生命设为敏捷+智力平均值）：
```
LITERAL 0     → [0]          # wizard index
LITERAL 0     → [0,0]
GET_HEALTH    → [0,45]
LITERAL 0     → [0,45,0]
GET_AGILITY   → [0,45,7]
LITERAL 0     → [0,45,7,0]
GET_WISDOM    → [0,45,7,11]
ADD           → [0,45,18]
LITERAL 2     → [0,45,18,2]
DIVIDE        → [0,45,9]
ADD           → [0,54]
SET_HEALTH    → []
```

### 设计决策

**基于栈 vs 基于寄存器**

| | 基于栈 | 基于寄存器（如 Lua 5.1）|
|--|--------|------------------------|
| 指令大小 | 小（1字节）| 大（32位，含操作数偏移）|
| 指令数量 | 多 | 少 |
| 实现难度 | 简单 | 复杂 |
| 代码生成 | 简单 | 较难 |

推荐：先做基于栈的虚拟机。

**值类型**：单一类型（最简）< 带标签的联合体（推荐，动态语言标准方案）< 接口/多态（开销大）。

**前端工具**
- 文本语言 → 需要编写词法/语法分析器，处理语法错误困难
- 图形化编辑器 → 对非技术用户友好，几乎无法创建非法程序，但需要实现 UI

### 使用时机
- 需要定义大量行为
- 行为需要被非程序员编辑或支持 Mod
- 安全性要求高（沙箱化）
- 典型应用：Lua（游戏脚本）、Kismet（Unreal 可视化脚本）

## 第12章 子类沙盒

> "使用基类提供的操作集合来定义子类中的行为。"

### 动机

大量子类（如数百种超能力）若各自直接调用外部系统，会导致：
- 代码冗余（多个子类重复相同操作）
- 子类与游戏各系统耦合（引擎改动波及所有子类）
- 难以在子类中统一执行不变量（如音效优先级）

### 模式结构

```cpp
class Superpower {
public:
    virtual ~Superpower() {}
protected:
    virtual void activate() = 0;  // 沙盒方法（子类必须实现）

    // 提供的操作（受保护，仅供子类调用）
    void move(double x, double y, double z) { /* 调用物理引擎 */ }
    void playSound(SoundId sound)            { /* 调用音效引擎 */ }
    void spawnParticles(ParticleType t, int n) { /* 调用粒子系统 */ }
    double getHeroX(); double getHeroY(); double getHeroZ();
};

class SkyLaunch : public Superpower {
protected:
    virtual void activate() {
        if (getHeroZ() == 0) {
            playSound(SOUND_SPROING);
            spawnParticles(PARTICLE_DUST, 10);
            move(0, 0, 20);
        } else { /* double jump or dive attack */ }
    }
};
```

所有外部耦合集中在基类，子类只与基类耦合。

### 基类如何获取所需状态

| 方法 | 优点 | 缺点 |
|------|------|------|
| 构造函数参数 | 保证初始化 | 每个子类构造函数都要传参 |
| 两阶段初始化（`power->init(particles)`）| 子类构造简单 | 容易忘记调用 init |
| 静态成员（`static ParticleSystem* particles_`）| 共享单一实例，内存更小 | 整个基类共享同一状态 |
| 服务定位器（`Locator::getParticles()`）| 无需外部主动传入 | 引入全局定位器 |

### 操作是否应提取到辅助类

若基类方法过多，可将相关操作提取到辅助类并通过 getter 暴露：
```cpp
class Superpower {
protected:
    SoundPlayer& getSoundPlayer() { return soundPlayer_; }
private:
    SoundPlayer soundPlayer_;
};
// 子类调用：getSoundPlayer().playSound(SOUND_SWOOP)
```

优点：减少基类方法数量，辅助类维护更容易，降低基类与外部耦合。

### 小结

| 要点 | 内容 |
|------|------|
| 核心思想 | 把所有外部耦合聚集到基类；子类只需实现沙盒方法 |
| 类层次 | 扁平：一个基类 + 大量直接子类（宽而浅）|
| 缺陷 | 脆弱基类问题——基类改动影响所有子类 |
| 基类过胖时 | 用组件模式（第14章）拆分，或提取辅助类 |

## 第13章 类型对象

> "通过创建一个类来支持新类型的灵活创建，其每个实例都代表一个不同的对象类型。"

### 问题

经典 OOP 方案：每种怪物一个子类（Dragon、Troll…）→ 添加/修改种族需要程序员修改代码并重新编译。

### 核心思想

将"类型信息"从编译时的类继承体系移到运行时的对象中。

```cpp
class Breed {
    int health_;
    const char* attack_;
public:
    Breed(int health, const char* attack)
        : health_(health), attack_(attack) {}
    int getHealth() { return health_; }
    const char* getAttack() { return attack_; }
    Monster* newMonster() { return new Monster(*this); }  // 工厂方法
};

class Monster {
    friend class Breed;
    int health_;
    Breed& breed_;
    Monster(Breed& breed) : health_(breed.getHealth()), breed_(breed) {}
public:
    const char* getAttack() { return breed_.getAttack(); }
};

// 创建：Monster* troll = trollBreed.newMonster();
```

现在设计师可以通过 JSON 数据文件定义新的种族，无需改代码重编译。

### 类型继承（种族继承种族）

两种方式：

**动态代理**（每次访问时向上查找）：
```cpp
int Breed::getHealth() {
    if (health_ != 0 || parent_ == NULL) return health_;
    return parent_->getHealth();
}
```
灵活，但有运行时遍历开销。

**复制代理**（构造时复制父类属性）：
```cpp
Breed(Breed* parent, int health, const char* attack)
    : health_(health), attack_(attack) {
    if (parent && health == 0) health_ = parent->getHealth();
    if (parent && !attack)    attack_ = parent->getAttack();
}
```
快速，但父类修改不自动传播。

**JSON 数据示例**：
```json
"Troll Archer": { "parent": "Troll", "health": 0, "attack": "fires an arrow!" }
```
`health: 0` 表示继承父类的血量。

### 设计决策

| 问题 | 封装 | 暴露 |
|------|------|------|
| 类型对象是否公开 | 对外隐藏，可重写行为；需提供转发方法 | 外部可直接访问；API 变宽 |
| 类型能否改变 | 简单、易调试 | 可减少对象创建；需验证状态合法性 |
| 派生支持 | 无继承（简单）/ 单继承（推荐）/ 多重（复杂） |

### 与其他模式的关系

- **享元模式（第3章）**：相似——都在实例间共享数据；享元重在节省内存，类型对象重在灵活性和组织
- **状态模式（第7章）**：类型对象代理静态类型数据；状态模式代理动态当前配置
- **原型模式（第5章）**：另一种在对象间共享数据的方式

## 第5篇 解耦模式

## 第14章 组件模式

> "允许一个单一的实体跨越多个不同域而不会导致耦合。"

### 问题

单一实体类（如 Bjorn）包含输入处理、物理、渲染、声效等所有逻辑 → 类臃肿、域间耦合、多人协作困难。

### 模式结构

将各域代码拆分到独立组件类中；实体类简化为组件容器。

```cpp
// 组件接口
class InputComponent   { public: virtual void update(GameObject& obj) = 0; };
class PhysicsComponent { public: virtual void update(GameObject& obj, World& w) = 0; };
class GraphicsComponent{ public: virtual void update(GameObject& obj, Graphics& g) = 0; };

// 通用容器（原来的 Bjorn 类）
class GameObject {
    int velocity, x, y;  // 跨组件共享的基础状态
    InputComponent*    input_;
    PhysicsComponent*  physics_;
    GraphicsComponent* graphics_;
public:
    void update(World& w, Graphics& g) {
        input_->update(*this);
        physics_->update(*this, w);
        graphics_->update(*this, g);
    }
};

// 创建 Bjorn：
new GameObject(new PlayerInputComponent(), new BjornPhysicsComponent(), new BjornGraphicsComponent());
// 创建演示模式 Bjorn（只换 input）：
new GameObject(new DemoInputComponent(), new BjornPhysicsComponent(), new BjornGraphicsComponent());
```

### 组件间通信的三种方式

| 方式 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| 共享容器状态 | 组件读写 `obj.velocity/x/y` | 组件间解耦 | 只能共享容器暴露的状态；顺序敏感 |
| 直接相互引用 | 图形组件持有物理组件指针 | 简单快速，可访问任意方法 | 组件间紧耦合 |
| 消息传递 | `container.send(MSG)`，广播给所有组件的 `receive()` | 组件解耦，容器简单 | 最复杂 |

实践：三种方式可混用——基础状态用共享，强关联组件用直接引用，松散通知用消息。

### 设计决策

**谁负责创建组件**
- 容器自己创建：保证不缺组件，但不灵活
- 外部传入：灵活可替换（用于演示模式、AI 控制等），推荐

### 解决的继承难题

继承体系中 Zone（碰撞）、Decoration（渲染）、Prop（两者都要）会产生"致命菱形多继承"问题。

组件方案：`Prop = GameObject + PhysicsComponent + GraphicsComponent`，无需多重继承，无代码重复。

### Unity 中的体现
Unity 的 `GameObject + Component` 体系正是此模式：`AddComponent<T>()`、`GetComponent<T>()`。

## 第15章 事件队列

> "对消息或事件的发送与受理进行时间上的解耦。"

### 动机

以音效系统为例，同步 API `Audio::playSound(id, volume)` 存在三个问题：
1. **阻塞调用者**：加载声音文件耗时，整个游戏卡顿
2. **无法批量处理**：同帧两只怪物同时嚎叫 → 叠加音量刺耳；超出声道上限则截断
3. **线程问题**：各游戏系统从自己的线程调用，没有同步原语会崩溃

核心问题：`playSound()` 的语义是"立即处理"，但音频系统未必有能力"立即处理"。解决：**将请求的接收和处理解耦**。

### 模式定义

事件队列是一个按照先进先出顺序存储一系列通知或请求的队列。发送者将请求置入队列并立即返回；处理器稍后从队列中取出并处理。发送者和受理者在时间上解耦。

### 环状缓冲区实现

```cpp
struct PlayMessage { SoundId id; int volume; };

class Audio {
    static const int MAX_PENDING = 16;
    static PlayMessage pending_[MAX_PENDING];
    static int head_;   // 读指针
    static int tail_;   // 写指针（指向下一个写入位置）

public:
    static void playSound(SoundId id, int volume) {
        assert((tail_ + 1) % MAX_PENDING != head_);  // 队列未满
        pending_[tail_].id = id;
        pending_[tail_].volume = volume;
        tail_ = (tail_ + 1) % MAX_PENDING;           // 环绕
    }

    static void update() {
        if (head_ == tail_) return;                  // 空队列
        ResourceId res = loadSound(pending_[head_].id);
        int channel = findOpenChannel();
        if (channel == -1) return;
        startSound(res, channel, pending_[head_].volume);
        head_ = (head_ + 1) % MAX_PENDING;           // 环绕
    }
};
```

**环状缓冲区优势**：固定数组，无动态分配，无元素移动，可缓存。

### 请求合并

在 `playSound()` 入队时扫描队列，若已有相同 id 的请求则取较大音量，直接返回（不再入队）：

```cpp
for (int i = head_; i != tail_; i = (i + 1) % MAX_PENDING) {
    if (pending_[i].id == id) {
        pending_[i].volume = max(volume, pending_[i].volume);
        return;
    }
}
// 没找到则正常入队
```

### 使用须知

| 注意点 | 说明 |
|--------|------|
| 全局事件总线是全局变量 | 带有全局变量的所有问题 |
| 事件处理时世界状态可能已变 | 事件数据需在发送时快照 |
| 反馈循环 | 处理事件时尽量不再发事件 |

### 设计决策

- **队列中是事件还是消息**：事件（已发生）→ 多播；消息（请求执行）→ 单播
- **谁能读取**：单播（封装好）/ 广播（灵活但需过滤）/ 工作队列（多消费者分担）
- **谁能写入**：单写者（隐式知道来源）/ 多写者（需注意反馈循环）
- **生命周期**：转移所有权 / 共享所有权（`shared_ptr`）/ 队列持有（对象池）

### 与其他模式的关系
- 观察者模式（第4章）的异步版本
- 命令模式（第2章）可以入队
- Go 的 channel 本质就是事件队列

## 第16章 服务定位器

> "为某服务提供一个全局访问入口来避免使用者与该服务具体实现类之间产生耦合。"

### 动机

音频系统在游戏各处被调用：物理系统、AI、UI……每处都 `AudioSystem::playSound(...)` 直接引用具体类。问题：**使用者与具体实现耦合**，换实现要改动所有调用点。

类比：让100个陌生人记住你的家庭住址，搬家时要挨个通知。更好的方案：**电话簿**——查名字得地址，搬家只需更新电话簿。这就是服务定位器。

### 模式结构

```cpp
// 抽象服务接口
class Audio {
public:
    virtual void playSound(int id) = 0;
    virtual void stopSound(int id) = 0;
    virtual void stopAllSounds() = 0;
};

// 具体实现
class ConsoleAudio : public Audio { /* ... */ };

// 定位器
class Locator {
public:
    static void initialize() { service_ = &nullService_; }
    static Audio& getAudio()  { return *service_; }
    static void provide(Audio* service) {
        service_ = (service == NULL) ? &nullService_ : service;
    }
private:
    static Audio* service_;
    static NullAudio nullService_;  // 空服务，什么都不做
};
```

**空服务（Null Object）**：`NullAudio` 实现接口但什么都不做，保证 `getAudio()` 永远返回有效对象，调用者无需检查 NULL。

### 日志装饰器

```cpp
class LoggedAudio : public Audio {
    LoggedAudio(Audio& wrapped) : wrapped_(wrapped) {}
    void playSound(int id) override { log("play"); wrapped_.playSound(id); }
    // ...
    Audio& wrapped_;
};

// 开启日志：
Locator::provide(new LoggedAudio(Locator::getAudio()));
```

### 服务定位方式对比

| 方式 | 优点 | 缺点 |
|------|------|------|
| 外部代码注册 | 灵活，可运行时替换；可控构造参数 | 依赖初始化顺序 |
| 编译期绑定（`#if DEBUG`）| 极快，无运行时开销 | 无法运行时切换 |
| 运行时配置（反射+配置文件）| 无需重编译；非程序员可配置 | 复杂，定位有开销 |

### 服务找不到时的处理

- **返回 NULL**：调用者自己处理，代码繁琐
- **断言终止**：简洁，但会暂停所有人开发
- **返回空服务（推荐）**：游戏继续运行，结合调试日志可追踪

### 与单例的区别

单例让类自己管理唯一实例且类知道自己是单例；服务定位器将"实现"与"如何找到它"分开，服务类本身无感知。Unity 的 `GetComponent<T>()` 结合了服务定位器与组件模式。

## 第6篇 优化模式

## 第17章 数据局部性

> "通过合理组织数据利用CPU的缓存机制来加快内存访问速度。"

### 核心原理

CPU 访问 RAM 需要数百个时钟周期；而访问缓存（L1/L2/L3）极快。CPU 每次从 RAM 读取时，会将一整个**缓存行（cache line，64~128字节）**加载进缓存。

**结论**：如果程序要处理的数据在内存中连续排列，缓存命中率高，性能极高；若数据分散（指针跳跃），每次都缓存未命中，等待数百周期 → 可相差 **50 倍**性能。

### 问题代码（指针跳跃）

```cpp
// 差：三层指针间接访问，每次都可能缓存未命中
while (!gameOver) {
    for (int i = 0; i < numEntities; i++)
        entities[i]->ai()->update();       // 两次指针解引用
    for (int i = 0; i < numEntities; i++)
        entities[i]->physics()->update();
    for (int i = 0; i < numEntities; i++)
        entities[i]->render()->render();
}
```

### 优化：连续数组

```cpp
// 好：直接存储组件值，线性扫描，缓存友好
AIComponent*      aiComponents      = new AIComponent[MAX_ENTITIES];
PhysicsComponent* physicsComponents = new PhysicsComponent[MAX_ENTITIES];
RenderComponent*  renderComponents  = new RenderComponent[MAX_ENTITIES];

while (!gameOver) {
    for (int i = 0; i < numEntities; i++) aiComponents[i].update();
    for (int i = 0; i < numEntities; i++) physicsComponents[i].update();
    for (int i = 0; i < numEntities; i++) renderComponents[i].render();
}
```

测试结果：比之前版本快 **50 倍**。

### 包装数据（活跃粒子排在数组前端）

将活跃粒子维持在数组头部，`numActive_` 跟踪数量，避免每帧 if 检查非活跃粒子：

```cpp
void ParticleSystem::activateParticle(int index) {
    assert(index >= numActive_);
    Particle temp = particles_[numActive_];
    particles_[numActive_] = particles_[index];
    particles_[index] = temp;
    numActive_++;
}
// 更新循环：for (int i = 0; i < numActive_; i++) particles[i].update();
```

### 热/冷分解

将每帧必用的数据（热）与偶尔用的数据（冷，如掉落物配置）分开存储：

```cpp
class AIComponent {
    Animation* animation_;   // 热：每帧用
    double energy_;
    Vector goalPos_;
    LootDrop* loot_;         // 冷：只在死亡时用，用指针间接访问
};
```

减少热数据结构的体积 → 一条缓存行能装更多对象 → 命中率提升。

### 设计决策

| 问题 | 方案 |
|------|------|
| 如何处理多态 | 避免继承（最快）/ 按类型分独立数组 / 指针数组（灵活但不缓存友好）|
| 游戏实体如何索引组件 | 指针（简单，移动困难）/ ID（灵活，慢）/ 实体即 ID（最彻底，兄弟组件查找靠索引对应）|

### 与其他模式的关系
- 组件模式（第14章）：配合使用，按组件类型分数组
- 对象池（第19章）：维护连续数组需要对象池支持

## 第18章 脏标记模式

> "将工作推迟到必要时进行以避免不必要的工作。"

### 动机（场景图世界变换）

场景图中每个节点有**局部变换**（相对父节点）和**世界变换**（绝对位置）。渲染需要世界变换，而世界变换 = 本节点局部变换 × 父节点世界变换（递归合并）。

**问题**：若每次修改局部变换就立即递归更新所有子节点的世界变换，一帧内4个变换可能触发10次矩阵乘法（大量冗余）。

**方案**：用一个脏标记（`dirty_` flag）记录"此节点世界变换是否已过期"。修改局部变换时仅设置标记；渲染时才按需重算。

### 示例代码

```cpp
class GraphNode {
    Transform local_;          // 局部变换
    Transform world_;          // 缓存的世界变换
    bool dirty_ = true;        // 是否需要重算

    void setTransform(Transform local) {
        local_ = local;
        dirty_ = true;         // 标记过期（不递归子节点）
    }

    void render(Transform parentWorld, bool dirty) {
        dirty |= dirty_;       // 父节点脏则自己也脏
        if (dirty) {
            world_ = local_.combine(parentWorld);
            dirty_ = false;
        }
        if (mesh_) renderMesh(mesh_, world_);
        for (int i = 0; i < numChildren_; i++)
            children_[i]->render(world_, dirty);  // 传播脏状态
    }
};
```

**技巧**：不在 `setTransform()` 递归标记子节点；而是在 `render()` 遍历时通过参数 `dirty` 向下传播，父脏则子强制重算。

### 使用条件

- 原始数据的修改频率 **远高于** 衍生数据的使用频率
- 增量更新困难（若容易增量更新就没必要用此模式）

### 使用须知

| 注意点 | 说明 |
|--------|------|
| 延时可能造成卡顿 | 推迟到需要时才算，若计算量大会在玩家看到结果时卡顿 |
| 必须在每次修改原始数据时设置脏标记 | 任何遗漏都会导致数据不一致 |
| 需要在内存中保存上次计算的衍生数据 | 空间换时间 |

### 设计决策

**何时清除脏标记**：
- 需要结果时（懒计算）：避免不必要计算，但可能在关键时刻卡顿
- 精心设定的检查点（如过场动画、存档点）：用户体验好
- 后台定时处理：需要线程支持

**粒度**：粒度细 → 只传输真正变化的数据；粒度粗 → 脏标记更少但传输更多未变化数据。

## 第19章 对象池

> "使用固定的对象池重用对象，取代单独地分配和释放对象，以此来达到提升性能和优化内存使用的目的。"

### 动机

频繁创建/销毁粒子等对象会导致：
- **内存碎片**：堆空间碎裂为小块，即使总量够也无法分配大块连续内存
- **分配速度慢**：内存管理器开销

对象池解决方案：**一次分配一大块，持续重用**，对内存管理器而言只有一次分配。

### 基础实现

```cpp
class Particle {
    int framesLeft_ = 0;
    double x_, y_, xVel_, yVel_;
public:
    bool inUse() const { return framesLeft_ > 0; }
    void init(double x, double y, double xVel, double yVel, int lifetime);
    void animate() {
        if (!inUse()) return;
        framesLeft_--;
        x_ += xVel_; y_ += yVel_;
    }
};

class ParticlePool {
    static const int POOL_SIZE = 100;
    Particle particles_[POOL_SIZE];   // 连续数组，一次分配
public:
    void create(double x, double y, double xVel, double yVel, int lifetime) {
        for (int i = 0; i < POOL_SIZE; i++) {
            if (!particles_[i].inUse()) {
                particles_[i].init(x, y, xVel, yVel, lifetime);
                return;
            }
        }
        // 找不到则忽略（或选择替换最不重要的）
    }
    void animate() {
        for (int i = 0; i < POOL_SIZE; i++) particles_[i].animate();
    }
};
```

创建是 O(n)。

### 空闲表优化（O(1) 创建）

利用未使用粒子的内存存储"下一个空闲粒子"的指针（union 复用内存）：

```cpp
class Particle {
    int framesLeft_;
    union {
        struct { double x, y, xVel, yVel; } live;
        Particle* next;   // 未使用时，链接到下一个空闲粒子
    } state_;
};

// Pool 维护 firstAvailable_ 指针
// create(): 取链表头  O(1)
// animate(): 死亡粒子插回链表头  O(1)
```

### 使用须知

| 注意点 | 说明 |
|--------|------|
| 闲置对象浪费内存 | 池大小需根据游戏实际需求调整 |
| 活跃对象数目恒定 | 池满时：忽略 / 强行回收最不重要的 / 动态扩容 |
| 每个对象内存大小固定 | 不同大小对象用不同池；或按最大对象尺寸分配（浪费）|
| 重用对象不会自动清理 | 需确保 init() 完整初始化所有字段 |
| GC 语言中池内对象引用不会被回收 | 不再使用时清空对象中的引用字段 |

### 设计决策

**对象是否知道自己在池中**
- 耦合（知道）：实现简单，可避免额外标志位（利用现有状态如 `framesLeft_`）
- 解耦（不知道）：通用 `GenericPool<T>`，但需额外 `bool inUse_[]` 数组

**谁来初始化重用对象**
- 池内部初始化：完全封装，但需转发所有初始化参数签名
- 外部初始化：`pool.create()` 返回指针，调用者自行 `init()`；更灵活，但需检查返回 NULL

### 与其他模式的关系
- 享元模式：享元重在并发共享；对象池重在时间上的重用
- 数据局部性（第17章）：对象池天然提供连续数组

## 第20章 空间分区

> "将对象存储在根据位置组织的数据结构中来高效地定位它们。"

### 动机

在 RTS 游戏中处理近战战斗，朴素方法是双重循环比较所有单位对，复杂度 O(n²)，单位数量大时性能崩溃。

**解决思路**：如果将对象按位置排序存储（1D 则可二分查找），就能快速找到附近对象。将这个思路扩展到 2D/3D 就是空间分区。

### 固定网格实现

```cpp
class Grid {
    static const int NUM_CELLS = 10;
    static const int CELL_SIZE = 20;
    Unit* cells_[NUM_CELLS][NUM_CELLS];  // 每格存链表头

    void add(Unit* unit) {
        int cellX = (int)(unit->x_ / CELL_SIZE);
        int cellY = (int)(unit->y_ / CELL_SIZE);
        // 插入双向链表头部
        unit->prev_ = NULL;
        unit->next_ = cells_[cellX][cellY];
        cells_[cellX][cellY] = unit;
        if (unit->next_) unit->next_->prev_ = unit;
    }

    void move(Unit* unit, double x, double y) {
        int oldCellX = (int)(unit->x_ / CELL_SIZE);
        int oldCellY = (int)(unit->y_ / CELL_SIZE);
        int newCellX = (int)(x / CELL_SIZE);
        int newCellY = (int)(y / CELL_SIZE);
        unit->x_ = x; unit->y_ = y;
        if (oldCellX == newCellX && oldCellY == newCellY) return;
        // 从旧格链表摘除，加入新格链表
        // ... （双向链表摘除 + add()）
    }
};
```

**战斗处理**：只比较同一格内的单位，O(n²) 降为每格内的小 O(k²)（k 远小于 n）。

**跨格攻击距离**：检查当前格 + 相邻 4 格（只检一半避免重复）：

```cpp
handleUnit(unit, unit->next_);         // 同格
if (x > 0) handleUnit(unit, cells_[x-1][y]);
if (y > 0) handleUnit(unit, cells_[x][y-1]);
if (x > 0 && y > 0) handleUnit(unit, cells_[x-1][y-1]);
if (x > 0 && y < NUM_CELLS-1) handleUnit(unit, cells_[x-1][y+1]);
```

**使用双向链表**：单位移动时 O(1) 从格中摘除并插入新格。

### 设计决策

**扁平 vs 层级分区**

| | 扁平（固定网格）| 层级（四叉树/八叉树）|
|--|----------------|----------------------|
| 实现复杂度 | 简单 | 较复杂 |
| 内存 | 固定 | 随对象分布变化 |
| 空旷区域 | 浪费空单元格 | 不细分，高效 |
| 对象密集区 | 同一格对象多，退化 | 自适应细分 |
| 移动对象 | 更新快 | 可能需调整多层 |

**分区是否依赖于对象集合**

| 固定分区（网格）| 自适应（BSP/k-d树）| 四叉树（介于两者之间）|
|----------------|--------------------|-----------------------|
| 增量添加快 | 批量构建高效 | 增量添加/删除均快 |
| 不均匀分布时退化 | 保证平衡 | 自适应，分区固定 |

**对象是否只存储在分区中**
- 只存分区：内存省，但遍历所有对象需扫描所有非空格
- 同时维护独立列表：遍历快，但需保持两者同步

### 空间分区数据结构对应关系

| 空间结构 | 对应一维结构 |
|----------|-------------|
| 网格 | 桶排序 |
| BSP / k-d 树 / BVH | 二叉查找树 |
| 四叉树 / 八叉树 | Trie 树 |

### 与其他模式的关系
- 数据局部性（第17章）：按空间组织数据天然提升局部性
- 对象池（第19章）：对象池中的连续数组方便空间分区遍历

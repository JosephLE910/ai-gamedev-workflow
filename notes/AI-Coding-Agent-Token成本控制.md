# AI Coding Agent 的 Token 成本控制

> 来源：[腾讯技术工程](https://mp.weixin.qq.com/s/x8ssQ-trmIqHMPlvQSE9SA)
> 作者：devinyzeng

## 一句话摘要

Token 成本优化的本质不是"少问一句话"，而是"让系统少重复搬运同一批上下文"。五层优化模型：使用习惯 → 模型路由 → 上下文压缩 → 代码图谱 → Agent 架构。

## 核心心智模型

**账单大头不是你的问题本身，而是系统替你背上的整包上下文**：System Prompt、Skill 定义、Tool 定义、历史会话、代码文件——你敲的那句话可能不到总 Token 的 1%。

### 一次请求的典型拆解

```
System Prompt      5K
项目说明文档      10K
Skill 定义        20K
Tool/MCP 定义     30K
历史会话         100K
代码文件          50K
用户问题         0.1K     ← 你只问了这么多
```

### 五种成本类型

| 成本类型 | 放大因素 |
|---|---|
| 输入 Token | 每轮重复携带，最大头 |
| 输出 Token | 回答越长越贵 |
| 推理 Token | 简单任务走高推理档位明显溢价 |
| 工具往返 | 一次工具调用可能比原问题还长 |
| 重试成本 | 每修一次，整包上下文再付一遍 |

**Prompt Cache 是基础**：缓存的是稳定前缀的处理结果。静态内容放前面、动态内容放后面 → 提升可复用比例。

## 五层优化模型

### 第一层：使用习惯（零工程成本，最大首波收益）

| 习惯 | 原理 |
|---|---|
| 一个 Session 一件事 | 切 Topic = 最便宜的 Context 管理 |
| 长会话及时 `/compact` | 对话记录不是资产，未压缩是负债 |
| 聊天记录不是数据库 | 长期信息外置到文档/Memory/知识图谱 |
| 指定输出格式 | 同时省输出 + 减少重试 |
| Skill 按需加载 | 高频常驻，低频按需触发 |
| MCP 越少越好 | 每多一个 = 多一份 Tool Definition + 选择成本 |
| CLI 优先于 MCP | 已有成熟 CLI 时别过度工具化 |
| 引用文件带完整路径 | 省掉一整条搜索链路 |
| 意图一次说完 | 别聊天式拆碎，四轮对话成本 = 一轮完整表达的几倍 |

### 第二层：模型路由

| 任务类型 | 推荐模型 | 理由 |
|---|---|---|
| 写 UT/commit/注释 | 便宜模型 | 模式稳定、模板化 |
| Code Review | 中高档 | 需要语义理解 |
| 架构设计/复杂 Bug | 强模型 | 长链路推理 |
| 批量分类/摘要 | 低价或 Batch | 规模大、成本敏感 |

关键原则：**Skill/Agent/Command 绑定模型**，让成本治理从"靠人记得切"变成"系统默认更省"。

### 第三层：上下文压缩工具

| 工具 | 压缩对象 | 典型节省 |
|---|---|---|
| **RTK** | 终端命令输出 | 89% |
| **Caveman** | AI 回复输出 | 65–75% |
| **headroom** | 所有进上下文的内容 | 47–92% |
| **context-mode** | MCP 工具结果 + 会话连续性 | 98%（工具输出） |

最低起点：先装 RTK + Caveman，五分钟见效。

### 第四层：代码图谱

让 AI 在动手读文件之前就知道该读哪里，避免"grep 找入口 → 读文件找关系 → 发现漏了 → 再 grep"循环。

| 工具 | 特点 | 节省 |
|---|---|---|
| **Graphify** | 零配置，Skill 形态 | 比直接读文件少 71.5 倍 Token |
| **CodeGraph** | MCP 服务器，持久化查询 | 16% 成本下降，47% 更少 Token |

### 第五层：Agent 架构

别让所有任务都挤进同一个上下文。Orchestrator-Worker 模式：

- **Orchestrator**（强模型）：规划、拆解、调度、汇总，只看任务状态
- **Worker**（便宜模型）：只做一件事，只看相关文件，完成即销毁
- 数据通过共享外置文件（JSON，非自然语言）传递
- 进度文件追踪状态（`progress.json`）
- 可并行的独立子任务同时启动

**单 Agent vs Orchestrator-Worker 成本对比**：同样任务节省 70–85% Token。

## 常见误区

1. 上下文越多越好 → 越多越贵，噪音越多越分神
2. MCP 越多越强 → 高频用的就两三个，其余是成本
3. 所有 Agent 都上最强模型 → 等于放弃路由
4. 聊天记录当长期记忆 → 不是记忆系统
5. 只看单价不看总成本 → 便宜模型更多重试可能更贵
6. Prompt 越短越好 → 丢了必要的 few-shot，重试更贵

## 与我们的体系的关系

我们的工作流天然对应了部分优化：

| 我们的实践 | 对应的优化层 |
|---|---|
| 🔒/🔧 分层（lock 只放 plan+review）| Agent 架构（Orchestrator-Worker） |
| git worktree 隔离执行者 | Agent 架构（并行隔离） |
| EXECUTOR_RULES "只读当前任务相关"| 使用习惯（精准上下文） |
| PLANNER_RULES "不重复规划" | 重试成本（减少无效往返） |
| LESSONS.md 外置经验 | 聊天记录不是数据库 |

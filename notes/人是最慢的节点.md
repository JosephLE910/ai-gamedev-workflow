# 人是最慢的节点，还怎么管 AI Agent？

> 来源：[腾讯研究院 AI 跃迁者调研 #4](https://mp.weixin.qq.com/s/tIx02ra7Y58xdchsTtzZtw)
> 嘉宾：张佳圆（Multica 创始人，前 TikTok 工程师）
> 访谈者：余一（腾讯研究院高级研究员）

## 一句话摘要

4 人团队 + 几十个 Agent 的开源项目 Multica（GitHub 2.75w Star），每周 1.2w Star 增长，每 10 秒触发一个 Agent 任务。核心洞察：**组织的产出效率瓶颈已经是人，而非 AI 或 Agent**。

## 七个金句

1. "整个组织的产出效率瓶颈，其实现在已经是人了，而非 AI 或者 Agent。"
2. "建太多层级，是对人类这种低效组织的一个拙劣模仿。"
3. "供给侧生产侧变得无限之后，决定不去做什么事情，可能来得更加重要。"
4. "即使你快速做一个错误的决策，要比你缓慢地做一个正确的决策来得更好，因为你做了错误的决策总是可以去修整它，但你犹豫不决的话，就会导致整个组织因为你这一块而 block 在这里。"
5. "你只要活得足够久，可能是你一个很大的壁垒了。"
6. "Multica 如果失败了，但是它启发了下一代操作系统的诞生，我觉得这可能就是很有意义的事情。"
7. "人的思考在 AI 时代是被低估的。AI 给你吐出来的东西可能是一个 P50 的东西，就是中位数。"

## Multica 的核心逻辑

**产品定位：协作层，而非 Agent 本身。**

不做 Agent，做的是 Agent 的编排与协作层。核心概念：

| 概念 | 说明 |
|---|---|
| 运行时 | Agent 运行的机器（本地 MacBook/Mac Mini/服务器），统一注册到 workspace |
| 智能体 | 相当于 AI 员工，可分配任务、设置角色 |
| Agent Team | 多个 Agent 组成的小队，有自己的工作流程 |

日常模式：创建任务 → 分配给 Agent 或 Agent Team → 人只需做 review → Agent 需要你介入时出现在 inbox。

## 组织变革

- 4 人都是 builder，没有固定岗位边界
- **去掉了人与 Agent 之间的中间传递层**
- 以前多人协作需要传递传达，现在是"一人 + 多个 Agent"端到端

## 对我们的启示

我们的工作流已经有类似的 Agent Team 雏形：

| Multica 概念 | 我们的对应 |
|---|---|
| 任务分配给 Agent Team | 规划者出 plan 分给执行者 |
| 人只需 review | 🔒 lock 层（规划者出 plan 需人批准）|
| Agent 需要介入时出现 | PLAN-02: 执行者遇到困难投递给规划者 |
| 一人 + 多 Agent 端到端 | 单人 + 规划者 + 执行者三角色协作 |
| 去掉中间传递层 | 文件交接代替上下文共享（PLANNER_EXCHANGE.md） |

值得思考的方向：当前我们一个项目只有一套 Agent Team（1 规划者 + 1 执行者）。如果任务足够大，是否需要像 Multica 一样，"一个项目有多组 Agent Team 并行推进不同模块"？
